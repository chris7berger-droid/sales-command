Session: Call Log delete bug fix

===============================================================================
WHAT WAS DONE
===============================================================================

Fixed Call Log entries not deleting (bug noted in v67 handoff).

Root cause — two issues in CallLogDetail.jsx handleDelete():

1. Proposals guard queried without deleted_at filter:
   The check for "does this job have a proposal?" was returning soft-deleted
   proposals too. Any job that ever had a proposal (even one already deleted)
   was permanently blocked with "This job has a proposal attached."
   Fix: added .is("deleted_at", null) to the guard query.

2. FK constraint on proposals table:
   Even after the guard passed, Postgres rejected the call_log delete because
   soft-deleted proposals still existed as rows with call_log_id set
   (proposals_call_log_id_fkey violation).
   Fix: before deleting call_log, null out call_log_id on any soft-deleted
   proposals for that job.

Final delete sequence in handleDelete():
  1. Block if active (non-soft-deleted) proposals exist
  2. Delete job_work_types (FK child)
  3. Null out call_log_id on soft-deleted proposals (clears FK)
  4. Delete call_log row
  5. Verify delete succeeded (RLS silent no-op check)
