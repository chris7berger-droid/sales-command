WHAT WAS DONE
===============================================================================

1. CRITICAL BUG FIX — RECIPIENTS SECTION NO LONGER DELETES FROM PARENT

   The RECIPIENTS section on ProposalDetail was rendering and mutating
   customer_contacts directly. The × delete button was running
   DELETE FROM customer_contacts, permanently removing contacts from
   the parent customer file whenever a user "removed them from the
   proposal". This explains empty Contacts lists on customers who have
   contacts in older proposals.

   Fix: section now renders proposal_recipients rows, with a proper
   separation of concerns. Delete removes the recipient row only; the
   customer_contact on the parent file stays intact.

   NOTE: prior data damage cannot be retroactively recovered — those
   customer_contacts were hard-deleted in prior sessions. The new
   "Save to Customer" button (item 3 below) is the backfill path for
   orphan proposal_recipients that lost their parent contact.

2. DB CHANGE — proposal_recipients schema update (LIVE in prod)

   First use of supabase/migrations/ folder in this repo. Pushed via
   `supabase db push` against linked "Command Suite DB" project.

   Migration: 20260416171150_proposal_recipients_phone_and_contact_fk.sql

     ALTER TABLE public.proposal_recipients
       ADD COLUMN IF NOT EXISTS phone text;

     ALTER TABLE public.proposal_recipients
       ADD COLUMN IF NOT EXISTS customer_contact_id uuid
         REFERENCES public.customer_contacts(id) ON DELETE SET NULL;

     CREATE INDEX IF NOT EXISTS idx_proposal_recipients_customer_contact_id
       ON public.proposal_recipients(customer_contact_id);

   - phone: snapshot per-recipient, also displayed on card
   - customer_contact_id: link to parent contact so PM/Office/Billing
     role stays in sync with the parent file
   - ON DELETE SET NULL: deleting a customer_contact elsewhere just
     blanks the FK on recipients; name/email/phone snapshots survive

3. RECIPIENTS PICKER — Pull from parent, New Contact, Save to Customer

   The "+ Add Contact" button now opens an inline picker:

     - Lists parent customer_contacts not yet attached to this proposal
     - Tap any → instantly inserts proposal_recipient with FK link
     - "+ New Contact" at the bottom expands name/email/phone/role form
       • Dedupes by email (case-insensitive) before creating
       • If email already exists on a parent contact, links that one
       • If not, inserts customer_contact THEN links recipient
     - "Done" closes the picker

   Delete button (styled pill, red text, replaced the bare × glyph):
     - Confirms "contact stays on customer file" in dialog
     - Deletes ONLY proposal_recipients row
     - customer_contacts row untouched

   "Save to Customer" button (dashed teal pill, only on orphan rows):
     - Shows only when customer_contact_id is null
     - One click: dedupes by email against existing customer_contacts,
       then either links to match or inserts new (role=Project Manager)
     - Backfill path for legacy proposal_recipients from before the FK

4. SIGNER TOGGLE — Viewer/Signer pill, one-signer enforcement

   Per-recipient pill next to the PM/Office/Billing role pill:

     - Default: Viewer (outlined, muted text)
     - Tap to toggle: becomes Signer (teal text on dark background)
     - Only one signer allowed — setting a new one demotes previous
       to viewer automatically in the same transaction
     - Writes proposal_recipients.role ('signer' | 'viewer')

5. CUSTOMER PROFILE CONTACT CARDS — Full redesign

   Contacts section on the customer profile page (Customers.jsx):

   Before: inline flex-wrap of 200px min-width cards with 13px name
   After: responsive CSS grid (repeat(auto-fill, minmax(280px, 1fr)))

     - Name: 18px / weight 800, dominant element
     - Role pill: 10px teal-on-dark below name
     - Email: 12.5px body text
     - Phone: 12.5px muted text
     - Primary badge: top-right corner, teal-on-dark pill
     - Subtle box shadow, 12px rounded corners

   + Add Contact button changed from ghost to primary variant
   (black background + teal text) to match the brand action style.

6. EDIT CUSTOMER MODAL — Dropped Contact Phone/Email fields

   These two fields on customers.* were a vestige of the pre-contacts-
   table single-contact model. With the dedicated Contacts card section
   now on the customer profile (with PM/Office/Billing roles and a
   primary flag), the Contact Phone/Email fields caused confusion —
   users saw two email fields with no clear relationship to the
   Contacts list below.

   Removed: "Contact Phone" and "Contact Email" Field components from
   CustomerModal, plus form state and payload keys.

   Relabeled: "Billing Contact Same?" → "Billing Same as Company?"
   (with body text updated to match).

   DB columns preserved: customers.contact_phone and
   customers.contact_email are still in the schema and still populated
   by CallLogDetail + NewInquiryWizard. All downstream fallback reads
   keep working (billing_email || contact_email || email pattern in
   invoices, QB sync, PDF modal, public signing page).
