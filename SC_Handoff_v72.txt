Session: Customer Billing Schedule (SOV / G702-G703) Phases 1-2 + Role Gating

===============================================================================
CONTEXT / PROBLEM STATEMENT
===============================================================================

Two billing pains the existing WTC-based invoicing doesn't handle:

1. Archive (History Locker) jobs can't be invoiced — they have no structured
   proposal_wtc rows, so NewInvoiceModal's bill-% workflow has nothing to
   bill against. First test case: legacy_id 6359 (DA Builders Muscle Motors,
   $73,298.97, archive_record_id bc9bb80d-e8d2-4d0d-a09e-651945f90af6).

2. Live jobs: customer contracts clump/rename WTCs into their own SOV
   line items with their own identifiers. Invoices built from WTC rows
   don't match the customer's contract framework, which causes billing
   disputes.

Decision: build a Customer Billing Schedule layer (G702/G703 framework)
that sits on top of WTCs. WTCs remain internal cost tracking; SOV is what
we bill on. Customer contract PDF is uploaded as evidence; SOV lines are
entered manually from that doc.

Full 5-phase plan saved in memory: project_customer_billing_schedule.md

1. PHASE 1 — DATA MODEL (DB CHANGES LIVE IN PROD)

   Migration: supabase/migrations/20260416175646_billing_schedule_and_archive_links.sql
   Applied via `supabase db push`.

   New tables (RLS-enabled, tenant-scoped, standard pattern):
     - public.billing_schedule
       • id uuid PK, proposal_id text NOT NULL UNIQUE FK proposals ON DELETE CASCADE
       • contract_sum numeric DEFAULT 0
       • retainage_pct numeric DEFAULT 5
       • status text DEFAULT 'draft' ('draft' | 'active' | 'locked')
       • contract_pdf_url text
       • tenant_id uuid DEFAULT get_user_tenant_id() FK tenant_config
       • timestamps + updated_at trigger
     - public.billing_schedule_lines
       • id uuid PK
       • billing_schedule_id uuid FK billing_schedule ON DELETE CASCADE
       • line_code text (customer's identifier, e.g. "A.1")
       • description text NOT NULL
       • scheduled_value numeric DEFAULT 0
       • is_change_order boolean DEFAULT false
       • co_number integer
       • ordinal integer DEFAULT 0
       • tenant_id + timestamps + updated_at trigger

   New columns on existing tables:
     - call_log.archive_record_id (uuid, nullable, FK archive.legacy_records)
     - proposals.archive_record_id (uuid, nullable, FK archive.legacy_records)
     - invoice_lines.billing_schedule_line_id (uuid, nullable, FK new table)
     - invoice_lines.description (text, nullable) — for per-period notes
     - invoices.retainage_pct, retainage_amount, retainage_released

   Altered: invoice_lines.proposal_wtc_id → now nullable (was NOT NULL).
   Historical rows unaffected.

   RLS: 4 standard policies per new table (select/insert/update/delete)
   gating on tenant_id = get_user_tenant_id(), role=authenticated.
   Matches the v69 lockdown pattern.

   Gotcha discovered during apply: CLAUDE.md claimed
   invoices.proposal_id was int8 FK proposals, but proposals.id is
   actually text. FK constraint failed on first push; fixed migration
   to use text. ALSO patched CLAUDE.md in same commit.

2. PHASE 2 — BILLING SCHEDULE UI SECTION (ProposalDetail)

   New component: src/components/BillingScheduleSection.jsx (~440 lines).
   Rendered as a full-width section below the 2-column grid on
   ProposalDetail, between the grid and the Internal Approve modal.

   States:
     - Loading spinner
     - Empty (no schedule): + Create Billing Schedule button
     - Active: three-tile header (Contract PDF · Contract Sum · Retainage %)
       + SOV line items table with inline add + inline edit rows
     - Locked: read-only banner, all edit controls hidden, CO add still
       allowed

   Behaviors:
     - Contract PDF upload → job-attachments bucket, path scoped to
       proposal.id; filename sanitized per CLAUDE.md storage rules
     - Contract Sum is auto-computed from sum of line scheduled_values
       and persisted on every mutation (no DB trigger; app updates)
     - Retainage % editable inline, blur-to-save
     - Lines table columns: #, Code, Description, Scheduled Value, CO tag,
       Edit/× actions
     - CO lines render with a purple "CO{n}" tag (matches v70 CO tag style)
     - Lock auto-detected by querying invoices where proposal_id = p.id
       AND deleted_at IS NULL; any existing invoice locks the schedule
     - Also locked if schedule.status = 'locked'

3. ROLE GATING — Sales upload only, Admin/Manager full access

   Per user: the Billing Schedule is an Admin/Manager task. Sales reps
   only upload the signed customer contract PDF; they must not edit
   amounts, retainage, lines, or create/delete the schedule.

   Implementation:
     - BillingScheduleSection accepts teamMember prop
     - canManage = ["Admin","Manager"].includes(teamMember.role)
     - When !canManage: entire section collapses to a "Contract Upload"
       card with upload/replace/view controls only; no other controls
     - When schedule doesn't exist yet AND user is Sales: shows
       "An Admin or Manager will create one after the customer's
       contract comes back."

   Gating is UI-only. RLS still permits any authenticated tenant user
   to read/write (needed for Sales to upload the PDF).

   Saved as a feedback memory: feedback_role_gating.md — applies to any
   future feature that touches money structures.
