===============================================================================
GOTCHAS / LESSONS
===============================================================================

1. DO NOT SET `thinking: { type: "adaptive" }` WHEN `tool_choice`
   FORCES A SPECIFIC TOOL.
   API returns 400: "Thinking may not be enabled when tool_choice
   forces tool use". The extract-sov fn had this bug initially;
   fixed in e091696. If you add extended thinking to another forced-
   tool call, you'll hit the same thing.

2. PDF base64 encoding in Deno — chunk at 0x8000 bytes when using
   `String.fromCharCode(...new Uint8Array(...))` or you'll stack-
   overflow on large contracts. See extract-sov/index.ts.

3. USE STRICT TOOL_USE FOR STRUCTURED EXTRACTION — we tried
   output_config.format initially, switched to strict tool schema +
   forced tool_choice. More reliable, and gives usage tokens back
   cleanly.

4. CUSTOMER DEDUP on archive import is case-insensitive EXACT, not
   fuzzy. Intentional — fuzzy would risk false merges.

5. MULTI-DOC ARRAY APPROACH WORKED — user pushed back hard on the
   initial "new billing_schedule_documents table" proposal ("massive
   over kill for the ability to have multiple pdf"). Text[] column
   + backfill was correct call. Lesson saved:
   feedback_build_right.md still applies, but simple > clean when
   the domain is "just a list of file URLs".

6. CLAUDE.md STILL SHOWS invoices.proposal_id AS int8 (was text
   in v72). Didn't touch. Verify against pg_catalog if Phase 3
   invoice code needs that FK type.

7. WIZARD'S CUSTOMER DEDUP fetches all customers without pagination.
   Will break at >1000 rows (project_supabase_row_limit.md). Fine
   for now — will need fetchAll() from supabaseHelpers.js eventually.
