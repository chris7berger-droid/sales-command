FIXES SHIPPED — IN COMMIT ORDER
===============================================================================

1. ARCHIVE FILTER DROPDOWNS — DISTINCT VIA RPC (commit 72db832 + migration)
   THE PROBLEM:
     ArchiveSearchView loaded up to 1000 rows from archive.legacy_records
     twice on mount (one call per filter field), then reduced client-side
     to get ~5 unique values per dropdown. Wrong shape of query, hit the
     PostgREST row cap, and wasted bandwidth on a feature that populates
     a dropdown.

   THE FIX:
     New archive.get_filter_options() RPC returns both filter-option sets
     (sources + types) in a single round-trip, tenant-scoped via
     archive.get_user_tenant_id() (the helper that was repaired in
     v77's 20260416230000_archive_rls_fix.sql). SECURITY DEFINER + locked
     search_path, EXECUTE granted only to authenticated.

   SHAPE:
     RETURNS jsonb — { sources: [...], types: [...] }.
     Uses DISTINCT subqueries with ORDER BY wrapped in jsonb_agg to keep
     sort deterministic.

   WIRING:
     archiveDb in src/lib/supabase.js gained a .rpc() method alongside
     .from(). Same PostgrestClient factory, same auth-token closure
     pattern, same archive-schema routing.

   RESULT:
     Mount-time archive search: 2 list fetches → 1 aggregated RPC.
     Single tenant still, but correct-by-construction for multi-tenant.

2. CUSTOMERS DETAIL PAGINATION (commit 9f5272c)
   THE PROBLEM:
     CustomerDetail load() in Customers.jsx:232 used direct
     supabase.from() for call_log and proposals (both scoped by
     customer_id). A customer with 1000+ jobs or 1000+ proposals would
     silently lose rows. Very unlikely today but completes the e50a438
     pagination sweep from v77.

   THE FIX:
     Both calls swapped to fetchAll() from src/lib/supabaseHelpers.
     The tricky bit was the proposals query — it uses a nested inner-join
     filter (call_log!inner + eq("call_log.customer_id", customer.id)).
     fetchAll forwards filter args verbatim, so:
       filters: [["is", "deleted_at", null], ["eq", "call_log.customer_id", customer.id]]
     works identically to the direct-call chain.

   VERIFIED:
     Loaded Lepori Construction on localhost — 4 jobs tab, 4 proposals
     tab both populate correctly. The nested-join-via-fetchAll was the
     one thing I was unsure about; confirmed before push.

3. BILLING CONTRACT UPLOAD TENANT GUARD (commit 938b280)
   THE PROBLEM:
     BillingScheduleSection.uploadContract() wrote to bucket path
     `${proposal.id}/contract-...` in job-attachments. The DB
     update that follows is RLS-protected, but the STORAGE upload
     itself is not tenant-scoped at the client. A tampered proposal
     prop could land a file under another tenant's folder — the DB
     row update would fail, but an orphaned file would remain in a
     tenant folder it doesn't belong to.

   THE FIX:
     Before touching storage, uploadContract() now:
       a) Fetches caller's tenant_id from tenant_config (RLS-scoped,
          returns only the caller's own row).
       b) Confirms the proposal row resolves under that tenant_id with
          an explicit .eq("tenant_id", userTenant).
       c) Aborts with alert("Upload blocked: proposal tenant mismatch.")
          if either lookup comes back empty.

   WHY EXPLICIT eq() NOT JUST RLS:
     v77 specifically called this pattern belt-and-suspenders. RLS could
     be misconfigured; the explicit filter is a second line of defense
     that makes the tenant check obvious at the call site.

   VERIFIED:
     Uploaded a contract PDF on localhost — instant success, existing
     flow unchanged. The new guard adds 2 round-trips before upload,
     acceptable cost on a file-upload path.
