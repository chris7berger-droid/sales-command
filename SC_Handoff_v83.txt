================================================================
WHAT WAS DONE
================================================================

Closed audit findings (2026-04-26 audit):
  - Critical #1: anon UPDATE on any signable proposal
  - Critical #2: anon SELECT on every signable proposal + children
  - High #4 (signature insert leg): anon INSERT into proposal_signatures
  - Equivalent leak on invoices + invoice_lines (viewing_token broad policy)

Approach: REPLACED broad "signing_token IS NOT NULL" anon policies with
policies that require the caller to present the matching token via a
custom HTTP request header (x-signing-token / x-viewing-token).

Frontend code that queries these tables now uses a dedicated public
Supabase client that pins the token in headers. Existing query logic
unchanged — minimal surgical edit.

LESSONS / RUNBOOK (re-usable pattern for future RLS work)
================================================================

The 6-gate sequence used here is the template:
  1. Build all changes on a branch (do NOT touch main)
  2. Vercel preview deploy (free safety net — DO NOT skip)
  3. Sign/test on preview URL in incognito (real anon conditions)
     -- if Vercel preview-protection is on, temporarily disable
        Vercel Authentication for preview deploys; re-enable after
  4. Merge PR — Vercel deploys frontend; old policies still active
  5. Test on PRODUCTION before tightening any DB policies
  6. Apply additive migration (new policies alongside old)
  7. Test on production again (overlap window — both sets active)
  8. Apply drop migration (old policies removed)
  9. Test on production a third time (strict enforcement only)
  10. Commit drop migration + rollback to main as a record

Critical: never skip the preview test. Yesterday's failure happened
because preview testing was skipped "to save time."

Critical: each migration must have a paired rollback file in sql/
written BEFORE applying.

Critical: SQL pasted into Supabase SQL Editor sometimes loses the
first line's "-- " comment marker on copy/paste — manually verify
the first line is a comment before clicking Run.

Critical: copy/paste from rendered chat sometimes injects markdown
link formatting into table names like [public.call](http://public.call)_log.
Inspect for brackets and URLs before running SQL.

CROSS-REPO IMPACT
================================================================

Tables affected (proposals, proposal_wtc, proposal_recipients,
proposal_signatures, invoices, invoice_lines, call_log) are shared
across all 4 Command Suite repos.

Re-grep sch-command, field-command, AR-Command-Center for direct
anon queries against these tables before next deploy of any of those.
If they query as anon WITHOUT the token header, those queries will
now return empty / fail. (Audit on 2026-04-26 said grep was clean,
but verify again at next deploy time.)

If they only access these tables via authenticated sessions OR via
edge functions using service_role, they are unaffected.
