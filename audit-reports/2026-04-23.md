## Critical findings

### 1. Missing role check on `invite-user` edge function — privilege escalation
- **File:** `supabase/functions/invite-user/index.ts:37-45, 80-85`
- **Category:** Client-side auth assumption / missing backend check
- **Evidence:**
  ```
  const { data: { user }, error: authErr } = await supabase.auth.getUser(token);
  if (authErr || !user) { return 401; }
  const { email, name, teamMemberId, ... } = await req.json();
  // No role check here
  await supabase.auth.admin.createUser({ email, email_confirm: true });
  // Attacker can overwrite ANY team_members row's auth_id — service role bypasses RLS
  await supabase.from("team_members").update({ auth_id: authId }).eq("id", teamMemberId);
  ```
- **Why it matters:** Any authenticated sales rep (even on a different tenant) can call this with `teamMemberId = <admin's team_member id>` and `email = attacker@x.com`. Because the edge function uses the service-role key, the `team_members.update` bypasses RLS and rewrites the admin's `auth_id` to the attacker's new Supabase user — full account takeover of that admin's tenant.
- **Fix:** Add `is_admin_or_manager()` verification by calling `supabase.rpc("is_admin_or_manager")` after validating the caller JWT, and scope `teamMemberId` updates by caller's tenant.
- **Effort:** 30 min

### 2. Missing role check on `delete-user` / `deactivate-user`
- **File:** `supabase/functions/delete-user/index.ts:36-80`, `supabase/functions/deactivate-user/index.ts:36-90`
- **Category:** Missing authorization
- **Evidence:**
  ```
  const { data: { user }, error: authErr } = await supabase.auth.getUser(token);
  if (authErr || !user) { return 401; }
  const { teamMemberId } = await req.json();
  // No role check, no tenant scoping
  await supabase.auth.admin.deleteUser(member.auth_id);
  await supabase.from("team_members").delete().eq("id", teamMemberId);
  ```
- **Why it matters:** Any authenticated user can delete or deactivate any team member in any tenant — including other tenants' admins. Service-role bypass + missing tenant scope = cross-tenant destructive action.
- **Fix:** Require `is_admin_or_manager()`; verify the target `team_members.tenant_id === caller's tenant_id` before deletion.
- **Effort:** 30 min

### 3. `proposal-signed` edge function — no auth, no signing-token verification, CORS `*`
- **File:** `supabase/functions/proposal-signed/index.ts:8-43`
- **Category:** Missing authentication / CORS
- **Evidence:**
  ```
  const corsHeaders = { "Access-Control-Allow-Origin": "*", ... };
  // No auth header check
  const { repEmail, ..., proposalId, callLogId } = await req.json();
  if (proposalId) {
    const sb = createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY);
    await sb.from("proposals").update({ status: "Sold", approved_at: now }).eq("id", proposalId);
    await sb.from("call_log").update({ stage: "Sold" }).eq("id", callLogId);
  }
  // Then emails arbitrary repEmail via Resend
  ```
- **Why it matters:** Unauthenticated attacker (CORS `*`) can POST any `{proposalId, callLogId}` and flip the proposal to `Sold` plus `approved_at=now`, corrupting financial state, billing pipeline, and QuickBooks sync. Also abuse vector for free email send via Resend to arbitrary addresses (phishing). No signing_token check ties the flip to actual customer consent.
- **Fix:** Require a valid `signing_token` for the proposal (look up row, verify token matches a field in the request), and tighten CORS to the allowed origins.
- **Effort:** 1 hour

### 4. Anon RLS on `proposals` — all sent proposals readable & mutable without knowing the token
- **File:** `sql/rls_core_tables.sql:138-147`
- **Category:** RLS posture
- **Evidence:**
  ```
  CREATE POLICY "proposals_public_sign" ON public.proposals
    FOR SELECT TO anon USING (signing_token IS NOT NULL);
  CREATE POLICY "proposals_public_sign_update" ON public.proposals
    FOR UPDATE TO anon USING (signing_token IS NOT NULL) WITH CHECK (signing_token IS NOT NULL);
  ```
- **Why it matters:** Any anon caller hitting `/rest/v1/proposals` (no filter) gets every row with a signing token — customer names, totals, job numbers, addresses — across all tenants. The UPDATE policy additionally lets anon rewrite `total`, `status`, `approved_at`, `customer`, etc. on every sent proposal. Tenant isolation collapses for the first licensee (HDSP) PII.
- **Fix:** Move anon access for the signing page behind an edge function that takes the token as input, looks up the single row with service-role, and returns only allowed fields. Or add a policy predicate that compares to a per-request header/RPC parameter.
- **Effort:** 2 hours

### 5. Anon RLS on `invoices` / `invoice_lines` / `proposal_wtc` — similar enumeration
- **File:** `sql/rls_core_tables.sql:190-192`, `sql/rls_child_tables.sql:172-180, 293-301`
- **Category:** RLS posture
- **Evidence:**
  ```
  CREATE POLICY "invoices_public_view" ON public.invoices
    FOR SELECT TO anon USING (viewing_token IS NOT NULL);
  ```
  `invoices.viewing_token` defaults to `gen_random_uuid()` per CLAUDE.md → every invoice has it → every invoice row is anon-readable.
- **Why it matters:** Entire invoice history + invoice_lines (amounts, descriptions) + proposal_wtc (materials, burden rates, markup, labor hours) is readable by any anon caller. Combined with finding #4 this is a full dump of financials for every tenant.
- **Fix:** Same remediation as #4 — route public invoice/signing views through edge functions keyed on the token.
- **Effort:** bundled with #4

### 6. Stripe webhook signature verification silently skipped when env var missing
- **File:** `supabase/functions/stripe-webhook/index.ts:67-73`, `supabase/functions/scc-stripe-webhook/index.ts:56-62`
- **Category:** Webhook signature
- **Evidence:**
  ```
  if (STRIPE_WEBHOOK_SECRET) {
    const valid = await verifyStripeSignature(body, sigHeader, STRIPE_WEBHOOK_SECRET);
    if (!valid) return new Response("Invalid signature", { status: 400 });
  }
  // else: fall through, event is trusted
  ```
- **Why it matters:** If the env var is ever unset (rotation mishap, deploy miscopy, new env), Stripe mode continues to process forged `checkout.session.completed` and subscription events. An attacker can mark any invoice Paid, activate any tenant's subscription, or toggle `subscription_status` fields. Stripe is in LIVE mode — real money flow.
- **Fix:** Treat missing secret as fatal: `if (!STRIPE_WEBHOOK_SECRET) return 500`. Always require `stripe-signature` header & constant-time comparison (current code uses `===`; move to `timingSafeEqual`).
- **Effort:** 15 min

## High findings

### 7. QB OAuth tokens stored unencrypted in `qb_connection`
- **File:** `supabase/functions/qb-auth/index.ts:92-98, 152-157`; `sql/rls_identity_tables.sql:99-103`
- **Category:** QuickBooks security
- **Evidence:** `access_token` + `refresh_token` inserted/updated as plain text. RLS blocks anon + authenticated, only service role reads them — so any edge function with service-role (most of them) can read QB tokens.
- **Why it matters:** Tokens grant full access to the tenant's QuickBooks books (invoices, customers, payments). A compromise of any edge function or service-role leak exposes every connected tenant's QB account. Industry norm is application-level encryption (KMS/libsodium).
- **Fix:** Encrypt `access_token` + `refresh_token` at the application layer using a Deno KV-stored key or Supabase Vault extension.
- **Effort:** 2 hours

### 8. QB token exchange/refresh response body logged in full on failure
- **File:** `supabase/functions/qb-auth/index.ts:79, 143`
- **Category:** Sensitive console log
- **Evidence:**
  ```
  console.error("qb-auth: token exchange failed:", JSON.stringify(tokenData));
  ```
- **Why it matters:** On non-2xx responses, Intuit still returns response bodies; in some error shapes the original request credentials or partial token state can appear. Edge function logs persist in Supabase dashboard / any log aggregation.
- **Fix:** Log only `tokenRes.status` + `tokenData.error`, never the full body.
- **Effort:** 5 min

### 9. `tenant_config.public_read USING (true)` exposes every tenant's financials to anon
- **File:** `sql/rls_identity_tables.sql:87-89`
- **Category:** RLS posture
- **Evidence:**
  ```
  CREATE POLICY "tenant_config_public_read" ON public.tenant_config
    FOR SELECT TO anon USING (true);
  ```
- **Why it matters:** Anon `/rest/v1/tenant_config` returns every tenant row including `default_burden_rate`, `default_ot_burden_rate`, `monthly_billing_goal`, `yearly_billing_goal`, `stripe_customer_id`, `stripe_subscription_id`, `subscription_status` — sensitive internal cost & business-metric data exposed to the open internet, cross-tenant.
- **Fix:** Restrict the anon policy to a minimal column set via a view (e.g., `tenant_public`) exposing only `company_name, logo_url, phone, email`, and drop the raw-table policy. Or migrate public pages to use signed public URLs via edge function.
- **Effort:** 45 min

### 10. CORS `Access-Control-Allow-Origin: *` on Stripe webhooks and `create-billing-session` / `proposal-signed`
- **File:** `supabase/functions/stripe-webhook/index.ts:56`, `supabase/functions/scc-stripe-webhook/index.ts:45`, `supabase/functions/create-billing-session/index.ts:20`, `supabase/functions/proposal-signed/index.ts:10`
- **Category:** CORS
- **Evidence:** `"Access-Control-Allow-Origin": "*"` literal.
- **Why it matters:** Stripe webhooks are server-to-server so `*` is acceptable there, but `create-billing-session` (authenticated, uses `credentials`-bearing auth tokens) and `proposal-signed` (see finding #3) should be origin-restricted to the same allowlist as the other functions.
- **Fix:** Reuse the `ALLOWED_ORIGINS + .endsWith('.vercel.app')` pattern from `invite-user`.
- **Effort:** 15 min

### 11. Vercel.app wildcard in CORS allowlist
- **File:** `supabase/functions/invite-user/index.ts:12` (and 15 other functions)
- **Category:** CORS
- **Evidence:** `const isAllowed = ALLOWED_ORIGINS.includes(origin) || origin.endsWith(".vercel.app");`
- **Why it matters:** Any attacker-controlled vercel.app preview URL (anyone can deploy to `*.vercel.app` for free) can make CORS requests to these functions with user credentials, enabling authenticated-session CSRF. Combined with the missing role checks (findings #1–#3), this widens the attack surface to drive-by phishing: a victim admin visits an attacker's vercel.app page which issues `fetch(..., { credentials: 'include' })` to `invite-user` / `delete-user`.
- **Fix:** Replace the `.vercel.app` fallback with a specific preview subdomain allowlist (e.g., `sales-command-*.vercel.app`), or remove entirely and rely only on the production origins.
- **Effort:** 15 min

## Medium findings

### 13. `create-billing-session` queries `tenant_config` with `LIMIT 1` instead of caller's tenant
- **File:** `supabase/functions/create-billing-session/index.ts:64-68`
- **Category:** Multi-tenant isolation bug
- **Evidence:** `.from("tenant_config").select("*").limit(1).single()` — returns whichever tenant_config row comes first.
- **Why it matters:** When SaaS gains a second tenant, any authenticated user in any tenant creates/manages the FIRST tenant's Stripe subscription. Immediate consequence once multi-tenant deployment happens.
- **Fix:** Look up tenant_id via `team_members.auth_id = user.id`, then scope `.eq("id", tenant_id)`.
- **Effort:** 20 min

### 14. Anon UPDATE policies use over-permissive `WITH CHECK (true)`
- **File:** `sql/rls_core_tables.sql:105-107`, `sql/rls_child_tables.sql:235-236`
- **Category:** RLS posture
- **Evidence:**
  ```
  CREATE POLICY "call_log_public_sign_update" ON public.call_log
    FOR UPDATE TO anon USING (...) WITH CHECK (true);
  ```
- **Why it matters:** `WITH CHECK (true)` means an anon signer can overwrite ANY column on `call_log` / `proposal_recipients` — e.g., change `sales_name`, `customer_name`, `billing_address`, etc. The app only intends to update `stage="Sold"` / `viewed_at`. Combined with #4 (enumeration), an attacker edits every signable job's metadata.
- **Fix:** Constrain `WITH CHECK` to verify that only the intended fields changed (via `OLD`-style triggers or by moving these updates into an edge function).
- **Effort:** 1 hour

### 15. `is_admin_or_manager()` not enforced on `tenant_config` updates
- **File:** `sql/rls_identity_tables.sql:80-83`
- **Category:** Role check
- **Evidence:** `tenant_config_update` allows any authenticated user in the tenant to update company_name, Stripe IDs, billing goals, etc.
- **Why it matters:** A non-admin sales rep can rewrite tenant config (logo, license number, burden rates → affects future calculations) directly via the REST API.
- **Fix:** Add `AND public.is_admin_or_manager()` to the USING/WITH CHECK clauses.
- **Effort:** 10 min

### 16. `.gitignore` doesn't cover all `.env*` variants
- **File:** `.gitignore:13,26`
- **Category:** .env hygiene
- **Evidence:** Current entries are `.env*.local` and `.env.prod`. Plain `.env`, `.env.production`, `.env.development`, `.env.staging` are NOT ignored.
- **Why it matters:** A dev running `cp .env.example .env` followed by `git add .` silently tracks production credentials. Stripe live + Supabase service-role keys land in git history.
- **Fix:** Replace both lines with a single `*.env` + `.env` + `.env.*` block, or use the recommended:
  ```
  .env
  .env.*
  !.env.example
  ```
- **Effort:** 2 min
