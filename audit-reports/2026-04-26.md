## Critical findings

### Anon role can UPDATE any signable proposal without knowing the token
- **File:** `sql/rls_core_tables.sql:144`
- **Category:** RLS posture — over-permissive USING clause
- **Evidence:**
  ```sql
  CREATE POLICY "proposals_public_sign_update" ON public.proposals
    FOR UPDATE TO anon
    USING (signing_token IS NOT NULL)
    WITH CHECK (signing_token IS NOT NULL);
  ```
- **Why it matters:** The publishable anon key ships in the React bundle (`src/lib/supabase.js:5`). With it, anyone can issue `PATCH /rest/v1/proposals?status=eq.Sent` with no `.eq("signing_token", ...)` filter and mutate **every** sent proposal in HDSP's DB — change `status`, `total`, `intro`, `wtc_verified`, etc. The intent was that the client *adds* `.eq(signing_token, token)`, but RLS does not enforce that. Same flaw in `call_log_public_sign_update` (`sql/rls_core_tables.sql:98-107`) and `proposal_recipients_public_update` (`sql/rls_child_tables.sql:227-236`).
- **Fix:** Replace `USING (signing_token IS NOT NULL)` with a per-row token match — e.g. require the caller to pass the token via a header or PostgREST RPC, or store the token in `request.headers ->> 'x-signing-token'` and policy on `signing_token = current_setting('request.jwt.claims', true)::json->>'token'`. The simplest correct fix: move all signing-page mutations behind the existing `proposal-signed` edge function (which already verifies the token at `supabase/functions/proposal-signed/index.ts:36`) and drop the anon UPDATE policies entirely.
- **Effort:** 2 hours

### Anon role can SELECT every signable proposal + WTC + recipients + signatures
- **File:** `sql/rls_core_tables.sql:138` (and `sql/rls_child_tables.sql:172, 209` etc.)
- **Category:** RLS posture — PII / financial data exposure
- **Evidence:**
  ```sql
  CREATE POLICY "proposals_public_sign" ON public.proposals
    FOR SELECT TO anon
    USING (signing_token IS NOT NULL);
  ```
  Same shape on `proposal_wtc_public_read`, `proposal_recipients_public_update` (USING), `invoice_lines_public_read` (`sql/rls_child_tables.sql:293`), `invoices_public_view` (`sql/rls_core_tables.sql:190`).
- **Why it matters:** Same threat surface as above — issuing `GET /rest/v1/proposals?select=*` with the anon key returns every proposal ever sent for signature, including customer name, totals, sales rep, signing token, full SOW, materials cost, markup, burden, discount. Once a token is harvested, the attacker has download/print PDF + can call `proposal-signed` (the edge fn now verifies the token, but the data has already leaked). The `viewing_token IS NOT NULL` policy on `invoices` is the same pattern — leaks customer billing PII and amounts.
- **Fix:** Same as above — gate anon reads behind a SECURITY DEFINER RPC that requires the token as an argument and returns a single row, or check the token via header/JWT claim inside the policy. The PublicSigningPage and PublicInvoicePage clients still pass `.eq("signing_token", token)`, so the token is never broadcast — the leak is purely the missing server-side enforcement.
- **Effort:** 2 hours

## High findings

### `tenant_config` exposes business financials to anyone with the anon key
- **File:** `sql/rls_identity_tables.sql:87-89`
- **Category:** RLS posture
- **Evidence:**
  ```sql
  CREATE POLICY "tenant_config_public_read" ON public.tenant_config
    FOR SELECT TO anon
    USING (true);
  ```
- **Why it matters:** Comment claims "no sensitive data exposed (tokens managed by edge fns)" but the table also holds `default_burden_rate`, `default_ot_burden_rate`, `default_tax_rate`, `monthly_billing_goal`, `yearly_billing_goal`, `conversion_rate_goal`, `proposals_sent_goal`, `stripe_customer_id`, `stripe_subscription_id`, `subscription_status`. Anon SELECT returns all of those — competitor recon, internal margin disclosure, and Stripe customer ID enumeration.
- **Fix:** Replace `USING (true)` with an explicit column-restricted view (e.g. `tenant_config_public` exposing only `company_name, tagline, logo_url, license_number, phone, email, website, address, city, state, zip`) and grant anon read on that view; drop the `USING (true)` policy on the base table.
- **Effort:** 30 min

### Anon-readable team_members exposes every employee email + phone
- **File:** `sql/rls_identity_tables.sql:60-62`
- **Category:** RLS posture / PII
- **Evidence:**
  ```sql
  CREATE POLICY "team_members_public_read" ON public.team_members
    FOR SELECT TO anon
    USING (active = true);
  ```
- **Why it matters:** Anon `GET /rest/v1/team_members?select=name,email,phone,role` returns every active rep and admin, weaponizing the bundle's anon key into an employee directory for phishing / credential-stuffing.
- **Fix:** Move the rep-by-name lookup behind a SECURITY DEFINER RPC (`get_rep_contact(name text)`) returning a single row, and replace this policy with an empty policy set or a strictly-scoped one that requires a proposal/invoice token as input.
- **Effort:** 1 hour

### Anon can INSERT proposal_signatures for any signable proposal
- **File:** `sql/rls_child_tables.sql:264-272`
- **Category:** RLS posture
- **Evidence:**
  ```sql
  CREATE POLICY "proposal_signatures_public_insert" ON public.proposal_signatures
    FOR INSERT TO anon
    WITH CHECK (EXISTS (SELECT 1 FROM proposals p
        WHERE p.id::text = proposal_id::text
          AND p.signing_token IS NOT NULL));
  ```
- **Why it matters:** No token match — anyone can spray bogus signature rows (`signer_name`, `signer_email`, `ip_address`, `pdf_url`) onto any sent proposal, polluting the audit trail used for legal / chargeback defense. Combined with the SELECT leak above, an attacker can also overwrite/forge a signature record before the legitimate signer arrives.
- **Fix:** Drop the anon INSERT policy; require all signature inserts to flow through the `proposal-signed` edge function, which already validates the token (`supabase/functions/proposal-signed/index.ts:31-41`).
- **Effort:** 30 min

### `xlsx` 0.18.5 — Prototype Pollution + ReDoS, no patch
- **File:** `package.json:11`
- **Category:** Dependency CVE
- **Evidence:** `npm audit` reports HIGH `xlsx`: GHSA-4r6h-8v6p-xvw6 (Prototype Pollution, fixed in 0.19.3) and GHSA-5pgg-2g8v-p4x9 (ReDoS, fixed in 0.20.2). `package.json` pins `^0.18.5`; `fixAvailable: false` in npm registry.
- **Why it matters:** `Import/` page parses user-supplied .xlsx in the browser. A crafted file can pollute `Object.prototype` or hang the tab. Risk to operators using the import wizard.
- **Fix:** Switch to the SheetJS CDN-distributed package `https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz` (or the maintained fork `@e965/xlsx`). Pin to `>= 0.20.2`.
- **Effort:** 30 min

## Medium findings

### Stripe webhook handlers and `proposal-signed` use `Access-Control-Allow-Origin: *`
- **File:** `supabase/functions/stripe-webhook/index.ts:58`, `supabase/functions/scc-stripe-webhook/index.ts:47`, `supabase/functions/proposal-signed/index.ts:10`, `supabase/functions/create-billing-session/index.ts:20`
- **Category:** CORS
- **Evidence:** `"Access-Control-Allow-Origin": "*"` returned on every response (not just OPTIONS).
- **Why it matters:** Stripe webhooks are fine — Stripe doesn't use a browser. `proposal-signed` and `create-billing-session` are called from the browser with a JWT/signing-token in the body, so a wildcard ACAO + no credentials means a malicious page can replay calls but cannot read the response set against another user's session. Lower than it looks but inconsistent with the per-origin allowlist used by `qb-auth`, `delete-user`, `invite-user`, etc.
- **Fix:** Mirror the `ALLOWED_ORIGINS` allowlist used elsewhere (`supabase/functions/qb-auth/index.ts:12-21`) on these four functions.
- **Effort:** 30 min

### `reset-password` is unauthenticated and unbounded
- **File:** `supabase/functions/reset-password/index.ts:43`
- **Category:** DoS / abuse surface
- **Evidence:** `await supabase.auth.admin.listUsers();` — full table scan on every public call; no captcha, no rate limiting, no IP allowlist.
- **Why it matters:** Anyone on the internet can fire it in a loop to (a) exhaust the admin API quota, (b) trigger Resend email sends to confirmed customers, and (c) enumerate users by timing — `listUsers()` returns the first page only (default 50), so the function silently fails to find users beyond page 1, behaving differently from "real not found."
- **Fix:** Replace `listUsers()` with `admin.getUserByEmail(email)` (or paginate explicitly), and gate behind a hCaptcha/Turnstile token validated server-side. Add per-IP rate limiting in Supabase function config.
- **Effort:** 2 hours

### `dompurify`, `postcss`, `brace-expansion` MODERATE CVEs

- **Category:** Dependency CVE
- **Evidence:** `dompurify <=3.3.3` (4 advisories incl. CWE-79 XSS bypass), `postcss <8.5.10` (XSS via unescaped `</style>`), `brace-expansion <1.1.13` (ReDoS).

### Console logs include rep emails and signing/QB-result payloads
- **File:** `src/pages/PublicSigningPage.jsx:350,364`, `src/pages/QBCallbackPage.jsx:25`, `src/components/ProposalDetail.jsx:511,569`, `supabase/functions/stripe-webhook/index.ts:90,134`, `supabase/functions/qb-auth/index.ts:59,76,124,140`
- **Category:** Sensitive log
- **Evidence:** `console.log("Calling proposal-signed edge function", { proposalId, callLogId, repEmail, salesName });` (PublicSigningPage:350) — written to public-page DevTools for any visitor including the customer signing.

## Low / hygiene findings

### `.gitignore` does not block `.env`, `.env.production`, `.env.development`
- **File:** `.gitignore:24-25`
- **Category:** `.env` hygiene
- **Evidence:** Only `.env*.local` and `.env.prod` are ignored. A developer running `cp .env.local .env` (a common mistake) would be able to commit the file.
- **Why it matters:** No tracked .env files today (`git ls-files | grep ^.env` returns nothing), so this is preventive.
- **Fix:** Add `.env`, `.env.local`, `.env.development`, `.env.production`, `.env*` to `.gitignore` and ship a tracked `.env.example`.
- **Effort:** 5 min

### `qb_connection` stores OAuth refresh tokens in plaintext
- **File:** `supabase/functions/qb-auth/index.ts:92-98`, schema in CLAUDE.md
- **Category:** QuickBooks security
- **Evidence:** `access_token`, `refresh_token` written and read as plain text columns. RLS denies anon + authenticated (only service role bypass), so client cannot read them — but a `pg_dump` or backup leak exposes long-lived QB refresh tokens (180-day window).
- **Why it matters:** Defense-in-depth: an attacker with a DB snapshot can persist QB access until tokens are revoked. Not exploitable today.
- **Fix:** Encrypt both columns with `pgsodium` or Vault, decrypting only inside the edge function. Alternatively store under Supabase Vault (`vault.secrets`).
- **Effort:** 4 hours

### Repo root has miscellaneous `fix_*.py` scripts and 60+ handoff text files

- **Effort:** 30 min
