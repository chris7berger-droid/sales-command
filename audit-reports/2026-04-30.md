#### M1. Hardcoded Supabase URL and publishable key fallback in public client
- **File:** `src/lib/supabasePublic.js:14-20`
- **Evidence:**
  ```js
  const SUPABASE_URL =
    import.meta.env.VITE_SUPABASE_URL ||
    "https://pbgvgjjuhnpsumnowuym.supabase.co";

  const SUPABASE_KEY =
    import.meta.env.VITE_SUPABASE_ANON_KEY ||
    "sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK";
  ```
- **Issue:** Production project URL + a `sb_publishable_*` key are baked in as fallbacks. Although the publishable key is intended to be browser-exposed (so this is not a service_role leak), hardcoding pins the project into the bundle, defeats env rotation, and contradicts the validation pattern in `src/lib/supabase.js`. Drift exists with `Login.jsx` (different literal value), so at least one key is stale.
- **Specs:** CLAUDE.md (env-var hygiene), OWASP-ASVS-V14.3, OWASP-A05.
- **Fix:** Remove the `||` literal fallbacks; throw on missing env (mirror `src/lib/supabase.js`).

#### M2. Hardcoded Supabase URL and publishable key fallback in Login reset flow
- **File:** `src/pages/Login.jsx:57,60`
- **Evidence:**
  ```js
  `${import.meta.env.VITE_SUPABASE_URL || "https://pbgvgjjuhnpsumnowuym.supabase.co"}/functions/v1/reset-password`,
  …
  headers: { "Content-Type": "application/json", apikey: import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK" },
  ```
- **Issue:** Second copy of the URL+publishable-key fallback. Drift between this literal (`v7XktVvkAlX7y5f6xoFjng_AaLaWKoK`) and the one in `supabasePublic.js` (`LL6wYC7kJDPQfGYeKAaLaWKoK`) means at least one is stale/incorrect — a silent failure path if env vars ever go missing.
- **Specs:** CLAUDE.md, OWASP-ASVS-V14.3, OWASP-A05.
- **Fix:** Reuse the validated `supabase` client / a shared env helper instead of inlining literals.

### Low

#### L1. Edge function logs full Intuit OAuth error body
- **File:** `supabase/functions/qb-auth/index.ts:79,143`
- **Evidence:**
  ```ts
  console.error("qb-auth: token exchange failed:", JSON.stringify(tokenData));
  ```
- **Issue:** On non-OK Intuit responses, the full body is serialized to function logs. Normal failure responses don't contain access/refresh tokens, but a partial-success or malformed response could, and `intuit_tid` plus internal IDs that aid attackers reading log dumps will leak. Server-side only, not browser-exposed.
- **Specs:** OWASP-ASVS-V7.1, OWASP-A09.
- **Fix:** Log only `tokenData.error` / `tokenData.error_description` / status — not the full JSON.

#### L2. Edge functions log customer PII (emails, names) on every invocation
- **File:** `supabase/functions/invite-user/index.ts:95` (also `send-invoice/index.ts`, `proposal-signed/index.ts`, `send-pay-app/index.ts`)
- **Evidence:**
  ```ts
  console.log("invite-user invoked", { email, name, teamMemberId, senderEmail, senderName });
  ```
- **Issue:** Steady-state info-level logs include customer/recipient PII, widening blast radius of any Supabase admin compromise and complicating GDPR/data-deletion. Not a credential leak.
- **Specs:** OWASP-ASVS-V8.3, OWASP-A09.
- **Fix:** Strip emails/names from steady-state logs; log opaque IDs only, or gate behind a DEBUG env flag.
