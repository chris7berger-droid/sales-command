### Critical

#### C1. `call_log` anon UPDATE policy uses `WITH CHECK (true)` — token holder can rewrite any column
- **File:** `supabase/migrations/20260427180000_add_token_gated_policies.sql:171`
- **Evidence:**
  ```sql
  CREATE POLICY "call_log_public_sign_update_token" ON public.call_log
    FOR UPDATE TO anon
    USING ( EXISTS ( SELECT 1 FROM public.proposals p
            WHERE p.call_log_id = call_log.id
              AND p.signing_token IS NOT NULL
              AND p.signing_token::text = public.request_signing_token() ) )
    WITH CHECK (true);
  ```
- **Issue:** A signing-token holder can update ANY column of the linked `call_log` row — `tenant_id`, `customer_id`, addresses, `qb_customer_id`, billing fields. The USING clause confirms ownership of the row but `WITH CHECK (true)` does not constrain the new row, so an attacker can re-point the row to another tenant or poison billing data.
- **Specs:** CLAUDE_RLS.md, OWASP-ASVS-V4.1.5, OWASP-A01.
- **Fix:** Replace `WITH CHECK (true)` with the same EXISTS check as USING; better, move the stage update into a `SECURITY DEFINER` RPC that only updates `stage`.

#### C2. Legacy seed `sql/rls_core_tables.sql` ships the 2026-04-26 anti-pattern
- **File:** `sql/rls_core_tables.sql:138-140` (also `:144-147`, `:86-94`, `:98-107`, `:190-192`)
- **Evidence:**
  ```sql
  CREATE POLICY "proposals_public_sign" ON public.proposals
    FOR SELECT TO anon
    USING (signing_token IS NOT NULL);
  ```
- **Issue:** Although migration `20260429190000_drop_public_read_policies.sql` drops these in the live DB, the seed scripts under `/sql/` still ship the exact 2026-04-26 anti-pattern for `proposals` SELECT/UPDATE, `call_log` SELECT/UPDATE, and `invoices` SELECT. Re-running these seeds (fresh deploy, dev reset, sibling repo onboarding) silently re-introduces the incident: anon key holders read every row where the column is non-null without the header check.
- **Specs:** CLAUDE_RLS.md (the documented anti-pattern), OWASP-A01.
- **Fix:** Replace the bare `signing_token IS NOT NULL` clauses in `/sql/rls_core_tables.sql` and `/sql/rls_child_tables.sql` with the `request_signing_token()` / `request_viewing_token()` match pattern, or delete the legacy seed files so future runs use only `supabase/migrations/`.

#### C3. `proposal_recipients` anon UPDATE policy permits arbitrary column writes
- **File:** `supabase/migrations/20260427180000_add_token_gated_policies.sql:105-122`
- **Evidence:**
  ```sql
  CREATE POLICY "proposal_recipients_public_update_token" ON public.proposal_recipients
    FOR UPDATE TO anon
    USING ( EXISTS (...token-match...) )
    WITH CHECK ( EXISTS (...same token-match...) );
  ```
- **Issue:** `PublicSigningPage.jsx:92-96` does an unbounded UPDATE on `proposal_recipients` keyed only by `proposal_id`. Combined with this policy, any anon holder of a signing token can mutate `contact_email`, `contact_name`, `phone`, `role`, `customer_contact_id` on every recipient of that proposal. WITH CHECK only keeps the `proposal_id` token-bound; it does not restrict columns. Attackers can flip `role` from `viewer` to `signer` on themselves and hijack the signing flow.
- **Specs:** OWASP-ASVS-V4.1.5, OWASP-A01.
- **Fix:** Move `viewed_at` marking into a `SECURITY DEFINER` RPC `mark_recipient_viewed(token)` that updates only the `viewed_at` column for recipients of the matching proposal; drop the anon UPDATE policy.

#### C4. `qb-auth` lacks admin/role check; any authed user can wipe + replace company QuickBooks tokens
- **File:** `supabase/functions/qb-auth/index.ts:90-98`
- **Evidence:**
  ```ts
  await sb.from("qb_connection").delete().neq("id", "00000000-0000-0000-0000-000000000000");
  const { error: insertErr } = await sb.from("qb_connection").insert({
    realm_id: realmId, access_token: tokenData.access_token, refresh_token: tokenData.refresh_token, ... });
  ```
- **Issue:** Any authenticated user can call `action='exchange'` to delete ALL `qb_connection` rows globally (cross-tenant blast radius — service-role bypasses RLS) and replace them with attacker-controlled `realmId`/tokens. No admin check, no `tenant_id` binding, no per-tenant scoping in the table.
- **Specs:** OWASP-ASVS-V4.2, OWASP-A01.
- **Fix:** Require admin role via `is_admin_or_manager()`; scope `qb_connection` by `tenant_id` and target only the caller's tenant row.

#### C5. `qb-record-payment` has no `tenant_id` verification on invoice

#### C8. `send-invoice` has no tenant verification and trusts caller-supplied amount/email
- **File:** `supabase/functions/send-invoice/index.ts:44`
- **Evidence:**
  ```ts
  const { invoiceId, customerEmail, customerName, amount, jobName, jobId, dueDate, senderEmail, intro } = await req.json();
  ```
- **Issue:** No DB lookup verifying that `invoiceId` belongs to caller's tenant, and `amount`/`customerEmail` come straight from the request. Any authenticated user can create a Stripe Checkout session for any amount, send to any email, and overwrite `stripe_checkout_url` on any tenant's invoice (line ~124). `senderEmail` is also used as the Resend `From` with no domain allowlist (spoofing).
- **Specs:** OWASP-ASVS-V4.2, OWASP-A01, OWASP-A04.

#### C9. `send-pay-app` performs SSRF via caller-supplied PDF URLs and trusts cross-tenant ids
- **File:** `supabase/functions/send-pay-app/index.ts:134-137`
- **Evidence:**
  ```ts
  const [payAppRes, invoiceRes] = await Promise.all([
    fetch(payAppPdfUrl),
    fetch(invoicePdfUrl),
  ]);
  ```
- **Issue:** `payAppPdfUrl` and `invoicePdfUrl` come straight from the request body and are fetched with no allowlist. An attacker can probe internal/metadata URLs (`http://169.254.169.254/`, `http://kong:8000/`, etc.) and have the bytes base64-attached to an email under their control. `payAppId`/`invoiceId`/`recipientEmail` are also not bound to the caller's tenant, so cross-tenant pay apps can be marked submitted (line ~235) and invoices marked Sent (line ~244).
- **Specs:** OWASP-ASVS-V12.6, OWASP-A10, OWASP-A01.
- **Fix:** Restrict PDF URLs to a Supabase Storage allowlist (`https://<project>.supabase.co/storage/v1/object/public/...`) and verify tenant ownership of `payAppId`/`invoiceId`.

#### C10. `proposal-signed` flips arbitrary `call_log.stage` based on caller-supplied id
- **File:** `supabase/functions/proposal-signed/index.ts:55-56`

- **Fix:** Read `call_log_id` from `proposals.call_log_id` (DB) instead of the request body, or assert equality before the update.

#### C11. `create-billing-session` selects `tenant_config` without binding to caller
- **File:** `supabase/functions/create-billing-session/index.ts:70-74`

- **Issue:** Caller is authenticated but the `tenant_config` row is fetched with `.limit(1).single()` — i.e., an arbitrary tenant. No `team_members` lookup mapping `user.id → tenant_id`. ### High

#### H1. `get_user_tenant_id` is `SECURITY DEFINER` with no `SET search_path`
- **File:** `sql/rls_identity_tables.sql:113-119`
- **Evidence:**
  ```sql
  CREATE OR REPLACE FUNCTION public.get_user_tenant_id()
  RETURNS uuid AS $$ ... $$ LANGUAGE sql SECURITY DEFINER STABLE;
  ```
- **Issue:** Called by every RLS policy in the codebase. Without `SET search_path = public`, a user with CREATE on a schema earlier in `search_path` can shadow `team_members`/`tenant_config` and force the function to return another tenant's id, defeating tenant isolation everywhere. `is_admin_or_manager()` correctly does set search_path; this one does not.
- **Specs:** CWE-426, OWASP-A01.
- **Fix:** Add `SET search_path = public` to the function definition.

#### H2. `tenant_config_public_read` seed policy: `USING (true)` exposes every tenant's PII to anon
- **File:** `sql/rls_identity_tables.sql:87-89`
- **Evidence:**
  ```sql
  CREATE POLICY "tenant_config_public_read" ON public.tenant_config
    FOR SELECT TO anon
    USING (true);
  ```
- **Issue:** Live migration `20260429190000` drops this, but the seed file ships it. Re-running the seed grants anon (publishable key holders) full-row access to every tenant's branding, phone, email, license_number, monthly/yearly billing goals, `stripe_customer_id`, `stripe_subscription_id`, `subscription_status` — bypassing the column-limited `get_public_tenant_config()` RPC built specifically for this.
- **Specs:** CLAUDE_RLS.md, OWASP-A01, OWASP-A02.
- **Fix:** Delete the policy from the seed file; require all anon access to flow through `public.get_public_tenant_config()`.

#### H3. `team_members_public_read` seed policy exposes full PII per active rep across tenants
- **File:** `sql/rls_identity_tables.sql:60-62`
- **Evidence:**

  ```sql
    FOR SELECT TO anon
    USING (active = true);
  ```
- **Issue:** Same pattern as H2 — dropped in live DB by `20260429190000`, but a fresh seed re-creates anon SELECT of every active team member's full row (auth_id, role) across every tenant. Single-column boolean predicate is itself the 2026-04-26 anti-pattern shape.

#### H4. Anon `proposal_signatures` INSERT does not bind `tenant_id` to the parent proposal
- **File:** `supabase/migrations/20260427180000_add_token_gated_policies.sql:131-140`
- **Evidence:**
  ```sql
  CREATE POLICY "proposal_signatures_public_insert_token" ON public.proposal_signatures
    FOR INSERT TO anon
    WITH CHECK ( EXISTS ( SELECT 1 FROM public.proposals p
        WHERE p.id::text = proposal_id::text
          AND p.signing_token::text = public.request_signing_token() ) );
  ```
- **Issue:** Validates the proposal/token match but does not constrain `tenant_id`. The column has DEFAULT `public.get_user_tenant_id()`, which for anon callers falls back to `SELECT id FROM tenant_config LIMIT 1`. In multi-tenant deploys every anon-inserted signature is stamped with the FIRST tenant's id, mis-attributing signatures and potentially hiding them from the legitimate tenant's RLS-scoped SELECT.
- **Specs:** OWASP-A01, OWASP-A04.
- **Fix:** Add `AND tenant_id = (SELECT tenant_id FROM public.proposals WHERE id::text = proposal_id::text)` to the WITH CHECK, or move signature insert into a `SECURITY DEFINER` RPC.

#### H5. Signing/viewing tokens are permanent — no expiry, no single-use, no revoke
- **File:** `supabase/migrations/20260427180000_add_token_gated_policies.sql:62-67`
- **Evidence:**
  ```sql
  CREATE POLICY "proposals_public_sign_token" ON public.proposals
    FOR SELECT TO anon
    USING ( signing_token IS NOT NULL
            AND signing_token::text = public.request_signing_token() );
  ```
- **Issue:** Tokens are generated once via `crypto.randomUUID()` (NewProposalModal.jsx:52, ArchiveProposalModal.jsx:101) and never rotated, expired, or consumed. `proposals.status = 'Sold'` does not invalidate the token. Anyone who ever sees the URL (forwards, support tickets, browser history, server logs, referer leak) retains permanent access to contract amount, customer billing address, attachments and SOW.
- **Specs:** OWASP-ASVS-V3.3, OWASP-A07.
- **Fix:** Add `signing_token_expires_at` and `signing_token_consumed_at` columns; require `now() < signing_token_expires_at AND signing_token_consumed_at IS NULL` in policy; consume the token on signing.

#### H6. PublicSigningPage uses `select("*")` under anon scope, leaking pricing internals

### Medium

#### M0a. `proposals` SELECT policy does not filter `deleted_at IS NULL`
- **File:** `sql/rls_core_tables.sql:133-135`
- **Evidence:**
  ```sql
  CREATE POLICY "proposals_select" ON public.proposals
    FOR SELECT TO authenticated
    USING (tenant_id = public.get_user_tenant_id());
  ```
- **Issue:** CLAUDE.md says `deleted_at = NULL` means active. Any list query that forgets `.is("deleted_at", null)` returns soft-deleted rows. Authorization (tenant scope) holds, but soft-deleted financial records remain readable indefinitely.

#### M0b. `invoices` SELECT policy does not filter `deleted_at IS NULL`

  ```sql
    FOR SELECT TO authenticated

  ```
- **Issue:** Same as M0a. Also impacts the anon viewing path: `invoices_public_view_token` has no `deleted_at` filter, so a soft-deleted invoice with `viewing_token` is still served to anon, and `PublicInvoicePage.jsx:30-36` does not add `.is('deleted_at', null)` either.
- **Specs:** OWASP-A01.

#### M0c. No rate limit on anon RPCs or token-gated reads
- **File:** `supabase/migrations/20260429180000_add_public_rpc_functions.sql:26`
- **Evidence:**
  ```sql
  GRANT EXECUTE ON FUNCTION public.get_public_tenant_config() TO anon;
  ...
  GRANT EXECUTE ON FUNCTION public.get_rep_contact(text) TO anon;
  ```
- **Issue:** `get_rep_contact(text)` takes free-text rep name — a name-enumeration oracle for harvesting email/phone. Anon SELECT on proposals/invoices also has no rate limit at the edge; UUID v4 entropy makes brute force impractical at internet scale (2^122) but rate limiting is standard defense-in-depth.
- **Specs:** OWASP-ASVS-V11.1, OWASP-A04.
- **Fix:** Add edge-level rate limiting (Cloudflare WAF, edge-function proxy, or `pg_throttle`).

#### M0d. `merge_customers` `FOR UPDATE`-locks rows before tenant check (cross-tenant probe oracle)
- **File:** `supabase/migrations/20260430120000_customer_delete_merge.sql:223-238`
- **Evidence:**
  ```sql
  -- (FOR UPDATE happens at lines 223-229; tenant check at 238)
  IF v_dup.tenant_id <> v_tenant_id OR v_survivor.tenant_id <> v_tenant_id THEN
    RAISE EXCEPTION 'TENANT_MISMATCH';
  END IF;
  ```
- **Issue:** `merge_customers` is `SECURITY DEFINER` and runs `SELECT ... FOR UPDATE` on both ids before the tenant check. Because SECURITY DEFINER bypasses RLS, the function locks rows belonging to other tenants while running, then aborts. The locks are an observable side effect (timing/contention) usable to enumerate customer UUIDs in other tenants. The `customer_merges` audit row is not written on abort, so the probing is unlogged.
- **Specs:** OWASP-A01, CWE-208.
- **Fix:** Read `tenant_id` first, abort on mismatch, then `FOR UPDATE`.

#### M0e. `reset-password` rate limit uses an in-memory `Map` per Edge isolate
- **File:** `supabase/functions/reset-password/index.ts:16`
- **Evidence:**

  ```ts
  const rateLimitMap = new Map<string, number[]>();
  const RATE_LIMIT = 3;
  const RATE_WINDOW_MS = 15 * 60 * 1000;
  ```
- **Issue:** Supabase Edge functions are deployed across multiple isolates; an in-memory Map cannot enforce a global limit. Concurrent requests across instances bypass it, burning Resend quota.
