#### M-A1 — Hardcoded Supabase anon-key fallback in `supabasePublic.js`
- **File:** `src/lib/supabasePublic.js:18-20`
- **Evidence:**
  ```js
  const SUPABASE_KEY =
    import.meta.env.VITE_SUPABASE_ANON_KEY ||
    "sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK";
  ```
- **Why it matters:** Anon keys are publishable, but baking a literal into source defeats env-driven config, blocks key rotation, and masks a missing-env-var deploy. The pattern in `src/lib/supabase.js:7-9` is correct — throw if missing.
- **Spec:** OWASP-A05; CLAUDE.md "Vercel env scoping"
- **Fix:** Remove the literal fallback; throw if `VITE_SUPABASE_ANON_KEY` is missing.

#### M-A2 — Different hardcoded anon-key fallback in `Login.jsx` (inconsistent)
- **File:** `src/pages/Login.jsx:61`
- **Evidence:**
  ```js
  apikey: import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK"
  ```
- **Why it matters:** The literal here (`v7XktVvk…`) differs from the one in `supabasePublic.js:20` (`LL6wYC7k…`). In an env-misconfigured deploy these two code paths would speak to different keys; at minimum one is stale. Two sources of truth ⇒ a guaranteed-to-rot fallback.
- **Spec:** OWASP-A05; CLAUDE.md
- **Fix:** Drop the literal; route the forgot-password call through `supabase.functions.invoke("reset-password", …)`.
