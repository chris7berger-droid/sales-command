# Sales Command Security Audit — 2026-05-11

## Summary

**0 critical · 6 high · 14 medium · 6 low — posture improving.**

Every C-rated finding from the 2026-04-30 audit is closed: the QB/billing
tenant-isolation gaps (C4–C8, C11), the SSRF in `send-pay-app` (C9), the
`proposal-signed` body-trust bug (C10), the broad anon read policies
(C1/C3/C12), the seed-SQL anti-pattern (C2), and the security-function
search_path gaps (H1/H7/S1) are all resolved in code today.

New surface concentrated in three areas:
1. **Latent multi-tenant footguns** — every Pass B finding (H-B1, H-B2, M-B1, M-B2, M-B3, M-B4) is non-exploitable under today's single-tenant DB but activates the moment a second tenant onboards (BACKLOG F7). Address before F7 ships.
2. **Phishing / spoofing surface in mail-sending functions** — H-C2 (`send-proposal` body-trust), M-C2/M-C3 (sender-email checks), L-C1 (single-tenant From). The verified-domain check is a domain check, not an ownership check.
3. **One real regression** — M-C1 (`send-pay-app` references undeclared `authHeader`) was introduced alongside the C9 SSRF fix; pay-app → QB sync is silently broken.

#### H-C1 — Permissive CORS allowlist accepts every `*.vercel.app` origin
- **File:** all 19 edge functions; e.g. `supabase/functions/send-invoice/index.ts:15`, `send-pay-app/index.ts:69`, `qb-auth/index.ts:17`, `stripe-webhook/index.ts:58`, `scc-stripe-webhook/index.ts:47`, `create-billing-session/index.ts:24`
- **Evidence:**
  ```ts

  ```
- **Why it matters:** Anyone can deploy `attacker.vercel.app`. The function reflects that origin into `Access-Control-Allow-Origin`, so a browser running attacker.vercel.app can issue credentialed cross-origin requests against `send-invoice`, `send-pay-app`, `qb-*`, `invite-user`, `delete-user`, `create-billing-session`, etc. With a logged-in victim, this becomes CSRF against money + admin endpoints. Seven QB functions and `send-pay-app` additionally accept `http://localhost*` — same shape, lower bar. Prior audit H8.
- **Spec:** OWASP-A05, OWASP-A07; Supabase prod checklist (CORS)
- **Fix:** Drop the `endsWith(".vercel.app")` and `startsWith("http://localhost")` fallbacks. Pin to the explicit `ALLOWED_ORIGINS` list; if previews are needed, allowlist the exact preview domain(s) used by CI.

#### H-C2 — `send-proposal` trusts caller-supplied `repEmail`/`signingUrl`; enables intra-tenant phishing
- **File:** `supabase/functions/send-proposal/index.ts:43, 63-68, 89`
- **Evidence:**
  ```ts
  const { customerEmail, customerName, repEmail, repName, ..., signingUrl, ... } = await req.json();
  const VERIFIED_DOMAINS = ["hdspnv.com", "scmybiz.com", "schmybiz.com", "salescommand.app"];
  const fromAddress = repEmail && VERIFIED_DOMAINS.includes(repDomain)
    ? `${senderName} <${repEmail}>` : `${senderName} <noreply@salescommand.app>`;
  ```
- **Why it matters:** Caller only has to be authenticated. Every business field — recipient address, From address (when domain is verified), the `Review & Sign Proposal` button URL (line 89), proposal number, customer name — is read straight from the request body with no DB reconciliation and no tenant scoping. Any low-privileged sales rep at any tenant can: (a) email arbitrary HTML + a chosen URL labeled "Review & Sign Proposal" to any address, (b) set `repEmail` to a coworker's verified-domain address to spoof them. Real phishing surface from inside the trust boundary.
- **Spec:** OWASP-A01, OWASP-A04; CLAUDE.md "money/admin functions must verify tenant_id"
- **Fix:** Accept only `proposalId`; load `signingUrl`, recipient, rep, and tenant from the DB row. Verify the caller's tenant matches `proposals.tenant_id`. Force the From header to either `noreply@salescommand.app` or the caller's own verified-domain address (looked up from `team_members.email`).

#### M-B3 — Authenticated SELECT policies on soft-delete tables don't filter `deleted_at IS NULL`
- **File:** `sql/rls_core_tables.sql:138` (`proposals_select`); same pattern on `invoices_select` and `call_log_select`
- **Why it matters:** `proposals`, `invoices`, and `call_log` all carry `deleted_at timestamptz` (CLAUDE.md). The SELECT policy filters only on `tenant_id`, not on `deleted_at`. Any user that bypasses the React app's client-side filter (raw PostgREST query, anyone) can resurface soft-deleted rows — invoices voided after dispute, proposals withdrawn for sensitive reasons.

#### M-D2 — User-controlled `Content-Type` on `job-attachments` uploads enables stored HTML on `supabase.co` origin
- **File:** `src/components/PayAppDetailModal.jsx:178, 202`; `src/pages/Customers.jsx:274`
- **Evidence:**
  ```jsx

  ```
- **Why it matters:** Browser sets `file.type` from the user's MIME hint. An attacker can upload an HTML file with `Content-Type: text/html`. - **Spec:** OWASP-A07; ASVS V11.1

#### L-C1 — `proposal-signed` hardcodes `from: estimates@hdspnv.com` (single-tenant outbound identity)
- **File:** `supabase/functions/proposal-signed/index.ts:119`
- **Why it matters:** Every tenant's customer-signing notifications appear to come from HDSP's domain. Not a security vuln but breaks tenant isolation on outbound mail; brand confusion for customers of other tenants.
- **Fix:** Pull from `tenant_config.email` for the proposal's tenant.

#### L-C2 — Edge functions return raw `error.message` to clients (info leak)
- **File:** every function's outer catch; e.g. `send-proposal/index.ts:143-144`, `send-invoice/index.ts:251-257`, `qb-sync-invoice/index.ts:337-352`, `create-billing-session/index.ts:215-221`, `send-pay-app/index.ts:483-489`
- **Why it matters:** Surface raw Postgres / QB / Resend error bodies to callers — sometimes includes IDs from other rows, internal column names, or stack messages.
- **Spec:** OWASP-A05; ASVS V7.4
- **Fix:** Return a generic message + a request ID; log the detail server-side.

#### L-C3 — `reset-password` rate limiter is per-instance, resets on cold start
- **File:** `supabase/functions/reset-password/index.ts:16-18, 46-55`
- **Why it matters:** In-memory map keyed by IP. Supabase scales horizontally; a new instance starts with empty state. The limit is a hint, not enforcement.
- **Spec:** ASVS V11.1
- **Fix:** Persist counts in a small KV table or call a `SECURITY DEFINER` RPC that tracks per-IP/per-email windows in Postgres.

#### L-E1 — `xlsx@0.20.3` resolved from `cdn.sheetjs.com` (non-npm)
- **File:** `package.json:20`; `package-lock.json` (xlsx resolved url)
- **Evidence:** `"xlsx": "https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz"`
- **Why it matters:** SheetJS de-listed from npm; this CDN tarball is documented in `package.json` and pinned by an `integrity` SHA-512 hash in the lockfile, so a swapped tarball would fail install. Noted (not a vuln), per skill rubric Pass E.
- **Spec:** OWASP-A08
- **Fix:** No action; review integrity hash on each upgrade.
