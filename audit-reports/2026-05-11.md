#### H-C1 ‚Äî Permissive CORS allowlist accepts every `*.vercel.app` origin
- **File:** all 19 edge functions; e.g. `supabase/functions/send-invoice/index.ts:15`, `send-pay-app/index.ts:69`, `qb-auth/index.ts:17`, `stripe-webhook/index.ts:58`, `scc-stripe-webhook/index.ts:47`, `create-billing-session/index.ts:24`
- **Evidence:**
  ```ts
  const isAllowed = ALLOWED_ORIGINS.includes(origin) || origin.endsWith(".vercel.app");
  ```
- **Why it matters:** Anyone can deploy `attacker.vercel.app`. The function reflects that origin into `Access-Control-Allow-Origin`, so a browser running attacker.vercel.app can issue credentialed cross-origin requests against `send-invoice`, `send-pay-app`, `qb-*`, `invite-user`, `delete-user`, `create-billing-session`, etc. With a logged-in victim, this becomes CSRF against money + admin endpoints. Seven QB functions and `send-pay-app` additionally accept `http://localhost*` ‚Äî same shape, lower bar. Prior audit H8.
- **Spec:** OWASP-A05, OWASP-A07; Supabase prod checklist (CORS)
- **Fix:** Drop the `endsWith(".vercel.app")` and `startsWith("http://localhost")` fallbacks. Pin to the explicit `ALLOWED_ORIGINS` list; if previews are needed, allowlist the exact preview domain(s) used by CI.

#### H-C2 ‚Äî `send-proposal` trusts caller-supplied `repEmail`/`signingUrl`; enables intra-tenant phishing
- **File:** `supabase/functions/send-proposal/index.ts:43, 63-68, 89`
- **Evidence:**
  ```ts
  const { customerEmail, customerName, repEmail, repName, ..., signingUrl, ... } = await req.json();
  const VERIFIED_DOMAINS = ["hdspnv.com", "scmybiz.com", "schmybiz.com", "salescommand.app"];
  const fromAddress = repEmail && VERIFIED_DOMAINS.includes(repDomain)
    ? `${senderName} <${repEmail}>` : `${senderName} <noreply@salescommand.app>`;
  ```
- **Why it matters:** Caller only has to be authenticated. Every business field ‚Äî recipient address, From address (when domain is verified), the `Review & Sign Proposal` button URL (line 89), proposal number, customer name ‚Äî is read straight from the request body with no DB reconciliation and no tenant scoping. Any low-privileged sales rep at any tenant can: (a) email arbitrary HTML + a chosen URL labeled "Review & Sign Proposal" to any address, (b) set `repEmail` to a coworker's verified-domain address to spoof them. Real phishing surface from inside the trust boundary.
- **Spec:** OWASP-A01, OWASP-A04; CLAUDE.md "money/admin functions must verify tenant_id"
- **Fix:** Accept only `proposalId`; load `signingUrl`, recipient, rep, and tenant from the DB row. Verify the caller's tenant matches `proposals.tenant_id`. Force the From header to either `noreply@salescommand.app` or the caller's own verified-domain address (looked up from `team_members.email`).

### Medium

#### M-A1 ‚Äî Hardcoded Supabase anon-key fallback in `supabasePublic.js`
- **File:** `src/lib/supabasePublic.js:18-20`
- **Evidence:**
  ```js
  const SUPABASE_KEY =
    import.meta.env.VITE_SUPABASE_ANON_KEY ||
    "sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK";
  ```
- **Why it matters:** Anon keys are publishable, but baking a literal into source defeats env-driven config, blocks key rotation, and masks a missing-env-var deploy. The pattern in `src/lib/supabase.js:7-9` is correct ‚Äî throw if missing.
- **Spec:** OWASP-A05; CLAUDE.md "Vercel env scoping"
- **Fix:** Remove the literal fallback; throw if `VITE_SUPABASE_ANON_KEY` is missing.

#### M-A2 ‚Äî Different hardcoded anon-key fallback in `Login.jsx` (inconsistent)
- **File:** `src/pages/Login.jsx:61`
- **Evidence:**
  ```js
  apikey: import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK"
  ```
- **Why it matters:** The literal here (`v7XktVvk‚Ä¶`) differs from the one in `supabasePublic.js:20` (`LL6wYC7k‚Ä¶`). In an env-misconfigured deploy these two code paths would speak to different keys; at minimum one is stale. Two sources of truth ‚áí a guaranteed-to-rot fallback.
- **Spec:** OWASP-A05; CLAUDE.md
- **Fix:** Drop the literal; route the forgot-password call through `supabase.functions.invoke("reset-password", ‚Ä¶)`.

#### M-E1 ‚Äî `pdf-lib@1.17.1` unmaintained (~4y stale) in customer-facing signing path
- **File:** `package.json:16`; `package-lock.json` (resolved `pdf-lib@1.17.1`)
- **Evidence:** `"pdf-lib": "^1.17.1"` (last npm publish 2022-05-12; today is 2026-05-11)
- **Why it matters:** Prod runtime dep used to assemble the signed-proposal PDF in the customer-facing signing flow. No known active CVE today, but the parser handles untrusted/attacker-influenced inputs (fonts, images embedded from arbitrary attachments). Crosses the 2-year unmaintained threshold from `CLAUDE.md`'s security expectations for critical paths.
- **Spec:** OWASP-A06 (Vulnerable & Outdated Components); skill rubric Pass E
- **Fix:** Track a maintained replacement, or pin to a vetted fork and document a triage cadence.

#### M-C1 ‚Äî `send-pay-app` references undeclared `authHeader`; QB sync silently broken
- **File:** `supabase/functions/send-pay-app/index.ts:466`
- **Evidence:**
  ```ts
  fetch(`${SUPABASE_URL}/functions/v1/qb-sync-invoice`, {
    method: "POST",
    headers: { "Content-Type": "application/json", "Authorization": authHeader },
  ```
- **Why it matters:** `authHeader` is never declared in this scope (`grep -n authHeader send-pay-app/index.ts` returns line 466 only). The fire-and-forget `qb-sync-invoice` call throws `ReferenceError` on every invocation; the surrounding `.catch(...)` swallows it. Pay-app ‚Üí QB sync is silently broken in production ‚Äî a data-integrity regression introduced alongside the C9 SSRF fix.
- **Spec:** CLAUDE.md "Data Integrity Rules"; OWASP-A09 (logging failures)
- **Fix:** Pull `authHeader` from `req.headers.get("authorization")` before the fetch, or forward the caller's JWT explicitly.

#### M-C2 ‚Äî `send-invoice` allows any tenant member to set `senderEmail` to a coworker's verified-domain address
- **File:** `supabase/functions/send-invoice/index.ts:29-32, 99-103`
- **Why it matters:** Amount and recipient are now DB-derived (prior H10 partially closed). However the `From` header is built from caller-supplied `senderEmail`, restricted only to `VERIFIED_DOMAINS`. Any sales rep at a verified-domain tenant can set `senderEmail` to their CEO's address and outgoing invoice notifications appear to come from the CEO. Intra-tenant impersonation only ‚Äî but the verified-domain check is a domain check, not an ownership check.
- **Spec:** OWASP-A04
- **Fix:** Source `senderEmail` from `team_members` keyed by `auth.uid()`; ignore the body value.

#### M-C3 ‚Äî `invite-user` skips verified-domain check on `senderEmail`
- **File:** `supabase/functions/invite-user/index.ts:72, 160`
- **Evidence:**
  ```ts
  const { email, name, teamMemberId, senderEmail, senderName } = await req.json();
  ...
  from: senderEmail || "noreply@salescommand.app",
  ```
- **Why it matters:** Unlike `send-invoice` and `send-pay-app`, this function passes the caller-supplied `senderEmail` directly into Resend's `from` with no `VERIFIED_DOMAINS` guard. Tenant-isolation is enforced for the invitee, but an admin can spoof any From string. Likely H11 from prior audit ‚Äî not fixed.
- **Spec:** OWASP-A04
- **Fix:** Apply the same `VERIFIED_DOMAINS` filter used in `send-invoice`/`send-pay-app`, then source `senderEmail` from `team_members.email` by `auth.uid()`.

#### M-C4 ‚Äî `extract-sov` fetches `pdf_url` with no scheme/host allowlist
- **File:** `supabase/functions/extract-sov/index.ts:115-153`
- **Why it matters:** Authorization correctly requires `pdf_url` to appear in a `billing_schedule.contract_pdf_url(s)` row the caller's tenant owns (lines 131-150). But `billing_schedule.contract_pdf_url` is column-writable by any tenant admin ‚Äî a malicious insider can store `http://169.254.169.254/...` or `http://internal-host/...`, then call `extract-sov` to have the edge runtime fetch and pass the bytes to Anthropic. Defense-in-depth gap; same shape `send-pay-app` already solved via `isAllowedStorageUrl`.
- **Spec:** OWASP-A10 (SSRF)
- **Fix:** Port `send-pay-app/index.ts:18-29`'s `isAllowedStorageUrl()` and reject any URL not in the Supabase Storage host allowlist.

#### M-C5 ‚Äî `scc-stripe-webhook` trusts `metadata.tenant_id` without verifying customer‚Üítenant binding
- **File:** `supabase/functions/scc-stripe-webhook/index.ts:88, 112, 140`
- **Why it matters:** Stripe signature verification (line 69-73) is in place, so events are authentic. But `tenantId = session.metadata?.tenant_id` is then used to `UPDATE tenant_config` directly. A customer who tampers with `metadata.tenant_id` on a self-service Checkout (or any manually constructed Checkout link) can overwrite another tenant's `stripe_customer_id`, `subscription_status`, and `apps`. Defense-in-depth gap.
- **Spec:** OWASP-A01, OWASP-A04
- **Fix:** Resolve tenant via `stripe_customer_id` lookup in `tenant_config`; refuse the event if `metadata.tenant_id` doesn't match the resolved row, or if the resolved row already has a different `stripe_customer_id`.

#### M-E2 ‚Äî Edge functions float `@supabase/supabase-js@2` on esm.sh; one pins `@2.39.0` (inconsistent)
- **File:** `supabase/functions/qb-auth/index.ts:2` (pins `@2.39.0`); all other 18 functions float `@2`
- **Evidence:**
  ```ts
  import { createClient } from "https://esm.sh/@supabase/supabase-js@2";          // 18 functions
  import { createClient } from "https://esm.sh/@supabase/supabase-js@2.39.0";    // qb-auth only
  ```
- **Why it matters:** Deno has no `package-lock.json`; `@2` resolves whatever esm.sh serves at deploy time. Different functions in the same deploy can talk to different minor versions. No import map / SRI either ‚Äî supply-chain risk via the esm.sh CDN. Also `deno.land/std@0.168.0` (used by `serve`) is from 2022-12.
- **Spec:** OWASP-A08 (Software & Data Integrity Failures)
- **Fix:** Add `supabase/functions/import_map.json` pinning every URL to a specific version, deploy with `--import-map`.

#### L-C2 ‚Äî Edge functions return raw `error.message` to clients (info leak)
- **File:** every function's outer catch; e.g. `send-proposal/index.ts:143-144`, `send-invoice/index.ts:251-257`, `qb-sync-invoice/index.ts:337-352`, `create-billing-session/index.ts:215-221`, `send-pay-app/index.ts:483-489`
- **Why it matters:** Surface raw Postgres / QB / Resend error bodies to callers ‚Äî sometimes includes IDs from other rows, internal column names, or stack messages.
- **Spec:** OWASP-A05; ASVS V7.4
- **Fix:** Return a generic message + a request ID; log the detail server-side.

#### L-C3 ‚Äî `reset-password` rate limiter is per-instance, resets on cold start
- **File:** `supabase/functions/reset-password/index.ts:16-18, 46-55`
- **Why it matters:** In-memory map keyed by IP. Supabase scales horizontally; a new instance starts with empty state. The limit is a hint, not enforcement.
- **Spec:** ASVS V11.1
- **Fix:** Persist counts in a small KV table or call a `SECURITY DEFINER` RPC that tracks per-IP/per-email windows in Postgres.

#### L-E1 ‚Äî `xlsx@0.20.3` resolved from `cdn.sheetjs.com` (non-npm)
