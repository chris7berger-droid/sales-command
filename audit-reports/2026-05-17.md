#### M-A1 — Hardcoded Supabase anon-key fallback in `supabasePublic.js` (persistent)
- **File:** `src/lib/supabasePublic.js:18-20`
- **Evidence:**
  ```js
  const SUPABASE_KEY =
    import.meta.env.VITE_SUPABASE_ANON_KEY ||
    "sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK";
  ```
- **Why it matters:** The publishable/anon key is intended to be public, but baking a literal into source defeats env-driven rotation, masks a missing-env-var deploy (preview, fork, new tenant), and pins the build to a single Supabase project. `src/lib/supabase.js:7-9` already throws on missing env — the public client should match.
- **Spec:** OWASP-ASVS V14.1; CLAUDE.md Security Rules
- **Fix:** Drop the literal; throw if `VITE_SUPABASE_ANON_KEY` is missing.

#### M-A2 — Hardcoded Supabase anon-key fallback in `Login.jsx` (persistent, distinct value)
- **File:** `src/pages/Login.jsx:61`
- **Evidence:**
  ```js
  apikey: import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK"
  ```
- **Why it matters:** A second hardcoded publishable key — different from `supabasePublic.js` (`LL6wYC7k…` vs. `v7XktVvk…`). Two sources of truth ⇒ one is stale; an env-misconfigured deploy would route the password-reset call through a stale key. Same rotation/lock-in concern as M-A1.
- **Spec:** OWASP-ASVS V14.1; CLAUDE.md
- **Fix:** Drop the literal; route through `supabase.functions.invoke("reset-password", ...)` to inherit the configured client's key.

#### M-A3 — `invite-user` logs the full magic-link object (action_link is bearer-equivalent)
- **File:** `supabase/functions/invite-user/index.ts:135`
- **Evidence:**
  ```ts
  console.log("generateLink result:", { linkData, linkErr: linkErr?.message });
  ```
- **Why it matters:** `linkData.properties.action_link` is the single-use magic link that authenticates the new user when clicked. Logging the whole object writes a live credential to the Supabase Functions log. Anyone with project `logs:read` (and Supabase support staff) can hijack the invite before the real recipient opens the email — they click the action_link, set a password, and own the account.
- **Spec:** OWASP-A09; OWASP-ASVS V7.1, V8.3
- **Fix:** Log only a boolean: `console.log("generateLink result:", { ok: !!linkData?.properties?.action_link, linkErr: linkErr?.message })`.

#### M-E1 — `pdf-lib@1.17.1` unmaintained (~4y stale) in customer-facing signing path (persistent)
- **File:** `package.json:16`
- **Evidence:** `"pdf-lib": "^1.17.1"` — last npm publish 2022-05-12; today is 2026-05-17.
- **Why it matters:** Prod runtime dep used to assemble the signed-proposal PDF in the customer-facing signing flow. No known active CVE today, but the parser handles attacker-influenced inputs (fonts, images embedded from arbitrary attachments). Crosses the 2-year unmaintained threshold for security-critical paths. `npm audit` clean, but unmaintained ≠ safe.
- **Spec:** OWASP-A06 (Vulnerable & Outdated Components)
- **Fix:** Track a maintained replacement (e.g., `@cantoo/pdf-lib` fork) or pin a vendored copy with a documented review cadence.

#### M-E2 — Edge functions float `@supabase/supabase-js@2` and `deno.land/std@0.168.0` on a third-party CDN (persistent)
- **File:** `supabase/functions/send-proposal/index.ts:1-2` (representative; 18 of 20 functions have the same shape)
- **Evidence:**
  ```ts
  import { serve } from "https://deno.land/std@0.168.0/http/server.ts";
  import { createClient } from "https://esm.sh/@supabase/supabase-js@2";
  ```
- **Why it matters:** Deno has no `package-lock.json`. `@supabase/supabase-js@2` floats whatever esm.sh serves at deploy time; `deno.land/std@0.168.0` is from 2022-12 and long superseded. No import-map, no SRI hashes. A CDN compromise, registry-tag poisoning, or transitive supply-chain incident ships compromised JS into every privileged function (service_role access: send-proposal, qb-*, stripe-webhook, delete-user, invite-user, reset-password) on the next deploy. `qb-auth/index.ts:2` pins `@2.39.0` — inconsistent with the rest.
- **Spec:** OWASP-A08 (Software & Data Integrity Failures)
- **Fix:** Pin every edge function to an exact version (`@2.99.0` to match `package.json:13`) and a current `deno.land/std`. Even better: commit `supabase/functions/import_map.json` + `deno.lock` and deploy with `--import-map`.

#### L-A3 — `send-proposal` logs customer + rep emails on every invocation
- **File:** `supabase/functions/send-proposal/index.ts:45`
- **Evidence:**
  ```ts
  console.log("send-proposal invoked", { customerEmail, repEmail, proposalNumber, jobName });
  ```
- **Why it matters:** Customer email, rep email, job name, and proposal number persist to Functions logs on every send. Logs are accessible to anyone with project `logs:read` (and to Supabase support staff). PII spillage outside the RLS boundary, with no business need for the values in plaintext logs.
- **Spec:** OWASP-A09; OWASP-ASVS V8.3; GDPR data minimization
- **Fix:** Drop email fields; `console.log("send-proposal invoked", { proposalNumber })`.

#### L-A4 — `invite-user` logs invitee + sender PII on every invocation
- **File:** `supabase/functions/invite-user/index.ts:95`
- **Evidence:**
  ```ts
  console.log("invite-user invoked", { email, name, teamMemberId, senderEmail, senderName });
  ```
- **Why it matters:** Same PII-spillage class as L-A3. Invitee email/name and inviter identity persist to logs on every invite.
- **Spec:** OWASP-A09; GDPR data minimization
- **Fix:** Drop email/name from the log line; `teamMemberId` alone traces flow.

#### L-A5 — `stripe-webhook` logs payment amount and payment_intent ID per invoice
- **File:** `supabase/functions/stripe-webhook/index.ts:96`
- **Evidence:**
  ```ts
  console.log("Payment completed for invoice:", invoiceId, "payment_intent:", paymentIntent, "amount:", amountTotal);
  ```
- **Why it matters:** Payment intent ID and dollar amount are written per payment. Same `logs:read` exposure surface as the other PII logs; amount is financial data.
- **Spec:** OWASP-A09; PCI-DSS 3.4 (data minimization)
- **Fix:** Log only `invoiceId` + a boolean success flag; move amount/intent to an audit table behind RLS if retention is desired.
