#### H-B1 — `work_types` anon SELECT `USING (true)` ignores tenant scope on a table that now holds per-tenant rows
- **File:** `supabase/migrations/20260515150000_work_types_public_read.sql:31-33`
- **Evidence:**
  ```sql
  CREATE POLICY "work_types_public_read" ON public.work_types
    FOR SELECT TO anon
    USING (true);
  ```
  vs. `sql/work_types_tenant_phase1.sql:13-17` which adds `tenant_id uuid REFERENCES public.tenant_config(id)` and `sales_sow text`, and `sql/work_types_tenant_phase1.sql:49-54` whose authenticated SELECT explicitly handles `tenant_id IS NULL OR tenant_id = public.get_user_tenant_id()`.
- **Why it matters:** The migration comment (lines 4-12) asserts the table has "no sensitive or tenant-scoped data," but the phase1 seed added `tenant_id` and `sales_sow` — per-tenant SOW boilerplate text. Any holder of the publishable anon key can now read every tenant's custom work_types including the `tenant_id` linkage and `sales_sow` IP templates. Today (single-tenant prod) the exposure is limited to one tenant's templates leaking to anon viewers; the moment F7 onboards a second tenant, this becomes cross-tenant. Same class as the 2026-04-30-closed `tenant_config_public_read` finding.
- **Spec:** CLAUDE_RLS.md (anon SELECT must not leak cross-tenant); OWASP-A01
- **Fix:** Restrict to `USING (tenant_id IS NULL)` so anon only reads system defaults, or route through a SECURITY DEFINER RPC that takes a viewing/signing token and joins to the token-bound proposal's tenant. `PublicInvoicePage.jsx` only needs the `name` string, which `get_public_proposal_view` already exposes as `work_type_name`.

#### H-B4 — Anon viewing/signing-token policies do not filter `deleted_at IS NULL` — soft-deleted rows remain customer-readable
- **File:** `supabase/migrations/20260427180000_add_token_gated_policies.sql:180-185, 194-203`; `supabase/migrations/20260515140000_proposal_wtc_public_view_token.sql:18-29`; `supabase/migrations/20260510120000_signing_token_expiry_and_consume.sql` (proposal viewing branches)
- **Evidence:**
  ```sql
  CREATE POLICY "invoices_public_view_token" ON public.invoices
    FOR SELECT TO anon
    USING (
      viewing_token IS NOT NULL
      AND viewing_token::text = public.request_viewing_token()
    );
  ```
- **Why it matters:** `invoices` and `proposals` are soft-delete tables (`deleted_at` per CLAUDE.md). Five anon SELECT/EXISTS policies (`invoices_public_view_token`, `invoice_lines_public_read_token`, the new `proposal_wtc_public_view_token`, `call_log_public_view_token`, and the viewing branch of `customers_public_read_token`) do NOT filter `deleted_at IS NULL`. A customer who once received an invoice or signing URL can continue to read the deleted record's pricing, billing data, and Stripe checkout URL indefinitely after operations "delete" it from the UI. Extends prior audit M-B3 (authenticated-only) to the anon surface — and the new `proposal_wtc_public_view_token` migration widens the leak to line-item SOW text.
- **Spec:** CLAUDE.md soft-delete invariant; OWASP-A01
- **Fix:** Add `AND deleted_at IS NULL` (resp. inside each EXISTS join on invoices/proposals) to all five anon viewing-token policies. Sweep-migration. Optional: rotate `viewing_token` to NULL on soft-delete as defense-in-depth.

#### H-B5 — New `proposal_wtc_public_view_token` (2026-05-15) inherits the no-expiry / single-use gap on `invoices.viewing_token`
- **File:** `supabase/migrations/20260515140000_proposal_wtc_public_view_token.sql:18-29`
- **Evidence:**
  ```sql
  CREATE POLICY "proposal_wtc_public_view_token" ON public.proposal_wtc
    FOR SELECT TO anon
    USING (
      EXISTS (
        SELECT 1 FROM public.invoices i
        JOIN public.invoice_lines l ON l.invoice_id::text = i.id::text
        WHERE l.proposal_wtc_id = proposal_wtc.id
          AND i.viewing_token IS NOT NULL
          AND i.viewing_token::text = public.request_viewing_token()
      )
    );
  ```
- **Why it matters:** The new policy ports the existing forever-valid `viewing_token` gap (M-B4 from prior audit / BACKLOG B13) onto `proposal_wtc`. Any past holder of an invoice viewing URL now reads SOW text, locked line totals, and work-type linkage indefinitely. Combined with H-B1, leaked invoice URLs keep revealing line-item descriptions long after the customer relationship ends.
- **Spec:** CLAUDE_RLS.md; OWASP-A04 (insecure design — no expiry)
- **Fix:** Land BACKLOG B13 (`viewing_token_expires_at` + trigger) as prerequisite, then add `AND i.viewing_token_expires_at IS NOT NULL AND i.viewing_token_expires_at > now()` to all five viewing-token policies in one migration.

#### H-C2 — `send-proposal` trusts caller-supplied `repEmail` AND `signingUrl` (persistent + extended)
- **File:** `supabase/functions/send-proposal/index.ts:43, 65-68, 89`
- **Evidence:**
  ```ts
  const { customerEmail, customerName, repEmail, repName, proposalNumber, jobName, signingUrl, ... } = await req.json();
  const VERIFIED_DOMAINS = ["hdspnv.com", "scmybiz.com", "schmybiz.com", "salescommand.app"];
  const fromAddress = repEmail && VERIFIED_DOMAINS.includes(repDomain)
    ? `${senderName} <${repEmail}>` : `${senderName} <noreply@salescommand.app>`;
  ```

- **Why it matters:** Two phishing surfaces from any authenticated tenant member: (1) **From spoofing** — any logged-in user can set `repEmail` to a coworker's (or another tenant's) verified-domain address (`ceo@hdspnv.com`) and Resend DKIM-signs it. (2) **Signing-URL substitution** — `signingUrl` is interpolated into the customer-facing "Review & Sign Proposal" button with no host allowlist; a malicious sales rep can email any address a real-looking proposal whose CTA points at attacker-controlled JS that harvests creds or impersonates the signing flow. - **Fix:** Accept only `proposalId`. Load `signingUrl`, recipient, rep, and tenant from the DB row (`proposals.signing_token` + tenant). Force `From` to `noreply@salescommand.app` or `team_members.email` looked up by `auth.uid()`. Verify caller's tenant matches `proposals.tenant_id`.

#### H-C3 — `send-pay-app` references undeclared `authHeader` — every success returns 500 and QB sync silently broken (persistent regression)
- **File:** `supabase/functions/send-pay-app/index.ts:466`
- **Evidence:**
