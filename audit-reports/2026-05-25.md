- **B1** — **`work_types` anon SELECT `USING (true)` leaks tenant-custom rows**
  - `supabase/migrations/20260515150000_work_types_public_read.sql:32`
  - Evidence: `CREATE POLICY "work_types_public_read" ON public.work_types FOR SELECT TO anon USING (true);`
  - The migration comment claims `work_types` is pure reference data with no tenant scoping. However `sql/work_types_tenant_phase1.sql:14` adds `ADD COLUMN tenant_id uuid` to the same table. After that seed runs, `USING (true)` exposes every tenant's custom work types (including `sales_sow` text bodies) to any anon caller with the publishable key. Today single-tenant; latent cross-tenant leak the moment a second tenant is provisioned.
  - Spec: CLAUDE_RLS.md, OWASP-ASVS V8.1, OWASP A01.
  - Fix: scope the anon SELECT to system rows only — `USING (tenant_id IS NULL)` — or move the join through a SECURITY DEFINER RPC that's already token-scoped.

- **B2** — **`get_public_proposal_view` does not filter `deleted_at`**
  - `supabase/migrations/20260510120000_signing_token_expiry_and_consume.sql:304`
  - Evidence: `SELECT p.id INTO v_proposal_id FROM public.proposals p WHERE p.signing_token IS NOT NULL AND p.signing_token::text = p_token AND p.signing_token_expires_at IS NOT NULL AND p.signing_token_expires_at > now()`
  - Soft-deleted proposals remain readable through the public signing RPC. An internal user soft-deleting a proposal does not invalidate the customer's link — the customer can still view the proposal and invoke `mark_proposal_signed` to flip a retracted proposal to Sold.
  - Spec: CLAUDE.md soft-delete contract, OWASP-ASVS V8.3, OWASP A01.
  - Fix: add `AND p.deleted_at IS NULL` to the token-resolution SELECT here, in `mark_recipient_viewed`, in `mark_proposal_signed`, and in every signing-token anon SELECT policy.

- **B3** — **Soft-deleted invoices remain readable via viewing-token policy**
  - `supabase/migrations/20260520095000_invoice_viewing_token_expiry.sql:41`
  - Evidence: `CREATE POLICY "invoices_public_view_token" ON public.invoices FOR SELECT TO anon USING (viewing_token IS NOT NULL AND viewing_token::text = public.request_viewing_token() AND viewing_token_expires_at IS NOT NULL AND viewing_token_expires_at > now());`
  - `invoices.deleted_at` is documented in CLAUDE.md but the anon SELECT policy and child policies (`invoice_lines`, `proposal_wtc` viewing branch, `call_log_public_view_token`, `customers` viewing branch) never check `deleted_at IS NULL`. A customer holding a viewing token for a cancelled/duplicate invoice keeps reading invoice, lines, and customer rows; the public payment URL stays live.
  - Spec: CLAUDE.md soft-delete contract, OWASP-ASVS V8.3, OWASP A01.
  - Fix: add `AND i.deleted_at IS NULL` to the invoice predicate in every viewing-token anon SELECT policy and downstream RPC.

- **B4** — **`get_public_tenant_config` returns `LIMIT 1` with no tenant scoping**
  - `supabase/migrations/20260429180000_add_public_rpc_functions.sql:21`
  - Evidence: `SELECT company_name, tagline, logo_url, phone, email, website, license_number, proposal_validity_days FROM public.tenant_config LIMIT 1;`
  - The RPC ignores which tenant the caller is signing for and returns whichever `tenant_config` row Postgres scans first. In a multi-tenant deploy (BACKLOG F7 is in progress) the signing page for Tenant B's proposal renders Tenant A's logo, phone, email, and license number — branding leakage and customer-trust breakage. The `x-signing-token` header is already on the request — derive tenant via proposals.
  - Spec: CLAUDE_RLS.md cross-tenant, OWASP-ASVS V8.1, OWASP A01.
  - Fix: take `p_token`, resolve `proposal.tenant_id` under the token, return `tenant_config` for that tenant; same shape for invoice viewing.

- **B5** — **`get_rep_contact` is unscoped — anon enumeration of team_members PII**
  - `supabase/migrations/20260429180000_add_public_rpc_functions.sql:33`
  - Evidence: `SELECT tm.name, tm.email, tm.phone FROM public.team_members tm WHERE tm.name = rep_name AND tm.active = true LIMIT 1;`
  - `EXECUTE` granted to anon. Any anon caller can supply a guessed `rep_name` and harvest `email` + `phone` across tenants. With multi-tenant onboarding, Tenant B's salesrep PII surfaces on Tenant A's customer page just because names match. Even single-tenant, this is an unauthenticated PII enumeration endpoint (try common first/last names).
  - Spec: OWASP-ASVS V8.1, OWASP A01, OWASP A07.
  - Fix: take a token parameter (signing or viewing), resolve `tenant_id` from the token, and scope the `team_members` lookup by `tenant_id AND name`.

- **B6** — **Rollback seed re-introduces `USING (true)` anon read on `tenant_config` + `team_members`**
  - `sql/rollback_drop_public_read_policies.sql:4`
  - Evidence: `CREATE POLICY "tenant_config_public_read" ON public.tenant_config FOR SELECT TO anon USING (true); CREATE POLICY "team_members_public_read" ON public.team_members FOR SELECT TO anon USING (active = true);`
  - One-click footgun: an operator hitting an RPC failure who runs this rollback re-creates exactly the policies that the 2026-04-30 audit (H2/H3) closed. Result: every tenant's `tenant_config` (including `stripe_customer_id`, `monthly_billing_goal`) and every active `team_members` row (auth_id, role, full PII) becomes readable with just the publishable anon key. No warning in the file.
  - Spec: CLAUDE_RLS.md, OWASP A01, OWASP-ASVS V8.1.
  - Fix: delete this file (the safe rollback is `revoke execute on function get_public_*` — not re-creating the anti-pattern). The prior audit's C12 made the same point about a sibling rollback file.

- **D1** — **PublicInvoicePage leaks WTC pricing internals via `select("*, proposal_wtc(*)")`**
  - `src/pages/PublicInvoicePage.jsx:44`
  - Evidence: `.from("invoice_lines").select("*, proposal_wtc(*, work_types(name))").eq("invoice_id", inv.id);`
  - The `proposal_wtc` anon SELECT policy is row-gated by `viewing_token` but not column-restricted. `*` returns `burden_rate`, `ot_burden_rate`, `tax_rate`, `markup_pct`, `materials` (jsonb with cost basis), `discount`, and other internal pricing fields the customer must never see. This is the same H6 shape that was fixed on PublicSigningPage via `get_public_proposal_view` RPC but was never applied to the invoice viewer.
  - Spec: CLAUDE.md (prior audit H6), OWASP A01, OWASP-ASVS V5.1.3.
  - Fix: introduce `get_public_invoice_view(p_token)` SECURITY DEFINER RPC returning a fixed customer-safe column list, or replace `proposal_wtc(*)` with an explicit allow-list omitting burden/markup/materials/tax_rate.

- **D2** — **PublicInvoicePage `select("*")` on invoices exposes Stripe/QB/internal identifiers**
  - `src/pages/PublicInvoicePage.jsx:33`
  - Evidence: `.from("invoices").select("*, proposals(...)")`.
  - `*` returns `qb_invoice_id`, `qb_payment_id`, `stripe_checkout_id/url`, `stripe_payment_id`, `tenant_id`, internal `description`, `sent_at`, `paid_at`. Even though row access is gated by `viewing_token`, the column projection leaks information unnecessary to render the customer view.
  - Spec: CLAUDE_RLS.md, OWASP A01.
  - Fix: project only the customer-visible columns explicitly (`id, status, amount, discount, sent_at, due_date, voided_at, void_reason, proposals(...)`).

- **D3** — **`Settings.handleSave` mutates `tenant_config` with no client role gate**
  - `src/pages/Settings.jsx:640` (Save button at `:673`).
  - Evidence: `const handleSave = async () => { ... await updateTenantConfig({ ...fields, ... }); }` — the Save button is `disabled={saving}` only, never `disabled={!canManage}`.
  - Any authenticated tenant member (including a plain Member/Sales Rep) can trigger handleSave from devtools or click the visible Save button. Combined with the missing `is_admin_or_manager()` gate on `tenant_config_update` (see B-MED finding below), a rep can rewrite billing goals, default burden/tax rates, validity windows, license number, monthly_billing_goal, etc. Defense in depth requires BOTH the UI gate AND the policy gate.
  - Spec: CLAUDE_RLS.md, OWASP-ASVS V4.1, OWASP A01.
  - Fix: disable the Save button when `!canManage` and add `AND public.is_admin_or_manager()` to the `tenant_config_update` policy USING/WITH CHECK.

- **F1** — **Proposal Summary Total falls back to stale `proposals.total` instead of `calcWtcPrice`**
  - `src/components/ProposalDetail.jsx:1081`
  - Evidence: `const sov = sovContractSum != null && sovContractSum > 0 ? sovContractSum : null; const fallback = parseFloat(p.total) || 0; const value = sov != null ? sov : fallback;`
  - When no SOV exists, the Summary panel's Total line displays `parseFloat(p.total)` (the stored `proposals.total`). `proposals.total` is only written on handleLock/handleSave and can be stale if a WTC was edited but not re-saved. CLAUDE.md Data Integrity Rule 2 explicitly requires the proposal summary total to be computed from live WTC data using `calcWtcPrice()`. The component already computes `breakdowns.reduce(...).price` from live data thirteen lines below (`:1094`).
  - Spec: CLAUDE.md Data Integrity Rule 2.
  - Fix: `const fallback = (wtcs || []).reduce((s, w) => s + calcWtcPrice(w), 0);` (importing `calcWtcPrice` from `src/lib/calc.js` if not already in scope).

- **A1** — **Permissive CORS via `origin.endsWith(".vercel.app")` in every money/admin function**
  - `supabase/functions/send-invoice/index.ts:15` (and 19 sibling functions: `send-pay-app/index.ts:69`, `send-proposal/index.ts:22`, `create-billing-session/index.ts:24`, `qb-record-payment/index.ts:76`, `qb-sync-invoice/index.ts:90`, `qb-void-invoice/index.ts:56`, `qb-link-customer/index.ts:79`, `qb-create-job/index.ts:124`, `qb-search-customers/index.ts:79`, `qb-auth/index.ts:17`, `invite-user/index.ts:13`, `delete-user/index.ts:12`, `deactivate-user/index.ts:12`, `reset-password/index.ts:22`, `stripe-webhook/index.ts:58`, `scc-stripe-webhook/index.ts:47`, `proposal-signed/index.ts:34`, `deactivate-payment-link/index.ts:13`, `extract-sov/index.ts:80`).
  - Evidence: `const isAllowed = ALLOWED_ORIGINS.includes(origin) || origin.endsWith(".vercel.app");`
  - Anyone can register `anything.vercel.app` and host JS that initiates credentialed cross-origin fetches against money/admin endpoints — the browser will attach the logged-in admin's JWT and CORS will whitelist the attacker origin. Every "authenticated" protection on these endpoints becomes CSRF-bypassable. Prior audit (H8) flagged this; still unfixed and now in 20 functions.
  - Spec: OWASP-ASVS V14.5.3, OWASP A05, CLAUDE.md (Production env vars / domains).
  - Fix: drop `endsWith(".vercel.app")` everywhere. For preview deploys, use a deployment-specific env-var token (or move preview testing to a dedicated function set), not a substring match.

- **C2** — **check-orphan-users has no authentication or method check**
  - `supabase/functions/check-orphan-users/index.ts:8`
  - Evidence: `serve(async (_req) => { ... await fetch("https://api.resend.com/emails", ... to: "chris7berger@gmail.com" })`
  - The handler ignores the request entirely. Any internet caller can hit it in a loop to (a) abuse Resend send quota with mail to `chris7berger@gmail.com`, (b) run `get_orphan_auth_user_count()` RPC under service role unbounded, and (c) inflate function-invocation billing. `supabase/config.toml` also has no `verify_jwt` entry for this function, so the platform doesn't gate it either.

- **C3** — **qb-auth has `verify_jwt = false` at the Supabase platform layer**
  - `supabase/config.toml:4`

  - Platform-level JWT verification is off.   - Spec: Supabase production checklist (verify_jwt), OWASP-ASVS V13.2.

- **C4** — **invite-user uses caller-supplied `senderEmail` as Resend `From` without verified-domain gate**

- **A2** — **Hardcoded publishable-key fallbacks in client bundle**

- **A5** — **Resend response bodies logged verbatim in five edge functions**
