## Findings by Severity

### Critical

### High

- **A1** — **invite-user logs full magic-link object containing one-time auth token**
  - `supabase/functions/invite-user/index.ts:135`
  - Evidence: `console.log("generateLink result:", { linkData, linkErr: linkErr?.message });`
  - `linkData.properties.action_link` is a single-use Supabase admin magic link that signs the invited user in. Anyone with Supabase function-log access can copy the link before the invitee uses it and complete password-set, achieving account takeover at admin/manager privilege.
  - Spec: OWASP-ASVS V7.1.1, OWASP A09, OWASP A07.
  - Fix: log `{ hasLink: !!linkData?.properties?.action_link, linkErr: linkErr?.message }` — never log `action_link`.

### Medium

- **A2** — **Hardcoded publishable-key fallbacks in client bundle**
  - `src/lib/supabasePublic.js:20` (`"sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK"`) AND `src/pages/Login.jsx:61` (`"sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK"`).
  - Two different fallback strings disagree, so any future key rotation silently breaks one path or the other. Also defeats env-var rotation (a missing env var keeps a stale baked-in key alive). Not a secret leak — these are publishable keys — but it violates CLAUDE.md's expectation that only env-supplied values are used.
  - Spec: OWASP-ASVS V14.2.1, CLAUDE.md (Production env vars).
  - Fix: drop the `||` fallback in both files; throw on missing env var as `src/lib/supabase.js:7-9` already does. In Login.jsx, import the configured client instead of duplicating URL+key inline.

- **A3** — **qb-auth logs Intuit token-exchange response bodies on failure**
  - `supabase/functions/qb-auth/index.ts:76` and `:147`.
  - Evidence: `console.error("qb-auth: token exchange failed:", JSON.stringify(tokenData));`
  - On error paths Intuit usually returns `{error, error_description}` (harmless), but a whole-object dump is brittle — any future endpoint shape that includes token material would write it to logs.
  - Spec: OWASP-ASVS V7.1.1, OWASP A09.
  - Fix: log only `{ status, error, error_description }`, never the whole response.

- **A4** — **invite-user logs invitee PII verbatim**
  - `supabase/functions/invite-user/index.ts:95`
  - Evidence: `console.log("invite-user invoked", { email, name, teamMemberId, senderEmail, senderName });`
  - Logs customer/team-member email + name on every invocation. Violates data minimization (GDPR Art. 5(1)(c)); `teamMemberId` is sufficient for correlation.
  - Spec: OWASP-ASVS V8.3.1, OWASP A09.
  - Fix: log `{ teamMemberId, hasEmail: !!email }` or drop entirely.

- **A5** — **Resend response bodies logged verbatim in five edge functions**
  - `supabase/functions/send-invoice/index.ts:268`, `send-proposal/index.ts:187`, `reset-password/index.ts:136`, `invite-user/index.ts:182`, `proposal-signed/index.ts:141`.
  - Evidence (send-invoice): `console.log("Customer email response:", emailRes.status, emailResBody);`
  - Resend error bodies can echo recipient/sender domain and validation details, landing customer PII in logs on failure paths. `send-pay-app` already has a `sanitizeProviderError` helper (line 42–53); the other five do not adopt it.
  - Spec: OWASP-ASVS V7.1.1, OWASP A09.
  - Fix: move `sanitizeProviderError` to `_shared/` and apply across all five before logging response bodies.

### Low

- **A6** — **stripe-webhook logs invoice amount + identifiers**
  - `supabase/functions/stripe-webhook/index.ts:97`
  - Evidence: `console.log("Payment completed for invoice:", invoiceId, "payment_intent:", paymentIntent, "payment_link:", paymentLinkId, "amount:", amountTotal);`
  - Business-sensitive amount in logs. Stripe is system of record — IDs alone trace.
  - Spec: OWASP-ASVS V8.3.4.
  - Fix: drop `amount` from the log line.

- **A7** — **send-proposal logs recipient + rep emails on every send**
  - `supabase/functions/send-proposal/index.ts:142`
  - Evidence: `console.log("send-proposal invoked", { proposalId, recipientEmail, repEmail, proposalNumber });`
  - Same hygiene concern as A4. `proposalId` traces it.
  - Spec: OWASP-ASVS V8.3.1, GDPR Art. 5(1)(c).
  - Fix: log `{ proposalId, proposalNumber, hasRecipient: !!recipientEmail }`.

- **A8** — **send-pay-app logs recipient email on invocation**
  - `supabase/functions/send-pay-app/index.ts:105`
  - Evidence: `console.log("send-pay-app invoked", { payAppId, invoiceId, recipientEmail });`
  - Same shape — ironic given the file already defines `sanitizeProviderError` just above.
  - Spec: OWASP-ASVS V8.3.1.
  - Fix: drop `recipientEmail` from the structured log payload.
