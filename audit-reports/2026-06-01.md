- **A1** — **Hardcoded publishable-key fallbacks in client bundle (carry-over)**
  - `src/lib/supabasePublic.js:20` AND `src/pages/Login.jsx:61`.
  - Evidence (supabasePublic.js): `import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_LL6wYC7kJDPQfGYeKAaLaWKoK";`
  - Evidence (Login.jsx): `apikey: import.meta.env.VITE_SUPABASE_ANON_KEY || "sb_publishable_v7XktVvkAlX7y5f6xoFjng_AaLaWKoK"`.
  - Two different fallback strings — a future key rotation silently breaks one path. Defeats env-var rotation generally. Not a secret leak (publishable keys), but it violates CLAUDE.md's expectation that only env-supplied values are used. Same shape as prior audit A2; unfixed.
  - Spec: OWASP-ASVS V14.2.1, CLAUDE.md (Production env vars).
  - Fix: drop the `||` fallback in both files; throw on missing env var as `src/lib/supabase.js:7-9` already does. In Login.jsx, import the configured supabase client and use `supabase.functions.invoke('reset-password', ...)` instead of hand-rolling fetch.

- **A2** — **Hardcoded Supabase URL fallbacks in client bundle**
  - `src/lib/supabasePublic.js:16` AND `src/pages/Login.jsx:58`.
  - Evidence (Login.jsx): `` `${import.meta.env.VITE_SUPABASE_URL || "https://pbgvgjjuhnpsumnowuym.supabase.co"}/functions/v1/reset-password` ``
  - Production project URL hardcoded as fallback. Not itself a secret (appears in DNS), but baking it in defeats env-driven environment separation — a staging/preview build with `VITE_SUPABASE_URL` unset will silently call prod.
  - Spec: OWASP-ASVS V14.2.1, CLAUDE.md (Production env vars).
  - Fix: drop the `||` fallback; require `VITE_SUPABASE_URL` via the central client and reuse it (or throw).

- **E1** — **`ws@8.19.0` uninitialized-memory disclosure (advisory GHSA-58qx-3vcg-4xpx)**
  - `package-lock.json:3025` — `"node_modules/ws"` resolves to `8.19.0`, in vulnerable range `>=8.0.0 <8.20.1` (pulled transitively via `@supabase/realtime-js`).
  - Advisory severity moderate (CVSS 4.4, PR:H). In this Vite SPA `ws` runs only as a browser WebSocket client and the app does not use Supabase realtime channels heavily — prod exposure is limited.
  - Spec: GHSA-58qx-3vcg-4xpx, OWASP A06.
  - Fix: `npm update ws` or bump `@supabase/supabase-js` to a release requiring `ws ≥8.20.1`.

- **E2** — **`pdf-lib@1.17.1` unmaintained (~4 yrs stale) in PDF signing path**
  - `package.json:17` — `"pdf-lib": "^1.17.1"` (released April 2021).
  - Used in the proposal PDF signing path (signed-proposals bucket). No active maintenance means future CVEs go unpatched. Security-adjacent (document integrity for customer-signed proposals). Same finding as prior E2; unfixed.
  - Spec: OWASP A06.
  - Fix: evaluate maintained fork (e.g. `@cantoo/pdf-lib`) or migrate signing to a maintained library; track in BACKLOG.

### Low

- **A3** — **`supabase/.temp/*` tracked in git (pooler URL + project ref leak)**
  - `supabase/.temp/pooler-url` (and sibling `project-ref`, `gotrue-version`, etc.) — 7 files tracked despite `.gitignore` covering `supabase/.temp/`.
  - Evidence: `postgresql://postgres.pbgvgjjuhnpsumnowuym@aws-1-us-west-1.pooler.supabase.com:5432/postgres` (pooler-url, single line).
  - Files were committed before the ignore rule landed. No password is present, so this is not credentials exposure — but it leaks the prod pooler host, role name, and port, narrowing the attack surface for credential-stuffing/pgbouncer probing. Project ref is also DNS-discoverable, so impact is informational hygiene.
  - Spec: OWASP-ASVS V14.3.2, OWASP A05.
  - Fix: `git rm --cached supabase/.temp/*` and commit; the directory is already gitignored.
