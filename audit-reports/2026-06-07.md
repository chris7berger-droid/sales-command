## Top 3 Urgent

1. **BCD-RET (NEW HIGH) — Retention-release path is unguarded at every layer.** New 2026-06-01 migration `20260601120000_invoices_retention_release.sql` added `retention_release_of` + `retention_released` columns but inherits only tenant-scoped UPDATE/INSERT RLS — no `is_admin_or_manager()`. `qb-sync-invoice/index.ts:104` uses `authenticateCaller` not `requireAdminOrManager`. `src/pages/Invoices.jsx:1319 handleBillRetention` runs unguarded and sends a client-supplied `amount`. A Sales Rep (or tampered client) can write off held retention, mint phantom release invoices with arbitrary amounts, and propagate them to QB. Fix in one PR: add admin gate at all three layers + `CHECK (amount = source.retention_amount)`.
2. **D10–D13 cluster (1 carry-over + 3 NEW HIGH) — Invoices.jsx + PayAppDetailModal money handlers run with no role gate.** `Invoices.handleDelete` (`:1219`, D10), `handleSaveEdit` (`:1275`, D11 NEW), `handlePullBack` + `handleVoidConfirm` (`:1392`, `:1414`, D12 NEW), and `PayAppDetailModal.handleSaveLines` + `handleDeletePayApp` (`:232`, `:263`, D13 NEW) all write money/state with no role check; `Invoices.jsx` doesn't even compute `canManage`/`canDelete` for `InvoiceDetail`. Any active team member can void/edit/delete any tenant invoice or pay app and corrupt the books. Fix: introduce `canManage = ["Admin","Manager"].includes(teamMember?.role)`; gate all five handlers + their buttons; AND add `AND public.is_admin_or_manager()` to `invoices`, `invoice_lines`, `billing_schedule_pay_apps`, `billing_schedule_pay_app_lines` UPDATE/DELETE RLS.
3. **C2 (CRITICAL) — Gate `check-orphan-users` immediately.** `supabase/functions/check-orphan-users/index.ts:8` still runs unauth: one `curl` is enough to invoke service-role `get_orphan_auth_user_count()` and trigger a Resend email per request. No `verify_jwt` entry in `config.toml`. Open 21+ days. Fix: add `[functions.check-orphan-users] verify_jwt = true` to `config.toml` AND require a timing-safe `CRON_SECRET` header.
