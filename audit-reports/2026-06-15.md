## Top 3 Urgent

1. **F2 — Retention-release invoice Edit zeroes `amount` (CRITICAL, NEW).** `src/pages/Invoices.jsx:1240` — `handleSaveEdit` has explicit preserve-or-recompute branches for archive (`isArchiveInvoice`) and pay-app (`billing_schedule_line_id`) invoice shapes, but NO branch for retention-release invoices. A retention-release invoice has zero `invoice_lines` (handleBillRetention only inserts the invoice row), so `newLines = lines.map(...) = []`, `newAmount = 0`, and the UPDATE writes `amount: 0`. The Edit button is shown at `:1741` for `isNew`. This is the identical mechanism CLAUDE.md Data Integrity Rule #6 (added 2026-06-04 after `14000c5` archive zeroing and `33c385e` pay-app zeroing) was written to prevent. Fix: add a `retention_release_of` preserve branch at the top of `handleSaveEdit` that returns `{ id, amount: parseFloat(inv.amount) || 0 }` for the release shape.

2. **B-family soft-delete sweep + B4/B5 unscoped public RPCs (HIGH × 7).** `proposals_public_sign_token` (`20260510120000_signing_token_expiry_and_consume.sql:201`), the WTC/call_log siblings at `:212` and `:228`, `invoices_public_view_token` (`20260520095000_invoice_viewing_token_expiry.sql:41`), and `mark_proposal_signed` (`20260522160000_b37_mark_signed_lineage.sql:37`) still don't filter `deleted_at IS NULL`. Combined with B4/B5 (`get_public_tenant_config`/`get_rep_contact` unscoped — `20260429180000_add_public_rpc_functions.sql:18,34`), a multi-tenant deploy leaks branding + rep PII across tenants AND keeps soft-deleted records customer-signable. Persistent at the same `file:line` for ≥21 days. Fix: add `AND deleted_at IS NULL` everywhere; make both public RPCs token-parameterized and resolve `tenant_id` through the proposal/invoice row.

3. **D-family client-only role gates (HIGH × 6) — and one NEW: D11 `handleBillRetention` has no role gate.** `Invoices.jsx:1319` (NEW) lets any authenticated user spawn a retention-release invoice via direct table writes. Pairs with the unfixed `Settings.handleSave` (`Settings.jsx:640`), `ProposalDetail.handleDelete` (`ProposalDetail.jsx:466`), `CallLogDetail.handleDelete` (`CallLogDetail.jsx:249`), `Invoices.handleDelete` (`Invoices.jsx:1208/1219` — was 1160, shifted by retention insertions), and `Home/SalesDash` client-filter trust (`Home.jsx:108`, `SalesDash.jsx:111`). Fix path is the same for all: add RLS UPDATE/DELETE policies gated by `is_admin_or_manager()` + scoped `sales_name = current_user_name` for the role-aware soft-delete paths.

## Findings by Severity

### Critical

- **F2** — **Retention-release invoice Edit zeroes `amount` (Rule #6 violation, NEW this week)**
  - `src/pages/Invoices.jsx:1240` (handler) + `:1741` (Edit button shown for `isNew`) + `:1319` (release invoice created with no `invoice_lines`).
  - Evidence: `const newLines = lines.map(l => { if (isArchiveInvoice) ...; if (l.billing_schedule_line_id) ...; const wtc = l.proposal_wtc; const wtcTotal = wtc ? calcWtcPrice(wtc) : 0; ... });` — no `retention_release_of` branch.
  - The release invoice is created at `:1342-1357` with `retention_release_of: source.id`, `status: "New"`, `amount: source.retention_amount` and NO matching `invoice_lines` rows. When `lines = []`, `isArchiveInvoice` is `false` (requires `lines.length > 0`), the `.map` short-circuits to `[]`, `newAmount = 0`, and the invoice UPDATE writes `amount: 0`. The retention dollars silently disappear from the customer-billable record. This is the exact mechanism Data Integrity Rule #6 was authored to prevent (the prior `14000c5` archive and `33c385e` pay-app incidents had the identical shape). NEW since 2026-06-01 — the retention-release feature shipped 2026-06-01 / 2026-06-09 without an explicit Rule #6 branch.
  - Spec: CLAUDE.md Data Integrity Rule #6 + Rule #7 ("hiding a field in the UI is not guarding it in the save"), OWASP-ASVS V5.3.4, OWASP A04.
  - Fix: insert a `retention_release_of` preserve branch as the first conditional in handleSaveEdit:
    ```js
    if (inv.retention_release_of) {
      return { id: l.id, billing_pct: null, amount: parseFloat(l.amount) || 0 };
    }
    ```
    AND short-circuit the outer compute to keep `newAmount = parseFloat(inv.amount) || 0` when `inv.retention_release_of` is set.
