# Sales Command Security Audit — 2026-06-22

## Summary

**0 Critical, 22 High, 14 Medium, 5 Low — posture IMPROVED.** The T1 security remediation (commit `b8fc1e9`, 2026-06-09) closed 6 prior Pass C findings — most importantly C1 (`.vercel.app` CORS bypass across 20 functions), C2 (`check-orphan-users` unauth+service-role), and C5/C6 (Stripe webhook idempotency). No carry-over CRITICAL remains. The RLS surface (Pass B) is unchanged — every B1-B17 prior finding is still persistent — and 4 NEW HIGH findings landed this week, all in features that shipped without admin role gates: `Invoices.handleBillRetention` (retention release, new this period), `ProposalDetail.handleSendToSchedule` (cross-driver write into Schedule Command's `job_wtcs`), `Invoices.handlePullBack` (now surfaced; may have been latent), and `PayAppDetailModal.handleSaveLines` divergence between `invoices.amount` and `invoice_lines.amount` (B42 in BACKLOG — escalated here under Data Integrity Rule #6 because qb-sync-invoice will ship stale per-line numbers). One E1 regression (new advisory GHSA-96hv-2xvq-fx4p added to the already-pinned `ws@8.19.0`).

## Top 3 Urgent

1. **Add admin gates to the four new/exposed money-mutating handlers in `Invoices.jsx` and `ProposalDetail.jsx`.** `handleBillRetention` (`src/pages/Invoices.jsx:1319`) inserts a NEW invoice row for the held retention amount with no role check; `handleSendToSchedule` (`src/components/ProposalDetail.jsx:513`) upserts `job_wtcs` cross-driver as the authenticated user; `handlePullBack` (`:1392`) resets Sent/Paid invoices back to New; `handleDelete` (`:1204`, persistent D10) soft-deletes any invoice. Fix: compute `canManage = ['Admin','Manager'].includes(teamMember?.role)` and gate button + handler; AND add `is_admin_or_manager()` to the `invoices` UPDATE/INSERT RLS and to the `job_wtcs` INSERT RLS in the Schedule Command repo. (B10/B11 RLS context applies.)
2. **Fix the `B42` pay-app save divergence under CLAUDE.md Data Integrity Rule #6.** `PayAppDetailModal.handleSaveLines` (`src/components/PayAppDetailModal.jsx:249`) updates the header `invoices.amount` from recomputed lines but never UPDATES the per-line `invoice_lines.amount`. `qb-sync-invoice` builds QB Invoice lines from `invoice_lines.amount`, so after a pay-app re-edit the next QB sync pushes stale per-line dollars and the QB invoice's header total + line sum diverge. Same incomplete-edge-coverage class as the 2026-04-20 and 2026-06-04 zeroing incidents — Rule #6 is exactly the rule the codebase added to prevent this. Fix: in `handleSaveLines`, also UPDATE each `invoice_lines` row's `amount` and `billing_pct` from the recomputed pay-app lines in the same write that updates the header (closes BACKLOG B42).
3. **Sweep the anon `deleted_at` filter gap (B2/B2b/B2c/B3) and the unscoped public RPCs (B4/B5).** Persistent across 4+ weeks. `proposals_public_sign_token`, `proposal_wtc_public_read_token`, `call_log_public_read_token`, `invoices_public_view_token` all still let anon callers read soft-deleted rows via valid token until expiry; `mark_proposal_signed` token lookup (B15) has the same gap and lets a customer sign a back-office-retracted proposal. `get_public_tenant_config` and `get_rep_contact` still return globally-first rows, leaking branding + rep PII cross-tenant the moment a second tenant is added. Fix: a single migration adding `AND deleted_at IS NULL` (or `AND p.deleted_at IS NULL` where joined) to every anon viewing/signing-token policy and to `mark_proposal_signed`'s primary scan; AND rewrite the two public RPCs to take `p_token` and resolve `tenant_id` through the proposal/invoice row.
