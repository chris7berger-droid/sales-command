# Sales Command Security Audit — 2026-06-28

## Summary

2 Critical, 25 High, 26 Medium, 9 Low — posture **materially improved on infra; new client-side regressions in pay-app flow**. The prior CRITICAL (`check-orphan-users` unauth) is fully resolved by the S6 cron-secret + `verify_jwt=true` shipped 2026-06-09. CORS substring bypass (C1/C17) is gone; all 22 functions now use the exact-host allowlist in `_shared/cors.ts`. Stripe webhook idempotency (C5/C6) is implemented via `processed_stripe_events` claim/done table. `PublicInvoicePage` invoices `select("*")` (D1) is closed — explicit column list + column-level anon GRANT (mig 20260625130000+140000). However, two CRITICAL items survive (`rollback_drop_public_read_policies.sql` re-introduces the 2026-04-26 anti-pattern, and `team_members` write policies still allow self-elevation), and two new client-side money-write modules (`PayAppDetailModal`, `NewPayAppModal`) shipped with **no role gate** on create/edit/delete handlers, while `PayAppDetailModal.handleSaveLines` is a **Data Integrity Rule 6 trap** — recomputes `invoices.amount` without cascading to `invoice_lines.amount` that the public viewer renders. `reset-password` references an undeclared `origin` variable that throws every time, leaks via raw `error.message`, and silently breaks Schedule Command branding.

## Top 3 Urgent

1. **B11 — `team_members_update` self-elevation (CRITICAL).** `sql/rls_identity_tables.sql:64` allows any tenant member to UPDATE any team_members row in their tenant, including their own `role`. A Sales Rep can flip themselves to Admin/Manager, which unlocks every other role-gated policy (tenant_config writes, invoice voids, manager dashboards). Fix: add `AND public.is_admin_or_manager()` to USING and WITH CHECK on `team_members_insert/_update/_delete`. This has carried over for at least three audit cycles and is the single most weaponizable gap in the codebase.
2. **B6 — `rollback_drop_public_read_policies.sql:4` rebuilds the 2026-04-26 anti-pattern (CRITICAL).** Live in-repo SQL that operator can run as "rollback" and that re-creates `team_members_public_read USING (active = true)` and `tenant_config_public_read USING (true)` — exposing every active team_member's email/phone/auth_id AND every `tenant_config` field (stripe_customer_id, stripe_subscription_id, monthly_billing_goal, license_number) to the publishable anon key. Fix: delete the file. Correct rollback is to revert the migration commit, not re-create policies the incident report named as the cause.
3. **PayAppDetailModal — three money handlers with no role gate AND a Rule 6 cascade trap (HIGH).** `src/components/PayAppDetailModal.jsx:232` (handleSaveLines), `:263` (handleDeletePayApp), and the sibling `src/components/NewPayAppModal.jsx:123` (handleCreate) all write to `billing_schedule_pay_apps`, `billing_schedule_pay_app_lines`, and `invoices` with **zero** role check (no `canManage` import, no `teamMember` prop). Any Sales Rep can mint a pay app, edit billed amounts post-send, or delete the linked invoice. Worse, `handleSaveLines` updates `invoices.amount` and `retention_amount` but does **not** propagate the new per-line amounts into `invoice_lines.amount` — which is what `PublicInvoicePage` and `InvoiceDetail` render. Customer's public page and the printable PDF show stale line dollars while the header total is fresh. Fix: (a) add `teamMember` prop + `canManage = isAdminOrManager` gate to all three handlers, (b) add RLS UPDATE/DELETE policies on `billing_schedule_pay_apps` restricted to admin/manager, (c) in `handleSaveLines`, after updating pay-app lines, UPDATE the matching `invoice_lines` rows by `billing_schedule_line_id + invoice_id`.
