## Top 3 Urgent

1. **C1 — `sql/rollback_drop_public_read_policies.sql:4` rebuilds the 2026-04-26 anti-pattern (CRITICAL, PERSISTENT — 4th cycle).** Live in-repo rollback script that any operator can paste into the SQL editor. 2. **C3 — `sql/db_hardening_phase1.sql:12` re-creates the tenant-blind `get_user_tenant_id()` (NEW CRITICAL).** SECURITY DEFINER body returns `SELECT id FROM public.tenant_config LIMIT 1;` for any caller — no `auth.uid()`/`team_members` check and no `SET search_path`. Migration `20260509120000_get_user_tenant_id_remove_fallback.sql` explicitly removed this body because it (a) breaks multi-tenant isolation once a second tenant is provisioned, and (b) any leaked JWT + this function returns a valid tenant_id satisfying every authenticated policy's tenant check. Fix: rewrite this seed to match the current prod body (`SELECT tenant_id FROM team_members WHERE auth_id=auth.uid() AND active=true LIMIT 1` with `SET search_path = public`).
3. **C2 — `sql/rls_identity_tables.sql:64` — `team_members` writes still allow self-elevation (CRITICAL, PERSISTENT — 4th cycle).** `team_members_update` USING/WITH CHECK only enforces `tenant_id = get_user_tenant_id()`. Fix: add `AND public.is_admin_or_manager()` to USING and WITH CHECK on all three policies; add a `update_my_profile()` SECURITY DEFINER RPC that whitelists non-role columns for self-edits.
