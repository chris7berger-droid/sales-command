## What's NOT done — next session, in this priority order

### Critical (still open from the audit)

- **C1** — `call_log` anon UPDATE policy uses `WITH CHECK (true)` —
  signing-token holder can rewrite any column on the linked `call_log`
  row. File: `supabase/migrations/20260427180000_add_token_gated_policies.sql:171`.
  Fix: replace `WITH CHECK (true)` with the EXISTS check, or move the
  stage update into a `SECURITY DEFINER` RPC.
- **C2** — Legacy seed scripts at `sql/rls_core_tables.sql` and
  `sql/rls_child_tables.sql` ship the 2026-04-26 anti-pattern across 9
  policies on 7 tables. Live DB is patched, but a fresh deploy
  re-introduces the incident. Fix: rewrite seed policies to require
  `request_signing_token()` / `request_viewing_token()` match.
- **C3** — `proposal_recipients` anon UPDATE policy permits arbitrary
  column writes (token holder can flip `role` viewer→signer).
  `supabase/migrations/20260427180000_add_token_gated_policies.sql:105-122`.
  Same fix shape as C1.
- **C9** — SSRF in `send-pay-app` — caller-supplied PDF URLs fetched with
  no allowlist; can hit `http://169.254.169.254/` and exfil bytes via
  email. `supabase/functions/send-pay-app/index.ts:134-137`. Fix:
  restrict PDF URLs to a Supabase Storage allowlist. Also has a tenant
  binding gap on `payAppId`/`invoiceId` — same _shared/tenantAuth pattern
  applies.
- **C10** — `proposal-signed` accepts `callLogId` from the request body
  with no check that `proposals.call_log_id === callLogId`. Any signing-
  token holder can flip another tenant's `call_log.stage`.
  `supabase/functions/proposal-signed/index.ts:55-56`. Fix: read
  `call_log_id` from the DB row, not the body.
- **C12** — `sql/rollback_drop_old_anon_signing_policies.sql` is a
  one-click footgun re-creating the 2026-04-26 anti-pattern. Fix: delete
  the file.

## Deploy — DO NOT MERGE TO MAIN UNTIL TESTED

The fixes on this branch include real behavior changes:

- New schema migration `20260501120000_qb_connection_tenant_id.sql` —
  must run before the functions deploy or QB calls will fail with
  "No QuickBooks connection found for tenant".
- `qb-auth` now requires `role IN ('admin','manager')` for exchange/refresh.
  If your team's `team_members.role` values don't match exactly, the
  reconnect flow will 403.
- `send-invoice` reads customer email from
  `proposals → call_log → customers.contact_email/email`. Invoices whose
  customer record has no contact email will fail with "No customer email
  on file". Audit your `customers` table before merging.
- `create-billing-session` no longer falls back to "any tenant" — every
  call now requires the caller to be in `team_members` with a tenant_id.
  Service-role internal calls are explicitly rejected (none expected,
  but worth verifying).

**Recommended deploy order:**

1. Apply migration `20260501120000_qb_connection_tenant_id.sql` to the
   target environment (staging first).
2. Deploy the 5 changed edge functions to staging.
3. Smoke test: QB exchange/refresh/status, send-invoice from a real
   proposal/customer pair, create-billing-session checkout + portal,
   qb-record-payment from stripe-webhook, qb-void-invoice from the UI.
4. Only then merge to main and deploy prod.
