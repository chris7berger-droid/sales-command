DECISIONS / CHOICES MADE
─────────────────────────
- Trigger over policy WITH CHECK: a `WITH CHECK (tenant_id =
  parent.tenant_id)` would also work but requires the caller to pass
  the right tenant_id. Trigger is bulletproof — caller's value is
  always overwritten with the parent's, regardless of what they send.
- SECURITY DEFINER + fixed search_path: matches H1/H7 pattern
  (`20260505181452_set_search_path_security_functions.sql`).
  Without it the trigger would couple to anon's RLS on `proposals`,
  which is redundant with the existing token-match policy and gives
  worse error semantics on token mismatch.
- Pre-prod testing on a scratch Supabase project, not local. The
  local stack can't spin up — the migrations ledger has at least one
  gap (`billing_schedule` is altered before any migration creates it).
  `supabase db pull` hits the same shadow-DB issue. Pivoted to a
  throwaway Pro-tier project (ref `eguyilfigafpwpwspxed`), tested
  with a two-tenant fixture, deleted after.
- Anon INSERT policy left token-only on purpose. Adding a
  tenant-binding `WITH CHECK` clause is redundant once the trigger
  derives tenant_id authoritatively.
- Older redundant DELETE policy ("Allow delete on proposal_signatures",
  role/sales-name based) left in place. Out of scope; a follow-up
  pass should consolidate it with `proposal_signatures_delete`.
