DEVIATIONS FROM PLAN (worth folding into ops knowledge)
────────────────────────────────────────────────────────
1. **Migration ledger gap is real and active.** The plan asserted the full
   ledger applies cleanly to a fresh project. It does not — the first
   migration (`20260416132135_billing_schedule_multi_docs.sql`) ALTERs
   `public.billing_schedule`, which was created out-of-band (not in any
   `sql/` seed file in the repo, presumably created via Studio direct DDL
   in March 2026). Existing v101 carryforward; documenting the workaround.

   **Workaround pattern: dump-restore + schema_migrations backfill.**
   Run from `~/sales-command`:
     # 1. Re-link prod (currently scratch-linked).
     supabase link --project-ref pbgvgjjuhnpsumnowuym --password "$PROD_PWD"
     # 2. Dump prod schema-only (skip auth — Supabase auto-creates).
     PGPASSWORD="$PROD_PWD" supabase db dump --linked \
       -s public,archive,storage,supabase_migrations -f /tmp/prod-schema.sql
     # 3. Re-link scratch.
     supabase link --project-ref <scratch-ref> --password "$SCRATCH_PWD"
     # 4. Apply via docker psql as postgres superuser through pooler
     #    (NOT supabase db query --linked — see Deviation 2).
     #    pooler URL is in supabase/.temp/pooler-url.
     docker run --rm --env "PGPASSWORD=$SCRATCH_PWD" -v /tmp:/tmp \
       public.ecr.aws/supabase/postgres:17.6.1.063 \
       psql -h aws-1-us-west-1.pooler.supabase.com -p 5432 \
            -U "postgres.<scratch-ref>" -d postgres \
            -f /tmp/prod-schema.sql
       # ON_ERROR_STOP=0 (default) is needed: storage schema statements
       # fail with "permission denied for schema storage" because the
       # storage schema is reserved for `supabase_storage_admin`. Public
       # and archive apply cleanly. Cross-references between schemas were
       # confirmed zero before relying on this.
     # 5. Backfill schema_migrations so push only runs the new migration.
     INSERT INTO supabase_migrations.schema_migrations (version)
     VALUES ('<each version up through prior to the new one>')
     ON CONFLICT (version) DO NOTHING;
     # 6. Now `supabase db push --linked` only attempts the new migration.

   **Security:** prod password via PGPASSWORD env (not in argv); shred
   dump file post-test (`dd if=/dev/urandom of=/tmp/prod-schema.sql
   conv=notrunc; rm`). Treat dump as sensitive — full SECURITY DEFINER
   bodies in cleartext.

2. **`supabase db query --linked` runs as a constrained role.** The
   management API token used by `--linked` is not a postgres superuser
   and cannot modify the storage schema. For schema-level work in scratch
   that the management API blocks, switch to **direct docker psql via
   the session-mode pooler** (port 5432, user `postgres.<ref>`, pooler
   host from `supabase/.temp/pooler-url`). Direct connections to
   `db.<ref>.supabase.co` are no longer routable on current Supabase
   plans — pooler-only.

3. **RLS WITH CHECK fires before NOT NULL constraints.** Plan predicted
   orphan INSERT would surface `23502 not_null_violation` when the
   `DEFAULT public.get_user_tenant_id()` resolves to NULL. Actual error
   is `42501 RLS row-level security policy violation` because RLS
   WITH CHECK evaluates the DEFAULT-resolved row first and denies before
   the NOT NULL constraint validates. Stricter deny, same outcome —
   no caller-visible behavioral difference, but worth noting in the
   handoff for future plan-writers expecting 23502.

ANTIFRAGILE OBSERVABILITY (steady-state, post-apply)
─────────────────────────────────────────────────────
The DO block in the migration was a one-shot apply-time gate. Steady-state
guard is `public.v_orphan_auth_users`. Recommend pairing with a scheduled
query that alerts ops if `count(*) > 0`. Reads must use service_role
(Studio SQL editor, scheduled cron edge functions) — the view is
SECURITY INVOKER over `auth.users` which `authenticated` cannot read,
and the migration left no GRANT to authenticated by design.

Suggested cron query (run on Supabase pg_cron or a scheduled edge fn):
  SELECT count(*), array_agg(email) AS orphan_emails
    FROM public.v_orphan_auth_users;
If non-zero, alert. Triage = provision a `team_members` row or call the
deactivate-user edge function to clear the auth_id.

team_members. Deferred — GoTrue owns auth.users writes and trigger
errors there surface as opaque 500s in signup. Re-evaluate when F7
(multi-tenant onboarding auto-provision) lands.
