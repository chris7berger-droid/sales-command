DECISIONS / CHOICES MADE
────────────────────────
- Recipient email enforcement: B (soft). User can type a recipient,
  but the server validates it against a server-built allowlist:
  customer_contacts.email ∪ customers.{billing_email, contact_email,
  email} for the pay app's OWN customer chain. Lowercased/trimmed
  equality. Tenant scope is implicit via the customer_id chain (pay
  app → schedule → proposal → call_log → customer, all tenant-scoped);
  customer_contacts has no tenant_id column. Most likely real-world
  failure mode: rep tries to send to a one-off email not on file,
  gets 400 "Recipient email must match a known contact." Fix is to
  add the email to customer_contacts.
- Single-use semantic for H5 (deferred to next session): action-only
  consume — signing fails after first success (mark_proposal_signed
  sets consumed_at), but get_public_proposal_view stays permissive on
  consumed_at so the customer can revisit the signed page.
- Two branches: C9/QB this session, H5 next session. Different review
  surface (edge functions vs schema migration + RPC). Per audit
  handoff lean.
- send-proposal trust hardening: filed as H11 (separate work item),
  not bundled into C9 branch. Keeps the C9 review surface tight; the
  H11 fix is a substantial rewrite of send-proposal with the same
  shape as the C8 fix on send-invoice.
