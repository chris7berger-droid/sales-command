DECISIONS / CHOICES MADE
────────────────────────
- Cherry-pick the 7 fix commits onto main rather than merge the whole
  fix/co-wizard-prefill-and-jobnum branch. Reason: the branch was
  based on feat/multi-gc-allocation which carries 13 docs/plans
  commits ahead of main (Multi-GC planning). Cherry-pick kept main's
  history clean and left the planning docs on the feature branch
  until implementation actually starts. Per [Cherry-Pick Contextual
  Coupling] memory.

- CO inheritance applies to BOTH wrap-into-parent AND standalone CO
  treatments. Reason: schema has no concept of a different-customer
  CO; standalone is just billing/tracking separation, not customer
  identity separation. Audit ratified this scope before build started.

- contactInfo edits during CO flow flow back to the parent customer
  record (kept customerMode='existing' triggering save()'s update
  block at :229-244). Matches non-CO existing-customer behavior. No
  extra flag to gate it.

- Parent proposal selected as "most-recent non-deleted" rather than
  status-filtered (e.g., status='Sold'). Simple, matches how other
  spots pick "the" proposal. If a parent had a draft + a sold, draft
  could be picked — Chris confirmed 6844 only has the archive (Sold)
  proposal so the question didn't bite; flagged for future revisit.

- PW=true on fresh archive-parent CO WTC is NOT caused by this code.
  Verified by removing the PW autoset block entirely — issue still
  expected to surface, because archive proposals have no
  proposal_wtc rows for my code to read. Likely sources: DB default
  on proposal_wtc.prevailing_wage column, or another code path
  somewhere. Deferred to a separate investigation session; not a
  ship blocker.

- Null parent.customer_id (legacy archive-import parents — see
  project_archive_no_skeleton memory) blocks at parentJob/coTreatment
  validateStep with a clear inline error, rather than falling back
  to the customer-steps flow. Audit-recommended path; surfaces the
  data gap explicitly.
