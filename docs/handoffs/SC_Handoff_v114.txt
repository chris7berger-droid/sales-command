DECISIONS / CHOICES MADE
────────────────────────
Plan-agent shape: general-purpose, not Plan subagent. The Plan
subagent (per its spec) lacks Edit/Write/NotebookEdit tools, so it
can return a plan as a message but cannot persist it. The file-as-
memory planning pattern requires durable amendment in the plan doc
on the feature branch, which only general-purpose can satisfy. Chris
ratified this framing before the spawn.

Scope picked B over A and C. A = §5 amendment only; B = §5 + §10 step
6 amendments same round; C = plan + write Migration 1b SQL. B because
(a) F16's stated unblock is step 6, and leaving step 6 text stale
creates a future trap where a fresh build agent reads the old
work_type_id-keyed join and ships a broken sync RPC, (b) one fresh-
context round is cheaper than two with reload, (c) one-shipping-
agent's-context > two amendment passes on cognitive load. C was
explicitly rejected as discipline violation (Loop #1 was locked in
plan mode; writing migration SQL is build territory) and as a
correctness risk (Migration 1b shape decisions — ON DELETE behavior,
NULL semantics, index partial-vs-full — emerge during the plan; writing
SQL while those are soft is exactly the bridge-then-fix anti-pattern).

8 ratifications, locked verbatim into plan doc — summary of the
notable calls (not the cosmetic ones):

#1 Multi-gen chain support REJECTED, not deferred. Chris pushed
   harder than the agent's "defer to v1+" rec: chains are *prohibited*,
   not postponed, and #4 enforces that at the clone gate.

#4 Hard block on clone-of-clone via
   `RAISE EXCEPTION 'NESTED_CLONE_NOT_SUPPORTED'` when
   `v_source.cloned_from_proposal_id IS NOT NULL`. No silent allow.
   Pairs with #1.

#6 Empty-result modal REJECTED suppression — modal always pops, even
   when sync produced zero pending and zero conflicts. Chris's
   reasoning: estimators are not computer-savvy; silent application
   erodes trust; the empty modal acts as safety backstop. This
   reshaped #7(c) — empty-state copy now needs explicit confirmation
   text, not "0 siblings will receive this" (which reads as
   "did anything happen?"). Locked copy:
   "Saved. No GCs needed this update — your other proposals don't
   carry this WTC, or this field is locally edited everywhere."

#7(e) Chris-added during walkthrough — GC names rendered from
   `customers.name` via JOIN on proposals.customer_id (already
   shipped in Migration 1a). Sync RPC return shape must include
   `customer_name` per sibling. Fallback when customer unassigned:
   render proposal number. Edge cases: truncate names at ~30 chars;
   for 5+ siblings show first 3 + "and N more." Was implicit in
   the data model; making it explicit prevents the build agent from
   shipping GC-A/B/C labels.

#8 Verified Round-3 AUDIT_LOG before locking. Agent's amendment
   claimed "AUDIT_LOG implied ratification" on start_date/end_date
   per-GC carry-over; grep of AUDIT_LOG showed zero mention. The
   plan doc itself at §4:548 had the soft "Suggest: per-GC, not
   source-driven — but ask Chris" — that was the latest call. Locked:
   field-level "never sync" rule + clone carries parent's value at
   clone time (estimator has a default to adjust per-GC).
