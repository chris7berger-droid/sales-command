DECISIONS / CHOICES MADE
────────────────────────
Three [DESIGN-OPEN] items raised by Amendment 1, all ratified same-
session via the walk-through pattern (one item, short context, lock,
next):

#1 Resolution helper placement — `useResolvedCustomer(p)` hook in
   `src/lib/proposalCustomer.js`, NOT inline `??` per component.
   Reasoning: the 2-line rule will appear in 3+ component files
   (ProposalDetail, ProposalPDFModal, and any caller that resolves
   customer off a proposal). Hook centralizes the rule; if Sweep-2
   adds another fallback layer or the field set widens, one file
   changes instead of N. Chris's framing was "safer per artifact" —
   he was tracking the inverse mental model (B is less local, not
   more) until clarified; correct conclusion either way. Locked B.

#2 View name — `v_proposal_customer_resolved`, NOT
   `proposals_with_effective_customer` or `v_proposals_customer`.
   Reasoning: repo has exactly one existing view (`v_orphan_auth_users`)
   so the `v_` prefix is convention. Picked the rule-naming over
   entity-naming — the view's whole purpose is the resolution rule
   (COALESCE customer_id chain), not the proposal+customer entity
   shape. Reads honestly as "the customer after the ?? rule is
   applied."

#3 Schema-cache curl probe target — prod read-only via anon key,
   NOT staging. Reasoning: sales-command has no staging Supabase
   project. Migration 1a was applied direct to prod. Probe is one
   `curl …select=id,customers!proposals_customer_id_fkey(id)&limit=1`
   — single row, read-only by construction, same shape as every
   Vercel-preview page load. Zero risk to verify against the actual
   deploy target. Spinning up staging just for one curl is theatre.

Other notable calls from the audit body itself (not ratifications,
but worth flagging as the shape of the session):

A1.1 — When V5 said "Class A derives single customer from a
   proposal," it meant the bare id only. The `customers` join
   object on the same 4 sites silently follows the same wrong
   branch when `proposals.customer_id ≠ call_log.customer_id`.
   Verified via direct read of all four call sites
   (send-invoice:72-73, send-pay-app:181-182, ProposalDetail:282-287,
   ProposalPDFModal:31-33). All four consume the joined `customers`
   record for downstream reads (billing_email, contact_email,
   customer_name, allowed-recipient set). Fix at id-line alone is
   incomplete. Locked dual `customers` embed via the
   `proposals_customer_id_fkey` FK (auto-named by Migration 1a
   lines 19-21).

A1.2 — `Customers.jsx:519`'s `call_log!inner(customer_id)` blocks
   `.or()` from reaching proposals where `proposals.customer_id`
   matches but `call_log.customer_id` doesn't. SQL INNER JOIN
   resolves before PostgREST applies `.or()`. V5's
   "`.or()` or refactor to a view" wording is correct in direction
   but `.or()` alone won't fix `:519`. Locked the view path.
