DECISIONS / CHOICES MADE

Eight [DESIGN-OPEN] items raised by Amendment 1, all ratified same-
session via the walk-through pattern (one item, short context, lock,
next). Full table in §10 Step 4 Ratifications block.

#1 A2.2 Invoices transitive dependency — plan-hygiene note only,
   no code change. The `:521-525` invoices fetchAll has no
   customer_id filter; customer scoping happens at `:530` via
   `jobIds.has(...)` intersection. When site #2 swaps to
   `customer_jobs` per recommendation (b), the intersection auto-
   heals. Captured as plan addendum so future invoice-by-customer
   surfaces don't bypass the intersection.

#2 A2.3 `customer_proposals` substrate sharing — SELECT FROM
   `v_proposal_customer_resolved` (step 3's view), not inlined
   COALESCE. Same reasoning as Step 3 Ratification #1 (centralized
   over inlined). One resolution rule lives in one schema object.
   Build-order coupling: step 3 view migration must land before
   step 4 RPC migration (already correct in §10 order).

#3 A2.4 Role gate — none. Read-only surface consumed by every
   authenticated user (incl. Sales). Matches
   `archive_filter_options_rpc` precedent (read = no gate; money =
   gate). Adding `is_admin_or_manager()` here would lock sales reps
   out of their own customers' jobs overnight.

#4 A2.4 Top-of-body guards — NO_TENANT raise + customer-tenant
   match check. Mirrors `delete_customer:119-138`. Surfaces config
   errors loudly instead of silent empty results (which would be
   visually identical to "customer has no jobs"). Defense-in-depth
   for F7 multi-tenant.

#5 A2.4 Defense-in-depth tenant filter — explicit
   `AND tenant_id = v_tenant_id` inside the final SELECT (on
   call_log for customer_jobs, on the view for customer_proposals,
   AND inside the EXISTS sub-SELECT on proposals). SECURITY DEFINER
   bypasses RLS; explicit predicate is the only protection against
   tenant leak under data drift. Belt + suspenders.

#6 A2.6 merge_customers gap scope — separate T2 BACKLOG row (B19)
   + own migration. Step 4 is read-side, merge is write-side;
   different concerns → different migrations → cleaner rollback
   granularity. Filed same session.

#7 A2.6 NewPayAppModal + PayAppDetailModal scope — append as A1.1
   Extension to step 3 amendment, not pull into step 4. Step 3
   class of bug; extension block lives under step 3 so step 3
   build session sees it first. Filed per [Schema Amendment Not
   Overwrite] — new block, not edit of upstream A1.1 text.

#8 A2.5 Pagination helper shape — sibling `fetchAllRpc(name, args,
   opts)` in src/lib/supabaseHelpers.js. Mirrors `fetchAll` shape
   exactly; loops `.range()` around `supabase.rpc(name, args)`.
   Centralizes the truncation guard in one place; keeps
   Customers.jsx call sites one-line swaps from `fetchAll` →
   `fetchAllRpc`.

Other notable session decisions (not ratifications, but worth
flagging):

A2.4 — Return type `SETOF public.v_proposal_customer_resolved` for
   `customer_proposals`. Postgres registers views in pg_type, so
   SETOF <view> is valid. PostgREST surfaces view-typed RPC returns
   the same as table-typed. Client receives one extra column per
   row (`effective_customer_id`) which is harmless; existing
   Customers.jsx selects only the columns it consumes. Fallback if
   PostgREST view-typed return errors at build: change return type
   to SETOF public.proposals and route predicate through
   `WHERE id IN (SELECT id FROM v_proposal_customer_resolved …)`.
   Same substrate-sharing intent; one extra sub-SELECT.
