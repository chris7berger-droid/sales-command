SC Handoff v117 — 2026-05-14
============================

SESSION SUMMARY
Build session that started as a customer-side bug report and turned
into a full structural fix. Tim Milton (problem customer? no — Lepori
is the problem customer; Tim was just the one who emailed) clicked his
invoice email's "View Full Invoice" link and saw "Oops · Invoice not
found" with a 400 in devtools. Plan-mode investigation found the
public invoice page query had been failing for 100% of anon viewers
since 20260502130000 shipped — two RLS policies cast
`i.job_id::int = call_log.id` but `invoices.job_id` stores
descriptive display strings (intentional denormalized copy used by 8
readers including QB sync edge functions), not stringified call_log
PKs. Cast threw inside the predicate, PostgREST returned 400.
Stripe Pay Now button never went through the broken path, so payments
kept going through while the View link was silently dead. Chose the
best-practices fix over the shortcut: added `invoices.call_log_id`
integer NOT NULL FK as the canonical pointer, backfilled from
proposals.call_log_id, rewrote both policies to use the clean FK.
Hard-deleted one orphan invoice + Draft proposal pair that blocked
the NOT NULL backfill. Two follow-up hotfixes shipped to main after
the migration landed: (a) writer was setting call_log_id from an
embed object that didn't include id, fix used the scalar column
proposals.call_log_id directly; (b) /invoice/ and /invoice-paid
customer routes were hitting the 3-second boot radar because the
gate variable only checked /sign/. Smoke clean across 7 invoices,
5 random proposals, and 9 Lepori signing links — all 200 OK as anon.
Numbering jumps to v117 because v113–v116 are on feat/multi-gc-allocation.

DECISIONS / CHOICES MADE
────────────────────────
- Best-practices fix over shortcut. Initial plan proposed routing
  RLS through proposals.call_log_id (no schema change) as the
  minimal fix. Discovery during grep showed `job_id` is
  intentionally a denormalized display copy used by 8 readers
  including 4 QB edge functions — removing or repurposing it would
  cascade. Real cleanest shape is a SEPARATE clean FK alongside
  job_id, not a replacement of it. Chris ratified
  "Commit + push all 5 + patch" route (full fix in this session)
  over the shortcut. Per [Build It Right] memory.

- Hard-delete the orphan invoice 09121 + proposal 3273564d-* inside
  the migration. Both were soft-deleted on 2026-05-11 within 15s of
  each other. Invoice was a $2.19 Stripe test charge with no QB
  sync; proposal was a Draft that never had a call_log_id assigned.
  Leaving them would have forced call_log_id nullable forever (or a
  later cleanup). Chris OKed the destructive op explicitly via
  AskUserQuestion before the SQL was written. Migration body
  documents the rationale inline so a future re-reader has the
  forensic trail.

- Cherry-pick the 20260513000000 migration file onto this branch
  instead of merging feat/multi-gc-allocation to main. Reason:
  feat/multi-gc-allocation is mid-plan (§10 step 4 just ratified);
  not ready to merge. The file IS the conceptual change needed for
  ledger consistency, no branch-local coupling to drop. Per
  [Cherry-Pick Contextual Coupling] memory.

- Two follow-up hotfixes (16320fd, 51cdaab) shipped directly to main
  with no preview branch. Reason: both were fixing regressions just
  shipped to main minutes earlier. Time-to-fix mattered. Per
  [Minimal Fix First] — fix the new code first, no detour through
  preview-deploy ceremony when we're correcting our own just-shipped
  bug.

- Renamed `isCustomerSigningRoute` → `isCustomerFacingRoute` even
  though [Stay Scoped] memory pushes against scope expansion. The
  rename matches the existing comment's intent verbatim
  ("Customer-facing public routes must skip the boot loader") and
  prevents the next added public route from re-introducing the same
  gap. Two-line edit, no behavior change beyond what the bug fix
  required.
