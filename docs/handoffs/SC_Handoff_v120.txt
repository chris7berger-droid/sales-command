DECISIONS / CHOICES MADE
────────────────────────
1. SQL-only path for the 6618 rebuild instead of having Chris do
   step 2 via the wizard UI. Chris's stated state was mindset 2,
   concern 10, potential 4 — drained. Less context switching =
   better. Pre-checked the call_log schema for NOT NULLs
   (job_name, tenant_id, qb_skip_sync) before writing the INSERT;
   all three preserved by SELECT ... FROM call_log WHERE id=3512.
   Risk was a constraint violation rolling back the transaction;
   acceptable since the partial state was reversible.

2. Filed B25 retroactively as Closed-in-same-session rather than
   skipping the row. Chris asked for "language that makes it easy
   to find because we keep fixing old bugs" — wrote the row with
   a dedicated **Symptom (searchable)** block listing the natural-
   language phrases the next grep would use ("silent save fail",
   "no error popup", "edit mode dropped", "changes don't persist
   after Save Changes").

3. Class fix in handleSave shipped same session as the data
   rebuild — not deferred. Without the class fix, the same row
   would re-cement "CO1" on its next edit. Pairing the class fix
   with the data fix makes both stick.

4. parseInt-on-form.job_number hardening was a regression check on
   my own first draft. First version of the class fix used
   `form.job_number || String(job.job_number)` — but the Job
   Number input is type="text" at :478, so free-text "6618 CO1"
   would leak into numPart and produce "6618 CO1 CO1 - ..." on
   save. Caught it before commit and tightened to
   `parseInt(form.job_number) || job.job_number`.

5. Did NOT remove the Manager Override button. Chris asked about
   it after Joe's recall; the wizard's manualJobNum input is
   type="number" so "CO1" can't be typed there. Removing the
   button wouldn't have prevented this bug class — the Standard
   Job + free-text Project Name path has the same exposure.
   Logged as an open companion product question in the B26 row's
   Notes rather than acting on it.

6. ERD-discipline lesson captured to Floating Notes (not as a
   memory) — ETET, "Every Terminal Every Time." Chris started
   the bug-hunt outside protocol because the day began with a
   protocol-verification task and his brain auto-extended the
   "no-protocol" mode. The first bug fix shipped without an ERD
   loop. Mid-second-bug, Chris recognized the drift and started
   Loop #7. Captured as a floating note + erdnote inside Loop #7.
