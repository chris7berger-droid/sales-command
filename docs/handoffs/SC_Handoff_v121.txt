   issue there — `.neq("id", wtcId)` no-ops when wtcId is null on first

   never-true — was investigated but not fixed. Locking sibling edits
   removes the practical exposure (siblings can't write a different
   rate now), so the sync-no-op-on-first-save is now belt-without-
   suspenders. Logged below as a parked finding.

- Original PW carry-through bug — self-resolved, not root-caused.
  Suspected mechanism: sibling auto-fill at WTCCalculator.jsx :1761
  reads `pwSibling.pw_rate || 0` and the oldest PW sibling may have
  been saved with pw_rate=0 if autosave fired before the user typed the
  rate. Auto-fill faithfully inherits 0. If this resurfaces, check the
  DB state of WTC 1 before assuming auto-fill is broken.
- Sync no-op on first save of new WTC — `handleSave` :1941 filters
  `.neq("id", wtcId)` where wtcId is null on a new-WTC INSERT (closure
  captures null even after setWtcId(newRow.id) queues). PostgREST
  translates `id=neq.null` as always-false; sync matches zero rows.
  Locking sibling edits makes this a no-op in practice now, but the
  bug is still latent. If sibling editability is ever re-introduced,
  fix the sync to use `newRow.id` as the exclude key.
