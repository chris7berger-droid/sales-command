DECISIONS / CHOICES MADE
─────────────────────────
- Picked the lock-condition change (option B in conversation) over
  deleting the invoice (option A) or a direct DB hack (option C).
  B is the right long-term fix — matches the user mental model of
  "pull back = editable again" and doesn't lose invoice/history.
- Loosened lock check to exclude only `status="New"` rather than
  whitelisting (`Sent`/`Paid`). Treats "any non-`New` status" as
  committed. Aligns with `canPullBack` semantics in
  `Invoices.jsx:1154` (`status !== "New" && status !== "Paid"`),
  which also treats `New` as the editable state.
- Filed B32 instead of fixing it in-session. Today's Plenium send
  was unblocked by deleting+recreating the draft pay app
  (PayAppDetailModal.jsx:266-304 already supported this cleanly via
  `ON DELETE CASCADE`), so the snapshot-drift fix wasn't urgent.
  Doing it properly (banner + selective sync preserving billed %)
  is half a day, didn't fit tonight's window.
- F30 (CC support) recommend allowlist-gated CCs, not free-form —
  consistent with audit C9's threat model. Did not implement in
  session per user's call.
- Did not attempt to pre-warn that the "Add SOV to Pay App" button
  would disappear after the lock-scope change — caught it on Chris's
  smoke. Lesson worth holding loosely: when changing a gate
  condition (`locked`), grep for all UI elements gated on the same
  state and re-validate them.
