- f87450e Fix Job Detail linked invoices undercount for jobs with
  drifted job_id
  - Root cause: `src/components/CallLogDetail.jsx:189` filtered
    invoices by `.in("job_id", jobNumbers)` against the current
    `display_job_number`. Per migration `20260514130000` + BACKLOG
    B20, 36 historical invoices have `job_id` storing full
    descriptive strings (e.g. "7448 - Customer - Address - Work")
    instead of the bare display number. Invoice #10004 was one of
    them — its job_id no longer matched "6359", so it fell out of
    the filter. Same `linkedInvoices` array drives Job Totals →
    Billed/Remaining/% Invoiced were all undercounting.
  - Fix: switched to `.in("call_log_id", callLogIds)` — the
    canonical FK added + backfilled by `20260514130000`. Both
    invoice-create paths (`Invoices.jsx:256`, `NewPayAppModal.jsx:183`)
    already write call_log_id. Dropped the now-unused `jobNumbers`
    tracking. Single hunk in CallLogDetail.jsx, ~10 lines.

- aec45af Job Totals: prefer SOV contract_sum over stale
  proposals.total
  - Second issue surfaced after the first fix: Job Totals "Sold"
    still read $73,299 when the signed contract = $62,466
    ($47,466 base + $15,000 CO 001). Confirmed via signed CO PDF
    from DA Builders.
  - Root cause: archive-import flow
    (`ArchiveProposalModal.jsx:96-107`) sets `proposals.total` from
    a manually-typed sold amount, then never recomputes when the
    SOV is built out. CLAUDE.md already calls this drift out
    ("`proposals.total` can be stale") — Job Totals card was
    violating that rule by summing `p.total` directly.
  - Initial attempt: PostgREST embed `billing_schedule(contract_sum)`
    in the proposals select. Embed shape didn't unwrap as expected
    (1-to-1 UNIQUE FK relationship) — value silently fell through
    to `p.total`.

- 03ffd2a Replace billing_schedule embed with explicit fetch keyed
  by proposal_id
  - Switched to an explicit second query indexed by proposal_id.
    Matches the direct-query pattern used elsewhere
    (`PayAppDetailModal.jsx:160`). Stored in
    `contractSumByProposalId` state. Sold reducer prefers SOV
    when > 0, falls back to `p.total` for non-pay-app jobs.

- b68178b Linked Items proposal row: same SOV-over-stale-total
  preference
  - The Job Totals card now read $62,466 correctly, but the
    Linked Items → Proposals row below it still rendered
    `fmt$(p.total)` = $73,299. Applied the same fallback chain
    on `CallLogDetail.jsx:841`.

- fdc013c ProposalDetail: show Contract Sum + HISTORY LOCKER pill
  - Two adds on `/proposals/:id`:
    (1) Summary card gets a "Contract Sum" row sourced from the
        live billing_schedule.contract_sum when present; falls
        back to "Total" + proposals.total otherwise. Mirrors
        Job Totals preference.
    (2) HISTORY LOCKER pill in header when
        `p.call_log?.archive_record_id` is set and the proposal
        itself isn't already `is_archive_proposal=true`. Catches
        the case the existing ARCHIVE pill misses — for proposals
        created via the standard "+ New Proposal" flow on an
        archive-imported call_log (per memory
        `[Archive Import Creates No Proposal]` — History Locker
        creates call_log only; user picks the proposal flow next).
  - Added `archive_record_id` to four proposal-load selects:
    `Proposals.jsx:36`, `Proposals.jsx:47` (list loader),
    `ProposalDetail.jsx:116/498/642/652`. (~5 occurrences in
    ProposalDetail because line 652 packs two select calls into
    one expression — `replace_all` caught both.)
  - Added `sovContractSum` state in ProposalDetail + a second
    `useEffect` for the fetch.
