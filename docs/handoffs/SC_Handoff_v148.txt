Shipped the deposit-tag feature to prod. The job was framed as "just a tag,"
over-engineered through the build/audit cycle, then radically simplified to a
one-field model and shipped. Cross-repo design (GC pay-app + retention vs direct)
was settled; the Schedule-side indicator was carved out as a separate Cycle 2.

THE MODEL (v2, what shipped):
- A job is flagged deposit-required on call_log (deposit_required, deposit_amount).
- You bill the deposit with the EXISTING invoice flow (pay-app for GC w/ retention,
  regular invoice for direct) — NO special deposit-invoice creation.
- You mark the invoice with a "Mark as deposit" toggle → sets call_log.deposit_invoice_id
  (one pointer per job). Single-select, badge, and required/due/paid state all derive
  from that pointer, active-filtered (voided_at/deleted_at IS NULL) + sent-gated.

KEY DECISIONS
═══════════════════════════════════════════════════════════════════════
- Deposit flag lives on call_log (job-level), NOT proposals — so proposal-less archive
  jobs can carry it and Schedule reads it off the job. (Job-detail = the home; see memory.)
- Identify the deposit invoice by ONE pointer (call_log.deposit_invoice_id), NOT a per-
  invoice is_deposit flag. Collapsed away: is_deposit column, partial unique index,
  clear-before-set compensation, deposit_pending void-survival. The over-engineering's
  bugs dissolved rather than needing patches.
- No special "create deposit invoice" path — bill with the existing flow, then tag.
  Chris explicitly does NOT want the create-side pre-fill complexity reopened.
- A proposal-less INVOICE is impossible (every create path sets proposal_id) — verified.
