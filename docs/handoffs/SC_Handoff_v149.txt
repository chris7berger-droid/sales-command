CORRECTNESS — send-blocking (priority order)
==================================================================

1. [CONFIRMED] Stale recipient email hard-blocks send (400)  -- MOST IMPORTANT
   send-invoice/index.ts:165 (+ Invoices.jsx:1189, migration 20260625120000:23)
   Allowlist gate `!allowedRecipients.has(norm(mainRecipient.email))` keys off live
   customer_contacts emails, but invoice_recipients.contact_email only re-syncs when
   edited through the recipient card. Edit/delete that contact via the Customers page
   -> recipient email goes stale -> previously-sendable invoice returns
   400 "Main recipient <x> is not a known contact for this customer", no obvious cause.
   Note: saveRecipient (1189) only mirrors the email into customer_contacts when
   r.customer_contact_id is set, so orphan/auto-seeded recipients write nowhere the
   allowlist can see.

2. [CONFIRMED] Main recipient with no email -> cryptic 400
   Invoices.jsx:667
   Old unconditional `if (!billingEmail) return` guard replaced by noMainBlock +
   0-rows-only billing-email check. Nothing verifies the chosen main has an email.
   Toggle an email-less viewer to Main -> Send stays enabled (disabled only on
   sending||noMainBlock; noMainBlock false when a main exists) -> edge fn aborts
   400 "Main recipient (none) is not a known contact" instead of a clear
   "main recipient is missing an email".

3. [CONFIRMED] Billing contact added FIRST -> duplicated as main + viewer (two emails)
   Invoices.jsx:1127
   Dedup guard `if (recipients.some(r => r.customer_contact_id === c.id)) return;`
   reads stale React state, then ensureMainSeeded() inserts billing contact as `main`,
   then pickExistingContact/createNewRecipient inserts the SAME contact as `viewer`.
   Result: listed twice, gets pay-link main email + redundant view-only copy.

4. [CONFIRMED] First viewer on email-less customer -> unsendable (rows-but-no-main)
   Invoices.jsx:1107
   ensureMainSeeded early-returns when custInfo.billingEmail is empty. Customer with no
   Billing Contact and no billing_email: first added recipient creates rows-but-no-main
   -> send-invoice branch (iii) 400 "recipients but no main recipient". User must know
   to promote via the Main/Viewer toggle to escape.

5. [CONFIRMED] Send modal omits customers.email -> card promises a recipient Send can't deliver
   Invoices.jsx:608 (vs card :1280, edge fn :98)
   Three divergent resolutions of the billing email. Card (1280) + edge fn (98) include
   customers.email; Send modal (608/629) does NOT. Customer whose only address is
   customers.email: card shows "Main · pay link", Send shows "No billing email on file"
   and handleSend (673) blocks — even though the edge fn WOULD have sent. Dead-ends a
   sendable invoice.

6. [CONFIRMED] New contact requires no email -> junk contact + silently dropped viewer
   Invoices.jsx:1141
   `if (draft.email && !isValidEmail(draft.email))` only validates when present.
   "+ New Contact" with name only -> blank-email customer_contacts row (permanent
   customer-file pollution) + blank-email viewer that the edge fn silently skips at send
   (warning only). Intended recipient never gets the copy.

7. [PLAUSIBLE] toggleMain non-atomic -> zero-main window
   Invoices.jsx:1123
   Two sequential UPDATEs (demote old main, promote new). Failure between them
   (network blip / RLS no-op) leaves rows-but-no-main -> 400 until user notices and
   re-toggles.
