=== The problem (why this work exists) ===
Customers were paying the EXACT contract amount (~2–3×/week, no customer-type pattern) while Sales Command
billed a ROUNDED-UP figure. Root cause, confirmed in code at design time: WTC prices were Math.ceil'd UP to
the whole dollar (calc.js:53/76 + 5 inline copies in WTCCalculator), and invoices inherit that by recomputing
live. The round-up (up to $0.99/WTC line) is a presentation artifact, NOT a contractual amount — the exact
figure customers pay equals our PRE-ceil line total (labor + mats + travel − discount). So the fix is to stop
the ceil from reaching the billed number for new work, while leaving everything already committed untouched.

=== Locked design decisions + WHY (the planning lens; alternatives rejected) ===
1. TRIGGER = an IMMUTABLE timestamp, NOT status. [LOCKED]
   Rule: a proposal prices exact iff its pricing era >= 2026-06-26 12:00 Central; else ceil (today's behavior).
   We rejected keying off status/"sent" because the app FREEZES price at Lock (status Draft) but RECOMPUTES the
   invoice live later (status Sent). A mutable trigger would freeze exact and bill ceil — re-creating the exact
   bug. created_at never changes → preview/freeze/invoice/PDF/snapshot always agree. This single decision is the
   spine of the whole feature.
2. CUTOFF = noon Central 2026-06-26. [LOCKED]  ***See operational flag below — this value needs a deploy-time call.***
3. FORWARD-ONLY. [LOCKED] Does NOT retroactively fix pre-cutoff proposals (including the one that triggered this
   work). Those are handled manually (override the invoice) or by recreating as a new proposal. The 11 existing
   unsent drafts STAY ceil; recreate one to flip it. Chris accepted this given only 11 exist.
4. SAFE DEFAULT = ceil. [LOCKED] calc fns default exact=false, so any unwired site keeps today's behavior — no
   path silently produces a WRONG number (worst case is "exact didn't kick in yet," caught in smoke).
5. MULTI-GC CLONES — durable fix via a NEW column, chosen over a clever shortcut. [LOCKED, option (b)]
   A clone gets a fresh post-cutoff created_at but copies the source's frozen (ceil) snapshot → freeze≠bill.
   We added nullable proposals.pricing_anchor_at; era = pricing_anchor_at ?? created_at; the clone RPC stamps
   the sister's anchor = COALESCE(source.pricing_anchor_at, source.created_at). REJECTED overloading created_at
   (makes the column lie about creation time → rots sorting/reporting) and JS-lineage joins at every price site
   (re-creates the coverage burden). Chris explicitly chose integrity/durability over the convenience of "no
   migration," which is why §2's no-migration lock was lifted for this ONE additive column. MultiGC is LIVE in
   prod (Chris corrected my stale-memory "not live" claim mid-design — verified in code: clone RPC deployed,
   MultiGCWizard rendered at ProposalDetail.jsx:1374, one sister out). So this was in-scope, not deferrable.
6. PDF cents-display DEFERRED to its own loop. [LOCKED] The web/HTML invoice already shows cents; only the jsPDF
   attachment lags, and an existing invoice.show_cents/fmt$c toggle + a retention-omission bug (invoicePdf.js:306)
   make it a separate, self-contained task. The PDF still receives a real proposal so the stored AMOUNT is exact.
7. E1 (sch-command penny tolerance) SCOPED OUT as adjacent. [LOCKED] Git isolates code, not the shared Supabase
   backend; sch-command's own fullyBilled/remaining math is the gap, it's unverified, and it's not a calc we
   export. Separate sch-command task if/when it bites.
