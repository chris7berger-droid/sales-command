=== Pre-existing Findings Surfaced (FILE as backlog — NOT for this branch) ===
HIGH — anon invoice-viewer can read proposals.signing_token (column-grant gap).
  public.proposals has NO column-level REVOKE for anon (confirmed: only function grants touch
  proposals; contrast invoices, which got REVOKE SELECT(...) in 20260625130000). The
  proposals_public_view_token RLS policy grants anon SELECT at ROW granularity on any proposal
  reachable from an invoice whose viewing_token the caller holds. With no column REVOKE, an anon
  holding a valid invoice viewing-token can hand-craft:
     GET /rest/v1/proposals?select=signing_token&id=eq.<pid>  (header x-viewing-token: <token>)
  and read the proposal's signing_token — the live secret gating the anon proposal-sign/update
  path (proposals_public_sign_token). Escalation: invoice-view -> proposal-sign capability, plus
  exposure of markup_override_pct, total, billing_terms_override, intro, etc. Same leak class the
  invoices column-REVOKE closed (stripe_checkout_url); proposals never got the equivalent.
  Why HIGH not CRITICAL: signing tokens expire/consume on signature (20260510120000), bounding the
  window to proposals that are both invoiceable and have a live signing token.
  FIX: mirror invoices hardening — default-deny then GRANT safe columns (REVOKE-all + GRANT, NOT a
  bare column REVOKE — see memory feedback_column_revoke_noop; a column REVOKE under a table grant
  is a NO-OP). FROM anon only, never FROM public. Then re-trace PublicInvoicePage + PublicSigningPage
  and smoke-verify AS ANON against prod. Separate shared-prod migration; own anon-surface trace.

LOW (defense-in-depth) — anon holds table-level INSERT/UPDATE/DELETE/TRUNCATE on proposals.
  These are Supabase's default GRANT ALL TO anon. INSERT/UPDATE/DELETE: RLS applies and there is
  NO permissive anon policy for them (all anon UPDATE policies dropped in 20260502130000 STEP 3;
  no anon INSERT/DELETE policy ever existed) -> default-deny, fully contained by RLS. TRUNCATE: NOT
  governed by RLS, but NOT reachable via PostgREST's HTTP interface -> not exploitable in practice.
  Net: not exploitable today, unnecessary grants. FIX: REVOKE INSERT,UPDATE,DELETE,TRUNCATE,
  REFERENCES,TRIGGER ON public.proposals FROM anon. File as defense-in-depth, not a blocker.
