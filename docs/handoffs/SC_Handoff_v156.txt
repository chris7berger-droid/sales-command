SESSION SUMMARY
----------------------------------------------------------------
Fixed the Multi-GC wizard's "add GC" customer picker truncating well
short of the full customer base. Screen 1 of MultiGCWizard fetched
`customers` with a plain unpaginated `.select().order("name")`, so
PostgREST returned only the first page and the client-side Commercial
filter ran against a truncated set — the GC list stopped short (Chris
saw it cap around 50). Prod has 2077 customers / 758 Commercial. Fix
swaps the raw select for `fetchAll()` from `supabaseHelpers` (the same
paginated helper the Customers page already uses). Merged to main,
smoke-verified by Chris, and a `/code-review high` blast-radius pass
came back clean. Branch deleted after merge.

----------------------------------------------------------------
DECISIONS / CHOICES MADE
----------------------------------------------------------------
- Used `fetchAll` (default pageSize 1000) rather than bumping a limit —
  matches the proven Customers-page pattern and the documented repo
  fix for the 1000-row PostgREST cap. Server row cap is >=1000 (the
  Customers page works with the same helper), so 2077 rows page cleanly
  as 1000 + 1000 + 77.
- Ran `/code-review high` (not `/runaudit`) for the blast-radius check —
  this was a solo `/fix` with no plan doc, so there's no `## Audit
  manifest` for `/runaudit` to consume. Correct tool for a 2-file diff.
- Diagnosed the customer count by querying prod from the linked
  `command-suite-db` repo — sales-command itself is unlinked from
  Supabase (migrations moved out 2026-06-29), so `db query --linked`
  fails here by design.
