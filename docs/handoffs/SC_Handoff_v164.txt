What it does:
  1. Edit-button gate widened New-only -> New + unpaid Sent-family (Sent / Waiting for Payment /
     Past Due) via new `canEditInPlace`; excludes Paid, voided, and any qb_payment_id-linked row.
     Label reads "Edit Sent Invoice" for the sent case.
  2. handleSaveEdit Paid/QB-payment write-guard (belt-and-suspenders behind the gate).
  3. `syncedLock = !!inv.qb_invoice_id` — for a synced invoice, number + discount + retention +
     per-line % are locked BOTH in the UI (disabled inputs) AND at the write path (preserve
     stored amounts, never recompute). Only description / intro / due date are editable.
  4. QB resync now AWAITS + SURFACES errors (was fire-and-forget `.catch(()=>{})`): reads
     fnErr.context.json() for the real QB fault; on failure the SC edit stays saved, the form
     stays open, and an alert tells the user to Save again to retry (idempotent — number+amount
     locked). This is the whole reason the PO->QB-memo flow is trustworthy.
  5. Reason-for-Edit field: required for synced edits, now a C.dark callout with amber label +
     border that PULSES (reasonPulse keyframe in tokens.js) while empty; the save-guard scrolls
     to + focuses it. (Iterated live: first version was too dim, then amber-on-linen muted — the
     fix is amber-on-dark per the teal-on-dark rule, helper text C.linenLight, no white.)
