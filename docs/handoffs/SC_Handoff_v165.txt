SESSION SUMMARY

Chris hit a rounding problem on Job 7442: the WTC's exact total of $37,154.09 was coming
through invoicing as $37,155.00, and the invoice "Exact" button didn't stop it. Traced the
invoice path in src/pages/Invoices.jsx + src/lib/calc.js and found it is NOT a code bug. The

Central 2026-06-26 — with pricing_anchor_at null, so usesExactPricing() returns false and the

re-approving today did NOT change it, because the pricing era is pinned to created_at, not to
approved_at. Confirmed the raw WTC price was exactly 37154.09 (a single line: drive_rate 1 x
drive_miles 37154.09). Chris ratified opting this one proposal into exact-penny (option A). Set
pricing_anchor_at = now() and total = 37154.09 on that proposal; uses_exact flipped to true.
Chris re-created the invoice and confirmed it carried $37,154.09 — "worked perfect." No branch
was needed (this was the /detach question that opened the session — answered: data fix, not code).

DECISIONS / CHOICES MADE
------------------------
- Root-caused before touching anything: the symptom (invoice ceils, "Exact" button doesn't help)
  is two different "exact" concepts colliding. usesExactPricing() = the pricing ERA (Math.round vs
  Math.ceil on the WTC base); the invoice modal's "Exact" toggle = roundInvoice/show_cents, which
  only controls whether CENTS DISPLAY. The display toggle can't un-ceil a total that was already
  rounded up upstream. So the fix had to be at the proposal's era, not the invoice.
- Picked A (opt this proposal into exact-penny) over B (accept the $0.91 legacy round-up). A is a
  one-row UPDATE, scoped to job 7442 only, and reversible (set pricing_anchor_at back to null).
- Set proposals.total = 37154.09 in the same UPDATE so the stored total matches the exact figure
  immediately, rather than requiring a re-lock in the UI to refresh it.
- Did NOT change any code. The "created before 2026-06-26 rounds up" behavior is intentional per
  the exact_penny_pricing plan; this was one legacy proposal Chris revived, not a systemic defect.
- No detached branch: the opening question was whether to /detach. Answer was no — a diagnostic +
  a data fix has no code to isolate and no migration, so nothing to collide with live work.
