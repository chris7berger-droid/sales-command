===============================================================================
PHASE 3 — PAY APP GENERATION (NEXT SESSION'S WORK)
===============================================================================

Trigger from this session: user asked "now how do I get the customer's pay
app filled out and an invoice generated in our system that matches?". Said
strongly: customer DA Builders does NOT accept Sales Command invoices —
they only pay against THEIR pay app template. So the integrated workflow
is mandatory; an internal-only invoice doesn't solve the real problem.

GOAL
  Given a signed Billing Schedule + the user's chosen "this app" billing %
  per SOV line, generate:
    a) A G702/G703 pay app PDF that matches the customer's template, AND
    b) A matching Sales Command invoice so when the customer pays, there
       is something in the system to attach the payment to.

THE FOUR PIECES (per v74 §C)
  1. NewInvoiceModal: branch on whether proposal has a billing_schedule
       yes → bill % per SOV line, retainage withheld
       no  → existing WTC-based flow (unchanged)

  2. Per-customer pay app template storage
       Need a place to upload the customer's G702 + G703 template (DA
       Builders' template was uploaded during v74 as a sample). Likely
       a customer_pay_app_template column on customers, OR a new
       customer_pay_app_templates table if multiple per customer needed.
       Decide based on whether one customer can have different templates
       per project type.

  3. G702/G703 PDF output
       jspdf is already bundled (used by ProposalPDFModal). Could either:
         - Render a PDF that mirrors the customer's template visually
           (jspdf hand-build), OR
         - Fill in a PDF form on top of the customer's uploaded template
           (pdf-lib supports form filling — heavier dep but matches
           template exactly).
       The "match the template" requirement strongly suggests pdf-lib
       form-filling unless the customer's template is consistent enough
       to recreate from scratch.

  4. Running totals — pay app math
       Per pay app: previous app, this app, balance to finish, retainage
       accrued (5% per line withheld), retainage released (when?).
       Need a billing_schedule_pay_apps table to track each app's
       per-line %, plus a billing_schedule_lines.previous_billed
       computed view OR rolled-up at-app-time totals.

DATA MODEL SKETCH (start here, refine in conversation)
  billing_schedule_pay_apps:
    id (uuid), billing_schedule_id (uuid FK), app_number (int),
    period_from (date), period_to (date), submitted_at (timestamptz),
    pdf_url (text), invoice_id (text FK invoices, nullable),
    status (text — draft | submitted | paid),
    created_at, updated_at, tenant_id

  billing_schedule_pay_app_lines:
    id (uuid), pay_app_id (uuid FK), billing_schedule_line_id (uuid FK),
    billed_pct_this_app (numeric), billed_amount_this_app (numeric)

  customers:
    + pay_app_template_url (text, nullable) — uploaded G702/G703 template

UX FLOW SKETCH (refine first thing in next session)
  From BillingScheduleSection → "+ New Pay App" button (visible when
  schedule is locked / approved):
    1. Modal opens. Shows each SOV line with: scheduled value, previously
       billed (sum of prior pay apps), input for "% complete this app"
       (or % to date — pick one and stay consistent with G703 format).
    2. Auto-calc: this app $, retainage withheld, balance to finish.
    3. Save → creates billing_schedule_pay_apps row + lines, generates
       G702/G703 PDF (using customer's template), creates SC invoice
       for (this app $ − retainage), links invoice ↔ pay app.
    4. Send → emails customer the PDF + invoice link.

OPEN DECISIONS BEFORE CODING
  - jspdf hand-build vs pdf-lib form-fill (depends on template
    inspection — open DA Builders template first session)
  - Single template per customer or multiple? (DA Builders may use
    different templates per project)
  - When does retainage release? (manual flag on a follow-up "release
    retainage" pay app vs auto on final?)
  - Are pay apps numbered globally or per-customer? (per-customer
    is standard)
  - Does Phase 3 need to backfill anything for the existing Muscle
    Motors job (which now has SOV lines saved this session)?

NON-PHASE-3 FOLLOW-UPS WORTH NOTING
===============================================================================

A. EYEBALL THE NEW "SAVE" BUTTON in WTC Materials → + Custom row.
   Not visually verified before deploy.

B. Settings page has NO role gating except for Billing (subscription).
   WorkTypesSection and MaterialsCatalogSection both let any role
   add/edit/delete. Per feedback_role_gating.md, money config should
   be Admin/Manager only. Pre-existing pattern, not a regression. A
   future "gate the whole Settings page" pass would be quick.

C. The legacy billing_schedule.contract_pdf_url column is still in the
   schema (kept for back-compat in v74). Eventually remove it after
   confirming no readers depend on it.

D. extract-sov has no rate limiting. Possible to spam-trigger up to
   the per-tenant catalog of doc URLs. Low risk for now; could add
   a per-user-per-day cap if it becomes an issue.

E. WTCCalculator's "+ Custom" Save button does no dedup check — if
   the user types the same name+kit_size as an existing catalog row
   (system or tenant) and clicks Save, you'll get a duplicate row
   in materials_catalog. The picker dedupes on display, so it'll
   still look right, but the catalog table will have crud. Could
   add a pre-insert check.
