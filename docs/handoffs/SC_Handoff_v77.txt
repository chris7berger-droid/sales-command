4. ARCHIVE SCHEMA — THE BIG FIND (commits 87aa685 + a1adc40 + migration ...230000)
   THE BUG (latent multi-tenant breaker):
     archive.get_user_tenant_id() returned `SELECT id FROM tenant_config
     LIMIT 1` regardless of caller. Every authenticated user got the SAME
     tenant_id from the helper — whichever tenant_config row Postgres
     happened to return first. RLS on archive.{legacy_records,
     import_batches} silently broken under multi-tenancy.

   COMPOUND BUG:
     src/lib/supabase.js exported archiveDb that used SUPABASE_KEY as the
     Bearer token (anon role, no auth.uid()). Even with a correct helper,
     archive RLS would never resolve the right tenant.

   THE FIX (sequence mattered):
     a) FRONTEND FIRST (87aa685): archiveDb now caches the current
        session.access_token via auth state subscription, rebuilds the
        PostgrestClient per query so the latest user JWT is sent. Falls
        back to anon key if no session.
     b) MIGRATION SECOND: replaced archive.get_user_tenant_id() with
        the auth.uid()→team_members lookup pattern from public.get_user_tenant_id
        (per SC_Handoff_v69), but DROPPED the tenant_config fallback —
        archive must never serve anon (per v69 anon access map), so
        NULL-on-no-match is safer (RLS denies the query).
     c) Tightened grants: REVOKE ALL FROM anon entirely; narrowed
        authenticated from ALL to USAGE on schema + SELECT/INSERT/DELETE
        on each table.
     d) Documented the schema via CREATE TABLE IF NOT EXISTS so future
        audits see it in supabase/migrations/ instead of /sql/.

   WHY THIS WAS THE MOST IMPORTANT FIND:
     Pushing the migration alone would have killed every HistoryLocker
     call (because archiveDb was anon-keying past the broken policies).
     The frontend fix had to land first; coordinated deploy.
     Status: fully live, single-tenant behavior unchanged. Multi-tenant
     correctness now in place for when scmybiz.com onboards tenant #2.

5. PAGINATION BUG — IMPORT WIZARD (commit dfd6fd8)
   Chris hit this live: tried to import an archived job for "Forrest A
   Dodge Construction" — customer clearly exists (1 of 2060), but the
   wizard's customer dropdown said "not found."
   Cause: ImportToLiveWizard.jsx loaded customers without pagination,
   hitting the 1000-row PostgREST cap. Forrest is past row 1000
   alphabetically → invisible to dropdown AND auto-match → wizard forced
   "Create New" → duplicate customer.
   Fix: replaced supabase.from("customers").select(...) with fetchAll(...).

6. PAGINATION SWEEP — 12 MORE BUGS ACROSS 6 FILES (commit e50a438)
   Audit #2 had INCORRECTLY cleared pagination as ✓. Re-swept after the
   wizard bug exposed the pattern. Found 12 more silent data-loss bugs:
     - Home.jsx: call_log + proposals
     - SalesDash.jsx: call_log + proposals + proposal_wtc + invoices
     - Proposals.jsx: proposals + invoices
     - Invoices.jsx: invoices list
     - Customers.jsx: invoices on customer detail
     - Managers.jsx: call_log + proposals (KPI math)
   All replaced with fetchAll() including correct filter syntax for
   .is("deleted_at", null) and conditional sales_name filters.

   PERFORMANCE NOTE: fetchAll on proposals/proposal_wtc with deeply
