MAJOR WORK
===============================================================================

1. CALL LOG — DELETE ATTACHMENTS
   - × button on each attachment pill (CallLogDetail) with confirm prompt.
   - Hardened against the silent-no-op pattern: storage.remove() returns
     { data: [], error: null } when a DELETE policy is missing — now we
     check data.length and surface a clear error instead of pretending it
     worked.
   - Migration adds the missing DELETE policy on the job-attachments
     bucket so the actual deletes succeed.
   - Memory note saved (feedback_storage_silent_noop.md) so future
     storage code remembers this gotcha.

2. ARCHIVE → LIVE: NO MORE SKELETON PROPOSAL
   - ImportToLiveWizard used to insert a proposal row with total = sold
     amount and zero proposal_wtc rows. That broke the New Invoice flow
     silently because the per-WTC % grid had nothing to render.
   - Removed the skeleton insert. Archive-imported jobs now create
     a call_log only. Users build a real proposal afterwards (standard
     OR archive-type — see #3).
   - CallLogDetail shows "No proposal yet — invoicing is disabled until
     a proposal is created" when linkedProposals is empty.

3. ARCHIVE JOB PROPOSAL (NEW PROPOSAL TYPE)
   - New column proposals.is_archive_proposal (boolean default false).
   - New ArchiveProposalModal — lightweight form: Sold Amount, Description,
     Work Types (tagging only), no WTC builder.
   - Reachable two ways: dedicated "+ Archive Job Proposal" button on
     CallLogDetail (only when archive_record_id is set), and a secondary
     button on the standard NewProposalModal.
   - ProposalDetail detects is_archive_proposal and swaps the WTC
     Calculators panel for an ArchiveProposalPanel showing editable
     Sold Amount + Already Billed (Historical) + tagged work types.
   - New Invoice modal detects archive proposals and swaps the per-WTC %
     grid for a single "Invoice Amount" capped at remaining =
     total - (historical_billed + sum of in-system invoices).
   - Standard proposal flow is fully untouched.

4. HISTORICAL BILLED AMOUNT
   - New column proposals.historical_billed_amount (numeric default 0).
   - Captured on archive proposal creation (Already Billed (Historical)
     input next to Sold Amount).
   - Editable on ProposalDetail's ArchiveProposalPanel (read-only view
     shows it side-by-side at full prominence; Edit reveals inputs).
   - "Edit historical billed →" link in the New Invoice archive
     breakdown closes the modal and jumps to the proposal detail.

5. ARCHIVE BADGE ON CALL LOG
   - Purple ARCHIVE pill next to the Sold stage pill on Call Log list
     rows and Call Log detail header when call_log.archive_record_id is
     set. Hover tooltip explains.

6. URL-BASED ROUTING (BIG REFACTOR)
   - Replaced the App-level state machine (active, initial*Id, navHistory)
     with real react-router URLs. Every section + detail now has a path:
       /home, /calllog, /calllog/:id, /proposals, /proposals/:id,
       /invoices, /invoices/:id, /customers, /customers/:id, etc.
   - Browser back/forward works natively across cross-section nav.
   - Refresh stays on the current detail.
   - URLs are now shareable.
   - Cross-section state hand-offs (Home → CallLog stage filter, CallLog
     → Proposals new-proposal job preselect) use react-router location
     state.
   - The earlier in-memory navHistory/popNav system was ripped out
     (was added earlier today as a half-measure; should have done routing
     in the first place — see Open Thread #4 / lessons-learned in memory).

7. CROSS-LINK NAVIGATION BUTTONS
   - Job/Proposal/Invoice detail headers each have small "Job →" /
     "Proposal →" / "Invoice →" buttons that navigate to the linked
     detail across sections (only shown when exactly one is linked).
   - Plus "+ Create Invoice" button on ProposalDetail when status=Sold
     (works for both standard and archive proposals).

8. PROPOSAL NUMBERING — REVERT-AFTER-OVERFIX
   - I introduced a duplicate-P# bug in ArchiveProposalModal by counting
     non-deleted proposals only. Briefly tried a universal renumber-
     on-delete change across NewProposalModal + ProposalDetail. Chris
     audited me and called out unnecessary edits to working original
     code. Reverted to minimal fix: ArchiveProposalModal now counts ALL
     (matching original NewProposalModal). Original monotonic numbering
     scheme preserved. Memory note saved (feedback_minimal_fix_first.md).

9. POLISH
   - Money inputs (Sold Amount, Already Billed) format with thousand
     separators on blur (96,520.06 not 96520.06).
   - New Invoice picker filters out soft-deleted proposals (was showing
     duplicate P-labels for deleted skeletons).
   - NewInvoiceModal step 2 guards against null selProposal during
     auto-select after preselectedProposal mounting.

1. PAGE REMOUNT ON LIST↔DETAIL ROUTE CHANGE
   /proposals and /proposals/:id are separate Route definitions, so
   navigating between them remounts the Proposals component. Same for
   CallLog/Invoices/Customers. Pages reload their data each time.
   CallLog has sessionStorage cache so it's snappy; others reload from
   network. Acceptable but if it feels janky, switch to a wildcard
   Route or nested Outlet pattern.
