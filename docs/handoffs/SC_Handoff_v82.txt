MAJOR WORK
===============================================================================

1. PER-INVOICE ROUND / EXACT TOGGLE
   - Problem: fmt$ rounds to whole dollars site-wide, so an archive invoice
     entered as $2,834.95 displayed (and invoiced to the customer) as
     $2,835.  Martin Harris doesn't round — they need $2,834.95.
   - Added nullable invoices.show_cents column (NULL = inherit from
     call_log.show_cents).
   - New Invoice modal has a Round / Exact toggle pinned near the top.
     Defaults from call_log.show_cents (off → Round, on → Exact).  Modal
     display and the saved show_cents flag both use the toggle.
   - PublicInvoicePage now prefers invoice.show_cents, falls back to
     call_log.show_cents.  So the customer-facing page matches what you
     chose at creation.
   - In-app invoice list / detail / PDF previews still render fmt$c always
     (full precision for your reference).  If that feels wrong later we
     can wire those to show_cents too.

2. APPROVE → QB BUTTON (pay-app-only customers)
   - Some GCs (e.g. Martin Harris) don't accept HDSP-format invoices at
     all — they require billing submitted through their own system
     (Procore / GCPay / Textura).  HDSP still needs the invoice in QB for
     record keeping so the incoming check can match an open invoice.
   - New "Approve → QB" button on InvoicePDFModal header, next to
     "Send Invoice".  Shown when status=New and !hideSend.
   - Action: calls qb-sync-invoice, marks invoice status=Sent + sent_at=now
     via the existing onSent plumbing, closes modal.  NO customer email.
   - Memory saved (project_pay_app_only_customers.md) distinguishing this
     flow from the HDSP-pay-app product (different thing entirely).

3. PER-INVOICE RETENTION
   - New columns: invoices.retention_pct (e.g. 10 for 10%) + invoices.
     retention_amount (the held-back dollars, computed at save).
   - New Invoice modal: Retention (%) field next to Due Date.  Live preview
     under input shows "Held back: $X · Net due: $Y".
   - InvoiceDetail edit: Retention (%) field next to Discount.  Same live
     held-back preview.
   - Read-only InvoiceDetail: StatCard grid grows to 4 columns when
     retention > 0 (Invoice Amount | Discount | Retention (X%) | Net Total).
   - InvoicePDFModal + PublicInvoicePage: "Less Retention (X%)" line
     appears above the boxed Amount Due.  Amount Due = amount − discount
     − retention.
   - qb-sync-invoice: retention posted as a DiscountLineDetail line with
     Description "Retention (X%) — held back".  QBO Plus / Advanced have
     native retainage but we don't depend on that tier — this keeps the
     QB net matching what's actually being billed now.
   - Scope (a) only: no release-retention workflow yet.  Held-back $ lives
     on the invoice row; when you're ready to bill the retention back, we
     build a "create release invoice from held retention" button (pulls
     from all invoices on the job and creates a single release invoice).

4. FIX: EDIT-INVOICE ZEROED ARCHIVE INVOICES (destructive bug)
   - Chris caught this between the retention ship and a real save:
     on an archive invoice, the edit view showed Line Value $0.00 and
     Billing % 0 (because archive lines have proposal_wtc_id=null and
     billing_pct=null, and the table calc'd from calcWtcPrice(null)=0).
   - Hitting Save Changes would have recalc'd newAmount = 0 and
     overwritten the actual invoice total.  On the Plenium $57,097.70
     invoice this would have wiped the AR.
   - Fix: isArchiveInvoice detected when every line has no wtc and no sov.
     Edit form shows a full-width "Invoice Amount ($)" input bound to
     editArchiveAmount.  Save logic branches: archive lines preserve the
     directly-entered amount; standard lines keep the WTC × % recalc.
   - Line items table handles archive lines: label "Archive Invoice",
     Line Value shows stored amount (or live editArchiveAmount in edit
     mode), Billing % shows "—".
