   -------------------------------------------------------
   Archive proposals have no proposal_wtc rows, so archive invoice lines
   were rendering "—" / $0.00 / stray "%". Now pulls alternative data:

                  proposal.call_log.job_work_types

     Billing %:   line.amount / proposal.total × 100

   Billing % follows the per-invoice Round/Exact toggle:
     Round (show_cents=false): integer percent → "100%"
     Exact (show_cents=true):  2-decimal       → "99.99%"

7. Clickable column sort on the shared DataTable
   ---------------------------------------------

     ↕ (faint white)  - sortable but inactive
     ↑ (teal)         - active, ascending

LESSONS / RUNBOOK
================================================================

A. Migration ledger drift after Studio SQL
   When SQL is pasted into the Supabase SQL Editor, supabase_migrations
   .schema_migrations does NOT get a row. Next time `supabase db push`
   runs, the CLI thinks every "applied via Studio" migration is still
   pending. The fix:

     supabase migration repair --status applied <ts1> <ts2> …

   Then `supabase db push` (add --include-all if the new migration
   timestamps before the just-marked ones).

   Prevention: ALWAYS run new migrations via `supabase db push`, never
   the SQL Editor. Studio is a last resort for emergency hotfixes only.

B. Auto-merge of touched-by-both files needs eyeballing
   The PR #6 auto-merge of PublicInvoicePage.jsx came out semantically
   correct (kept both sides' intent) but auto-merge can be syntactically
   clean and behaviorally wrong. Always grep the touched file post-merge
   for both sides' identifiable changes before assuming it merged
   correctly.

C. v-numbered handoffs must be checked against main before naming
   The v83 collision (one v83 on a feature branch + a different v83
   on main, both 2026-04-27) was a numbering accident waiting to bite.
   Before naming the next handoff, check `git show origin/main:SC_Handoff_v??.txt`
   to verify the number is free.
