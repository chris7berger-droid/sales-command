2. QuickBooks auto-sync skip — two-axis design
   ---------------------------------------------
   Problem: Importing legacy archive jobs that already exist in QB,
   and using the existing Archive Job Proposal flow, double-posts to
   QB (creates duplicate sub-customer + duplicate invoice + duplicate
   payment).

   Design (after planning agent surfaced the mixed-proposal case):
   skip if EITHER signal is true:
     - proposals.is_archive_proposal === true   (existing column)
     - call_log.qb_skip_sync === true           (new column)

   This handles the user's real workflow: an imported archive job can
   carry both an archive-style proposal (auto-skipped via
   is_archive_proposal) AND a fresh new-live proposal (skipped or
   synced based on the per-job flag).

   Migration 20260428120000_call_log_qb_skip_sync.sql:
     - alter table call_log add column qb_skip_sync boolean default false
     - backfill = true for every row where archive_record_id is not null
     - guard: AND qb_skip_sync = false so re-running the migration won't
       clobber a manual override
     - partial index on qb_skip_sync = true

   Edge functions — all 4 honor the flag, return HTTP 200 with shape
   { success: true, skipped: true, reason: "is_archive_proposal" |
     "qb_skip_sync" } so the void-flow user-visible error path stays
   silent and the other three function's .catch wrappers behave the
   same as before:

     - qb-create-job: accepts optional proposalId in body. If proposal
       is archive-style OR job has skip flag → return success-skipped.
     - qb-sync-invoice: skip check runs BEFORE the qbCustomerId throw
       so archive jobs (which have null qb_customer_id) skip cleanly
       instead of surfacing "Job not synced to QuickBooks yet".
     - qb-record-payment: REORDERED — call_log lookup now happens
       BEFORE the qb_invoice_id null-check for the same reason.
     - qb-void-invoice: skip check before the existing "not in QB"
       branch. Response shape carefully avoids the `error` field so
       Invoices.jsx's surfaced error path stays quiet.

   Frontend wiring:
     - ProposalDetail.jsx + PublicSigningPage.jsx: now pass proposalId
       alongside callLogId in qb-create-job invoke calls.
     - ImportToLiveWizard.jsx: inserts qb_skip_sync: true on every
       archive import (belt-and-suspenders, in case the migration
       backfill is ever rolled back or skipped).
     - CallLogDetail.jsx: new checkbox in JOB INFO section under the
       existing show_cents toggle. Label: "Skip QuickBooks auto-sync
       on new proposals (archive-style proposals are always skipped)".
       Mirrors the show_cents pattern (form state, handleSave, button
       chassis).

   QB SKIP badge — visible confidence indicator:
     - ProposalDetail header: teal-on-dark "QB SKIP" pill next to the
       Draft status pill (and after the ARCHIVE badge if archive-
       style). Hover for tooltip explaining the trigger.
     - Proposals list row: same badge in compact form in Status column.
     - Renders only when sync would be skipped (no badge = will sync,
       which is the default behavior for the dominant case).
     - qb_skip_sync threaded into the call_log joined selects in both
       ProposalDetail.jsx and Proposals.jsx so the badge has data to
       read.

   (commits 625f28b, f553ffe)

================================================================
LESSONS / RUNBOOK
================================================================

A. Multi-surface features deserve a planning pass
   This session: 1 migration + 4 edge functions + 3 insert paths +
   2 UI surfaces + sibling-repo sweep + deploy ordering. Spinning up
   the Plan agent before any code was written caught:
     - The qb-record-payment ordering bug (had to move the call_log
       lookup BEFORE the qb_invoice_id check)
     - The qb-void-invoice response-shape requirement (no `error`
       field, since that path is user-surfaced)
     - The mixed-proposal case the user expected (drove the design
       to use BOTH proposals.is_archive_proposal AND
       call_log.qb_skip_sync, OR'd together)

B. Stacked feature branches work fine for related-but-distinct work
   feat/qb-skip-sync was branched from feat/calllog-add-co-button
   (not main) so the user could test both features on a single Vercel
   preview while we built. Merge order to main: dependency first
   (CO), then the stacked branch (qb-skip-sync). Both merged cleanly
   with --no-ff merge commits preserving the feature history.

C. Edge-function-first deployment is the safe path for a multi-
   surface bug fix
   The order DB → edge functions → frontend means the bug fix is
   already live as soon as edge functions deploy. Frontend just adds
   UI affordances. If a frontend deploy fails or has to be rolled
   back, the bug fix stays in place.

D. Plan agent open-question protocol
   Plan agent flagged 3 open questions. User answered 1/2/3 in a
   single message; we revised the design when #2 changed the answer
   ("expects mixed-proposal case" → swapped from one-flag-on-call_log
   to OR'd-with-is_archive_proposal). Trust but verify works both
   ways: planner surfaced the limitation we'd missed; user
   surfaced the workflow we'd miscaptured.
