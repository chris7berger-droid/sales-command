Topic: Contact picker + Billing Contact override on Import-to-Live and
       New Inquiry wizards. Sales reps were re-typing contact and billing

================================================================
WHY
================================================================

Two wizards prompt for contact + billing info while creating a job:
  - HistoryLocker/ImportToLiveWizard.jsx (Archive → Live, step 3)
  - NewInquiryWizard.jsx (contactInfo step)

Both ignored the customer's saved contacts. Plenium Builders has Erica
Thomas on file with role='Billing Contact', plus PMs Patrick and Will,
yet step 3 still presented blank Billing Contact Name/Phone/Email
fields marked REQUIRED. The user had to re-type Erica's info on every
import.

The fix: shared picker that fetches customer_contacts on customer
selection, presents existing contacts as a Job-Contact dropdown, and
auto-fills + locks the Billing Contact block whenever a row with
role='Billing Contact' exists. Falls back to legacy customers.billing_*
denormalized fields, then to manual entry.

WHAT WAS DONE
================================================================

1. Plan agent up front
   --------------------
   Surface touched: customer_contacts, customers.billing_*, two
   wizards, primary-contact rule, multiple-Billing-Contact edge case,
   RLS, save-side legacy bridge. Plan agent designed a shared
   ContactBillingPicker component, recommended per-customer fetch on
   selection (matches NewInquiry's existing pattern + ProposalDetail
   pattern), and identified that ArchiveImportWizard, bulk Import.jsx,
   and CallLogDetail are NOT additional surfaces (verified — only the
   two wizards take a per-record contact + billing step).

2. New shared component: src/components/ContactBillingPicker.jsx
   -------------------------------------------------------------
   * Fetches customer_contacts on customerId change (useEffect with
     fetchSeqRef abort token to ignore stale responses on rapid
     customer toggling).
   * pickBillingContact() helper: filters role='Billing Contact', prefers
     is_primary, falls back to most-recent created_at.
   * Picker UI states:
     - 0 contacts → no dropdown, "No saved contacts on file" hint,
       manual contact inputs visible.
     - N contacts → "Job Contact" dropdown labeled "<Name> — <Role>",
       with a "+ Use a different contact (manual entry)" toggle
       revealing manual inputs as an opt-in fallback.
   * Billing Contact lock: when found, renders a dark/teal locked card
     (matches the "Matched Customer" pattern at ImportToLive line 414),
     hides manual billing inputs entirely, and fires
     onBillingLockChange(true, contactId) so parents can track
     billingSourceContactId in form state.
   * Multiple Billing Contacts: warns "Multiple billing contacts on
     file — using <Name>. Consolidate on customer record." Non-blocking.
   * RLS error path: if SELECT throws, surfaces "Couldn't load saved
     contacts: <reason>" instead of looking like an empty list.

3. ImportToLiveWizard.jsx (Archive → Live)
   ---------------------------------------
   * Step 3 "Contact & Billing" replaced with <ContactBillingPicker />.
     Passes requireBilling, showBillingTerms (terms select stays
     inline since it's part of step 3 visually).
   * Form state: added billingSourceContactId (null when manual,
     contact uuid when picker locked it).
   * New-customer save: after the customers insert, also inserts a
     customer_contacts row with role='Billing Contact', is_primary=true
     when the user filled the billing block. Captures error (alert on
     fail; non-blocking — customer was saved).
   * No change to existing-customer save path — that path was already
     correctly NOT writing customer billing fields.

4. NewInquiryWizard.jsx
   --------------------
   * contactInfo step: inline contact name/phone/email block replaced
     with <ContactBillingPicker hideBilling={data.customerMode === "new"} />.
     Billing UI for new customers stays in customerSelect step (where
     the "separate billing contact?" checkbox lives) — the picker only
     adds the billing locked card / manual block for existing-customer
     mode in this wizard.
   * customerSelect onChange: removed the inline customer_contacts
     fetch into additionalContacts (picker owns it now). Now also
     prefills legacy customers.billing_* fields onto form state, so
     when the picker fetch returns no Billing Contact row, the manual
     block is pre-populated from the legacy denormalized columns.
   * Save: when billingSourceContactId is set, OMIT billing_name/
     billing_phone/billing_email from the customers update (avoids
     overwriting customer_contacts-sourced data with a mirror of
     itself). billing_terms still writes through.
   * New-customer save: inserts a customer_contacts Billing Contact
     row when the manual billing block was used. Dedupes against the
     additionalContacts insert by (lowercased name + email) so a user
     who also added a Billing Contact in the additionalContacts list
     doesn't get a duplicate.
   * additionalContacts batch insert: now captures error; alert on fail.

5. Insert error capture (commit 584dce9)
   --------------------------------------
   Three customer_contacts.insert() calls were silently swallowing
   errors. Audit finding M0k flagged this exact pattern in
   Customers.jsx:175 and elsewhere — "RLS can silently no-op." All
   three new/touched inserts now capture error and alert non-blocking
   so the user knows the customer saved but the contact didn't, and
   can recover by editing the customer record.

LESSONS
================================================================

A. Plan agent before code, when the surface is wide
   The picker spec has subtle rules: 0/1/N picker states, multiple
   Billing Contacts edge case, when to write legacy columns vs not,
   stale-cache handling, additionalContacts dedupe. Asking the Plan
   agent first surfaced edge cases that would have leaked out as bugs
   during testing. Same lesson as v89.

B. Pre-commit cross-check before pushing a feature branch
   Discovered an accidental duplicate commit (ab69186) on the feature
   branch that was identical to b2380b2 already on main. Caught by
   running git log on the branch before push. Once seen, dropped via
   git reset --hard origin/<branch>. Lesson: when working in
   ~/sales-command from multiple terminals, the active branch in the
   working tree determines where commits land — switch to main before
   doing unrelated docs work in another terminal.

C. Capture insert errors at the call site, not just delete errors
   CLAUDE.md "RLS can silently no-op" is written about deletes but the
   pattern applies to inserts too — RLS rejection on insert returns an
   error in the response, but if the caller doesn't destructure
   `{ error }` it disappears silently. Audit M0k formalized this.
   Three new inserts in this branch now capture errors with non-
   blocking alerts on failure.

D. Branches in working tree shared across terminals are a footgun
   Same as B — if both terminals point at ~/sales-command, the active
   branch is global to that working tree, not per-terminal. Use git
   worktrees or distinct checkouts for parallel work.

Test 8 (mid-wizard customer change) initially FAILED — switching the

contact + billing data persisting in form state. Two root causes:

  A. NewInquiryWizard customerSelect onChange (and ImportToLiveWizard
     customerMatch select onChange) only set contact/billing fields
     when the new customer had truthy values — so a new customer with
     no contact_phone left the prior customer's phone in form state.
     Fix: unconditional reset on customer change with empty-string
     fallback.
  B. ContactBillingPicker only updated its local contacts state AFTER
     the new fetch returned, so the previous customer's locked Billing
     Contact card visually persisted during the loading window.
     Fix: clear contacts immediately on customerId change before
     starting the new fetch.

Lesson E (new): conditional set with `value || existing` is the wrong
default for a "switching context" handler. When the user changes which
record they're acting on, ALWAYS reset all derived fields, even to ""
or null. Otherwise stale data from the previous context bleeds into
the new one. Same shape of bug shows up in any "select X to populate
form" handler. Add to the audit-bait list.
