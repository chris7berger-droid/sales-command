WHAT SHIPPED

A. Smarter customer auto-match in Import to Live wizard (34f926a)
   - File: src/components/HistoryLocker/ImportToLiveWizard.jsx
   - Old: exact case-insensitive name match only, fell through silently
     to "Create New" with the archive name prefilled — easy to miss
     and create dupes.
   - New: exact → acronym (IVGID ↔ Incline Village General
     Improvement District) → token-overlap with stopword filter
     (LLC, Inc, Co, Corp, Company, Ltd, etc.). Token overlap requires
     ≥2 shared meaningful tokens, ≥50% archive containment, and a
     clear lead over second-best to prevent attaching to a wrong
     customer.
   - Plus: native <select> on Use Existing replaced with the existing
     SearchSelect component so the long customer list is searchable.

B. is_billing_contact flag (b110b83)
   - Migration: supabase/migrations/20260506100000_customer_contacts_is_billing_contact.sql
       ALTER TABLE customer_contacts ADD COLUMN IF NOT EXISTS
       is_billing_contact boolean NOT NULL DEFAULT false;
       UPDATE customer_contacts SET is_billing_contact = true
         WHERE role = 'Billing Contact' AND is_billing_contact = false;
   - Already applied to prod via supabase db push --linked.
   - UI:
       * "Use as billing contact" checkbox in Edit Contact modal,
         alongside "Mark as primary contact"
         (src/pages/Customers.jsx)
       * BILLING badge on contact cards, sits next to PRIMARY
       * ContactFlagToggle helper extracted to dedupe checkbox markup
   - Resolution updated everywhere — back-compat reads
     `is_billing_contact OR role='Billing Contact'`:
       * src/components/ContactBillingPicker.jsx (pickBillingContact,
         multipleBilling, billingContactIdFor exports)
       * src/pages/Invoices.jsx (loadContact for billing email)
       * src/components/PayAppDetailModal.jsx (loadAll cust hydration)
       * supabase/functions/send-invoice/index.ts — REDEPLOYED with
         --no-verify-jwt
   - Wizards that auto-create role='Billing Contact' rows
     (NewInquiryWizard, CallLogDetail, ImportToLiveWizard) NOT
     touched — back-compat resolver handles them.
   - Edit Contact modal seeds is_billing_contact from
     `contact?.is_billing_contact ?? (contact?.role === "Billing Contact")`
     so existing legacy rows render with the box ticked.

C. Import to Live wizard re-import collision (cc71eab)
   - File: src/components/HistoryLocker/ImportToLiveWizard.jsx
   - Detection: legacy lookup now also fetches archive_record_id and,
     when it matches the current record, child counts (proposals +
     invoices, both filtered by deleted_at IS NULL).
   - Same-archive collision UI on Step 7 replaces the radios with:
       1. "Open Existing Job #NNNN" (always available; calls
          onSaved({ jobId: legacyTakenBy.id }) which navigates to

          successful import).
       2. "Replace Existing Import" (disabled if existing has any
          proposals or invoices; click opens typed-confirm modal,
          user types `REPLACE NNNN` to enable Confirm).
   - Replace flow at handleImport time:
       * Re-checks child counts (concurrency safety — another tab
         might have added work since the wizard loaded).
       * Deletes job_work_types where call_log_id = legacyTakenBy.id.
       * Deletes the call_log row.
       * Falls through to the normal "useLegacyJobNum" path which
         now finds the number free.
   - validate() refuses to leave Step 7 if same-archive collision is
     active and replaceMode wasn't set — user must pick Open or
     Replace, can't just hit Next.
   - Cross-archive collision (legacy num held by a different
     archive's call_log) keeps the prior "force Assign New" behavior
     unchanged.

OPEN — B7 (root cause for the original 7406 archived=true mystery)
──────────────────────────────────────────────────────────────────
DB-side checks all came back clean:
   * call_log.archived column default = false
   * No INSERT triggers on call_log (only trg_call_log_updated_at,
     BEFORE UPDATE → set_updated_at())
   * tenant_config archive_after_months = 12, archive_stages = ["Lost"]
     — auto-archive in CallLog.jsx:64 won't match a fresh "Sold" row
   * 7406 row: created_at = 2026-05-06 16:30 UTC, stage = Sold,
     archived = true, archive_record_id set
Yet the row landed archived=true. Defensive fix prevents recurrence
on imports going forward, but we don't know what flipped that
specific row. Two suspects to rule out next time:
   1. Accidental click of "Move to Old Jobs" on CallLogDetail.jsx:414
      during testing (silent UPDATE, no confirmation).
   2. An unfound code path that updates call_log.archived after the
      wizard's insert.
Repro test for the next session: do a fresh archive import on prod
(not the IVGID Chateau record — pick another), check which tab the
new job lands in. If Active, B7 likely was a stray click and can be
closed. If Old, we have a real bug and need to instrument.
