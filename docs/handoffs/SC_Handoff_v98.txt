DECISIONS LOCKED (don't reopen unless evidence forces)
──────────────────────────────────────────────────────
Q1   One call_log per project. Use existing call_log.job_name
     for project title (no new column). Sisters stacked under
     it. On Mark Awarded → winner Sold, sisters auto-flip to Lost
     with lost_reason + timestamp. Reversible. Never hard-deleted.

Q2   Source-driven sync for content (proposals.intro, proposal_wtc
     sales_sow + jsonb columns: field_sow / materials / sub_areas
     / travel; size; unit; discount; discount_reason). Per-GC for
     commerce (customer, contact, RFP#, due, markup). Sister
     overrides supported via proposal_wtc.locally_edited_fields[];
     conflict prompt on source-edit hitting overridden field.

Q3   Resolved by Q1 (sister auto-Lost on award).

Q4   Both entry surfaces — ProposalDetail "+ Send to Additional
     GCs" and CallLogDetail "+ Add Another GC". Single wizard.

Q5   Forward-only. No retro-link UI. Manual SQL for rare in-flight
     cases.

Sweep-1  Add nullable proposals.customer_id FK → customers.
         NULL falls back to call_log.customer_id (preserves
         existing data semantics). Client switches
         p.call_log?.customer_id → p.customer_id ?? p.call_log?
         .customer_id everywhere it reads.

Sweep-2  Add nullable proposals.markup_override_pct numeric.
         Per-proposal multiplier on top of per-WTC markup_pct.
         Powers Screen 3 of wizard cleanly.

CRITICAL ISSUES OPEN (Plan agent's first delivery, 2026-05-06)
──────────────────────────────────────────────────────────────
C1   mark_proposal_signed RPC (in 20260502120000_signing_flow_
     security_definer.sql:107-116) auto-flips call_log.stage =
     'Sold' system-wide on the FIRST sister to sign. With Q1's
     shared call_log, the first signer becomes de facto winner
     before "Mark Awarded" is even invoked. ProposalDetail.jsx:
     601-614 does the same client-side, then triggers qb-create-
     job. Plan §6 must either replace this pathway when sisters
     exist OR explicitly accept "first signer = winner."

C2   Clone RPC violates H6 invariant: sisters get locked = false
     but inherit locked_line_total. Per
     20260505190200_proposal_wtc_locked_line_total.sql:32-43,
     locked_line_total is NULL for unlocked WTCs. Fix in migration
     draft: locked_line_total => NULL. Sisters must lock + send
     themselves (mirrors today's flow).

C3   proposals.status enum-vs-text-column unverified. Plan §6
     introduces literal 'Lost'. If status is text with no CHECK,
     fine. If enum, ALTER TYPE ... ADD VALUE 'Lost' required.
     Same for call_log.stage if enum.

C4   proposals.id is TEXT, not uuid (per CLAUDE.md:94 — invoices.
     proposal_id text FK proposals). Plan §4 RPC declares
     p_source_proposal_id uuid AND cloned_from_proposal_id uuid.
     Both wrong. Audit table FK to proposals also needs text.

S1   Customers.jsx:257-258, 519 filters customer-detail jobs/
     proposals by call_log.customer_id directly via PostgREST.
     After Sweep-1, sister proposals where proposals.customer_id
     = X but call_log.customer_id = Y will be INVISIBLE on
     customer X's page. Real regression vector. Fallback pattern
     alone won't fix this — likely needs a denormalized column,
     view, or RPC.
